# CRM AI 教练助手二期优化计划

> 日期：2026-04-26
>
> 文档状态：plan / candidate design
>
> 适用范围：`CRM 客户档案 + AI 教练助手`

---

## 1. 这份文档解决什么问题

当前 `CRM 客户档案 + AI 教练助手` 已经不再是纯概念验证。

它已经具备了可用的 MVP 闭环：

1. 客户档案页可打开 AI 教练助手。
2. AI 能读取客户上下文并回答问题。
3. 已支持正式回复流式输出和可展示思考摘要。
4. 已支持客户专属补充信息。
5. 已支持历史对话回看与基于同一 `session_id` 继续追问。

但如果把目标定义为“健康教练日常工作的决策辅助台”，当前版本仍然处于：

> 能用，但还不够稳；能演示，但还不够可治理；能回答，但还不够可复盘。

所以二期的任务不是再堆几个按钮，而是把这条线从“可运行 MVP”推进到“可持续运营的正式能力”。

---

## 2. 当前阶段判断

### 2.1 当前已经能做什么

当前正式代码真值已经覆盖：

1. 客户 360 档案基础聚合。
2. AI 提问、非流式与流式回复。
3. 可展示的思考摘要流。
4. 客户专属补充信息维护。
5. Prompt 分层与场景策略切换。
6. 历史对话列表与历史消息回看。
7. 医疗复核标记与基础审计留痕。

### 2.2 当前半能做什么

以下能力“方向已对，但还没真正收口”：

1. 安全门禁：
   目前有过敏词和医疗关键词检查，但仍偏文本关键词级，离结构化安全门禁还有距离。
2. 历史会话续聊：
   目前可以继续沿用旧 `session_id`，但没有完整恢复当时的场景、输出模式和 prompt 元信息。
3. 上下文治理：
   已有 `selected_expansions` 参数与白名单思路，但还没形成正式的模块展开能力。
4. 审计复盘：
   已有消息和上下文快照留痕，但还缺少 `prompt_version / prompt_hash / scene_key / output_style` 的正式持久化真值。

### 2.3 当前还不能说完成的

以下能力目前不能写成 official 完成：

1. 结构化安全门禁。
2. 完整的 AI 调用可追溯审计。
3. 教练可控的上下文展开治理。
4. 强一致的历史会话续聊语义。
5. 可长期迭代的前端组件结构。

---

## 3. 二期优化目标

二期不追求“再加更多 AI 花样”，而是追求 5 件事：

1. 更安全：
   让 AI 的建议边界更稳定，不能只靠关键词碰运气。
2. 更可追溯：
   每次 AI 输出都能回答“当时用了哪版 Prompt、哪个场景、哪些上下文”。
3. 更可控：
   教练能决定是否展开某些重点模块，而不是只能吃系统默认摘要。
4. 更一致：
   历史会话回看和继续聊不能前后切换策略而不自知。
5. 更可维护：
   前端抽屉和后端编排都要为后续继续加功能预留结构，不再越改越重。

---

## 4. 二期工作线

### 4.1 工作线 A：安全门禁 2.0

#### 当前问题

当前后置安全检查主要依赖：

1. 过敏原是否在回答中出现。
2. 是否命中少量医疗关键词。

这对健康教练场景不够。

因为很多风险不是“出现了某个词”，而是“建议方向本身不适合当前客户”。

#### 二期目标

把安全门禁从“文本关键词检查”升级到“结构化风险门禁 + 文本补充检查”。

#### 建议实现

1. 先构建 `SafetyDecisionInput`：
   从 `safety_profile` 提取：
   - `risk_level`
   - `allergies`
   - `contraindications`
   - `medical_history`
   - `sport_injuries`
   - `prescription_summary`

2. 再定义规则分层：
   - `hard_block`
   - `medical_review_required`
   - `coach_warning`

3. 规则示例：
   - 若回答建议高强度运动，但 `contraindications` 中存在关节/运动禁忌，则至少进入 `medical_review_required`。
   - 若回答触发停药、换药、处方、诊断表述，则直接 `medical_review_required`。
   - 若回答包含过敏食材，则至少 `coach_warning`，严重时 `medical_review_required`。

4. 输出结构化门禁结果：

```json
{
  "safety_result": {
    "level": "medical_review_required",
    "reason_codes": ["medical_term", "contraindication_conflict"],
    "notes": ["建议与当前运动禁忌冲突，需要复核"]
  }
}
```

#### P0 验收口径

1. 不再只依赖关键词。
2. 至少能识别：
   - 医疗表述风险
   - 过敏冲突
   - 运动禁忌冲突
   - 高风险客户保守输出
3. 前端能明确展示“为什么要复核”。

---

### 4.2 工作线 B：审计真值 2.0

#### 当前问题

当前已记录：

1. `session_id`
2. `message_id`
3. 消息内容
4. token 用量
5. latency
6. context snapshot hash
7. used_modules

但仍缺少会影响结果解释的关键真值：

1. `prompt_version`
2. `prompt_hash`
3. `scene_key`
4. `output_style`
5. 是否使用了思考流

#### 二期目标

做到“回头审一条历史 AI 建议时，能完整解释它的生成条件”。

#### 建议实现

1. 为 `crm_ai_sessions` 增加：
   - `scene_key`
   - `output_style`
   - `prompt_version`
   - `prompt_hash`

2. 或者为 `crm_ai_context_snapshots` 增加：
   - `prompt_version`
   - `prompt_hash`
   - `scene_key`
   - `output_style`
   - `selected_expansions`

3. 推荐口径：
   - `session` 记录会话级默认策略
   - `context_snapshot` 记录每轮实际调用参数

4. 历史会话详情接口返回这些值，前端可在历史对话顶部展示：
   - 当时工作场景
   - 输出模式
   - Prompt 版本

#### P0 验收口径

1. 历史对话详情页能看到本轮调用使用的：
   - 场景
   - 输出模式
   - Prompt 版本
2. 审计表能支持问题追查，而不是只能看聊天内容。

---

### 4.3 工作线 C：上下文治理 2.0

#### 当前问题

报告里已经定义了：

1. `always`
2. `default`
3. `expandable`
4. `display_only`
5. `never`

也已经有 `selected_expansions` 参数。

但在当前正式链路里，这块还没有真正形成“教练可控展开模块”的能力。

#### 二期目标

把“上下文可解释、可展开、可控预算”真正落地。

#### 建议实现

1. 在 AI 抽屉里增加“附加参考模块”选择区：
   - 血糖明细
   - 饮食明细
   - 睡眠明细
   - 运动明细
   - 积分明细
   - 任务明细

2. 每次最多允许展开 2 个模块。

3. 后端在构建 `CustomerProfileContext` 时正式消费 `selected_expansions`：
   - 非展开模块只走摘要
   - 被展开模块进入代表性样本/统计视图

4. 在 UI 中显式显示：
   - 默认用了哪些模块
   - 额外展开了哪些模块
   - 本轮上下文预算是否偏大

#### P0 验收口径

1. 教练能显式勾选 1-2 个附加模块。
2. 后端构建的上下文确实随勾选变化。
3. AI 返回结果和审计快照里能看出展开模块。

---

### 4.4 工作线 D：历史续聊一致性 2.0

#### 当前问题

当前历史会话已经能回看，也能继续沿用旧 `session_id` 提问。

但如果打开一段旧会话后，前端当前选择的：

1. `scene_key`
2. `output_style`

已经和当时不同，那么同一条会话前后就会跑在不同策略下。

这在用户感知上是“同一次会话”，在系统实际上却可能已经换了 Prompt 语义。

#### 二期目标

把“历史对话回看”和“历史对话继续聊”真正做成一致体验。

#### 建议实现

1. 历史会话详情加载时，同时恢复：
   - `scene_key`
   - `output_style`
   - `prompt_version`

2. 继续追问时默认沿用历史会话策略。

3. 如果教练手动改了场景或输出模式，应显式提示：

```text
你正在修改这段历史会话的工作场景/输出模式。
后续回复将按新策略继续生成。
```

4. 可以考虑提供：
   - `按原策略继续`
   - `基于当前策略继续`

#### P0 验收口径

1. 打开旧会话后，侧栏场景与输出模式自动同步。
2. 用户改策略时有显式提示，不再静默切换。

---

### 4.5 工作线 E：前端结构优化 2.0

#### 当前问题

当前 `AiCoachPanel.vue` 已经承担了：

1. 头部
2. 侧栏
3. 历史会话
4. 上下文展示
5. 客户配置
6. 聊天消息流
7. 输入框
8. 大量样式

这意味着后续每加一个功能，回归范围都会越来越大。

#### 二期目标

把前端从“能跑的大组件”改造成“可长期迭代的功能组合”。

#### 建议拆分

建议至少拆成：

1. `AiCoachHeader`
2. `AiCoachSidebar`
3. `AiContextPanel`
4. `AiProfileNotePanel`
5. `AiChatMessages`
6. `AiChatInput`
7. `AiSessionHistoryList`

同时把状态分层：

1. `session state`
2. `message stream state`
3. `context config state`
4. `history state`

#### P1 验收口径

1. `AiCoachPanel.vue` 不再继续膨胀。
2. 后续新增“收藏模板 / 搜索历史 / 复制引用上下文”时不需要反复改一整个大文件。

---

## 5. 推荐开发顺序

### 5.1 P0

先做这 4 项：

1. 安全门禁 2.0
2. 审计真值 2.0
3. 上下文治理 2.0
4. 历史续聊一致性 2.0

原因：

1. 这 4 项直接影响结果的可信度。
2. 它们决定 AI 建议能不能真正用于教练日常工作。
3. 先做这些，后面的体验增强才有正式意义。

### 5.2 P1

再做：

1. 前端拆分
2. 历史对话搜索
3. 会话标题摘要
4. 会话收藏/置顶

### 5.3 P2

再考虑：

1. 旧 AI 历史摘要接入
2. 更多可展开模块
3. 模板化教练动作建议
4. 复制到发送中心/交接单的更深闭环

---

## 6. 建议拆分为研发任务

### 任务 1：安全门禁规则引擎一期

输出：

1. `SafetyDecisionInput`
2. `evaluate_ai_answer_safety()`
3. 结构化 `reason_codes`
4. 前端风险原因展示

### 任务 2：AI 审计字段补全

输出：

1. migration
2. 模型字段补齐
3. 写入链路补齐
4. 历史会话详情接口补齐新字段

### 任务 3：上下文展开模块落地

输出：

1. 前端展开选择 UI
2. 后端消费 `selected_expansions`
3. 审计快照记录实际展开模块

### 任务 4：历史会话一致性修复

输出：

1. 打开历史会话时恢复策略
2. 切换策略时提示
3. 续聊语义校验

### 任务 5：AI 抽屉前端拆分

输出：

1. 拆组件
2. 降低主文件复杂度
3. 保持现有交互不回归

## 7. 验收标准

### 7.1 产品验收

1. 教练能明确知道 AI 参考了哪些信息。
2. 教练能控制是否展开某些重点模块。
3. 高风险输出不会静默通过。
4. 历史会话回看和继续聊的策略是一致的。
5. 历史记录不只是“聊天日志”，而是可复盘的 AI 决策记录。

### 7.2 技术验收

1. 后端审计字段完整。
2. Prompt 版本与场景真值可查询。
3. 前端大组件复杂度下降。
4. 既有流式链路不回归。

### 7.3 风险验收

1. 不因新增审计字段破坏旧会话查询。
2. 不因安全门禁增强造成普通问答全部误伤。
3. 不因上下文展开导致 token 预算失控。

---

## 8. 当前 blocker

当前最值得优先暴露的 blocker 不是“模型效果不够聪明”，而是：

1. 结果解释性不足：
   无法完整追溯一次回答的 Prompt 真值。
2. 风险控制仍偏弱：
   仍主要依赖文本关键词门禁。
3. 上下文治理还没真正接通：
   `selected_expansions` 仍未形成正式能力。

如果这 3 点不先解决，后续继续加场景和按钮，只会让系统“看起来更强”，但不一定“更可靠”。

---

## 9. 最值得继续推进的线

如果只允许二期先做一轮，我建议优先级是：

1. `安全门禁 2.0`
2. `审计真值 2.0`
3. `上下文治理 2.0`
4. `历史续聊一致性 2.0`

这是最能把 AI 教练助手从“能用 demo”推进到“正式业务辅助能力”的一组动作。
