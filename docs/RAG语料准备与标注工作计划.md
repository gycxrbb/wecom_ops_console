# RAG 语料准备与标注工作计划

> 文档日期: 2026-04-26  
> 适用范围: 健康教练 AI 助手 RAG 语料治理  
> 目标读者: 项目负责人、运营负责人、教练主管、内容整理人员、后续开发人员  
> 关联方案: `docs/RAG集成到wecom项目手册.md`

---

## 1. 目标

技术人员开始建设 RAG 底座时，业务侧需要同步完成语料准备和前期标注。当前公司知识分散在飞书文档、微信聊天、教练个人经验、图片、视频、课件和历史话术里，如果不先治理，后续即使向量库和 AI 接好了，也会出现召回不准、推荐素材不可用、内部话术误发客户、重复内容太多等问题。

本计划目标是建立一条通顺链路：

```text
散落资料
  ↓
集中盘点
  ↓
清洗脱敏
  ↓
结构化标注
  ↓
小样本试点
  ↓
进入话术管理 / 素材库 / RAG 索引层
  ↓
AI 教练可检索、可引用、可推荐
```

---

## 2. 总体原则

1. 先整理高频场景，不追求一次性收全。
2. 先做人工可控标签，不让 AI 自动决定正式标签。
3. 先区分“内部参考”和“可发客户”，避免安全事故。
4. 先做文本语义，再接图片/视频 URL。
5. 先做 P0 试点集，再批量铺开。
6. 所有内容都要能追溯来源，不做无来源知识。

---

## 3. 资料来源盘点

### 3.1 来源类型

| 来源 | 示例 | 处理方式 |
| --- | --- | --- |
| 飞书文档 | SOP、课程文档、控糖知识、餐评指南 | 先登记链接和文档责任人，再摘录可用段落 |
| 微信聊天 | 教练与客户的优秀回复、群内答疑、客户追问 | 只摘录话术和问题模式，必须脱敏 |
| 教练个人话术 | 常用回复、破阻力话术、激励话术 | 进入话术候选池 |
| 图片素材 | 餐盘示例、控糖图解、习惯打卡图 | 进入素材候选池，补图片说明 |
| 视频素材 | 课程片段、饮食方法、运动讲解 | 进入素材候选池，补摘要和转写 |
| 课件/培训材料 | 内训 PPT、课程脚本、健康知识 | 拆成知识卡片候选 |
| 历史发送内容 | 发送中心模板、群运营内容 | 可作为话术候选，不直接等于正式知识 |

### 3.2 第一批优先场景

第一批不建议全量整理，先围绕高频且能明显提升 AI 可用性的场景：

1. 晒餐提醒
2. 餐评反馈
3. 晚间加餐/零食控制
4. 外食选择
5. 血糖波动复盘
6. 体重平台期安抚
7. 不打卡/掉队唤回
8. 饮食习惯科普
9. 情绪性进食陪伴
10. 教练跟进复盘

建议 P0 目标量：

| 类型 | P0 数量 | 说明 |
| --- | ---: | --- |
| 固定话术 | 100-150 条 | 每个高频场景 10-15 条 |
| 知识卡片 | 50-80 条 | 每条 100-500 字 |
| 图片素材 | 30-50 个 | 每个必须有图片说明 |
| 视频素材 | 10-20 个 | 每个必须有摘要，优先补转写 |
| 评估问题 | 50-100 条 | 用来测召回是否准确 |

## 4. 语料分层

所有内容整理时先分层，不同层的处理方式不同。

### 4.1 话术

定义：教练可以参考、改写或复制给客户的表达。

示例：

1. “今天这餐整体比昨天更稳，尤其是主食量控制得不错...”
2. “晚上想吃零食不是意志力差，很多时候是晚餐结构不够稳...”
3. “这次血糖高先别慌，我们先一起复盘三件事...”

进入方向：

1. 后续进入“话术管理”。
2. 可被 RAG 检索。
3. 可在 AI 回复后作为推荐话术展示。

### 4.2 知识卡片

定义：相对稳定的健康管理知识、方法论、解释材料。

示例：

1. 为什么控糖人群要关注餐后 2 小时血糖。-jpg
2. 减重期间晚餐如何搭配更不容易饿。-png
3. 外食点餐时如何判断隐藏油脂。-jpg

进入方向：

1. 后续作为 `manual_card` 或外部文档资源进入 RAG。
2. 可注入 AI 回答作为依据。
3. 可关联图片/视频素材。

### 4.3 素材

定义：图片、视频、文件、图文内容。

进入方向：

1. 文件本体进入素材库 `materials`。
2. 摘要、说明、标签进入 RAG 索引层。
3. AI 前端只推荐可用、可发、已审核素材。

注意：图片/视频 URL 本身不能产生语义。必须补：

1. 业务标题
2. 内容摘要
3. 图片说明或视频转写
4. 适用场景
5. 禁忌/谨慎

### 4.4 评估问题

定义：用来测试 RAG 能否召回正确语料的问题集。

示例：

1. “客户说晚上很想吃甜食怎么办？”
2. “用户午餐吃了面条血糖高，怎么回复？”
3. “用户三天没晒餐了，怎么提醒不反感？”

进入方向：

1. 不直接进入知识库。
2. 用于开发验收和模型/检索策略评估。

## 5. 标注字段模板

建议先用飞书多维表格或 Excel 做人工整理。每一行代表一个候选资源。

### 5.1 通用字段

| 字段 | 必填 | 示例 | 说明 |
| --- | --- | --- | --- |
| `source_id` | 是 | `feishu_doc_001` | 人工编号即可 |
| `source_type` | 是 | `speech`, `knowledge`, `image`, `video` | 资源类型 |
| `source_location` | 是 | 飞书链接/微信群名/文件路径 | 原始来源 |
| `owner` | 是 | 张教练 | 内容责任人 |
| `title` | 是 | 晚间零食破阻力话术 | 业务标题 |
| `raw_content` | 否 | 原始摘录 | 未清洗内容，内部暂存 |
| `clean_content` | 是 | 清洗后的内容 | 后续进入 RAG 的正文 |
| `summary` | 是 | 这段话用于... | 50-200 字 |
| `status` | 是 | `candidate`, `approved`, `disabled` | 先用 candidate |
| `reviewer` | 否 | 教练主管 | 审核人 |
| `review_note` | 否 | 可发客户，但需避开疾病承诺 | 审核备注 |

### 5.2 标签字段

| 字段 | 必填 | 推荐值 |
| --- | --- | --- |
| `customer_goal` | 是 | `weight_loss`, `glucose_control`, `habit_building` |
| `intervention_scene` | 是 | `meal_checkin`, `meal_review`, `obstacle_breaking`, `habit_education`, `emotional_support`, `qa_support` |
| `customer_stage` | 建议 | `onboarding`, `first_week`, `adaptation`, `plateau`, `relapse_recovery`, `maintenance` |
| `question_type` | 建议 | `craving`, `dining_out`, `late_night_snack`, `high_glucose`, `low_motivation`, `no_checkin` |
| `content_kind` | 是 | `script`, `knowledge_card`, `image`, `video`, `mixed` |
| `tone` | 建议 | `professional`, `encouraging`, `gentle`, `firm`, `competitive` |
| `visibility` | 是 | `coach_internal`, `customer_sendable` |
| `safety_level` | 是 | `general`, `nutrition_education`, `medical_sensitive`, `doctor_review` |
| `delivery_channel` | 建议 | `private_chat`, `group_chat`, `send_center`, `offline_followup` |

### 5.3 素材专属字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `file_name` | 是 | 原文件名 |
| `file_type` | 是 | image/video/file |
| `material_url` | 否 | 已上传后的素材 URL |
| `cover_url` | 否 | 视频封面或图文封面 |
| `alt_text` | 图片必填 | 图片内容说明 |
| `transcript` | 视频建议 | 视频转写文本 |
| `duration_seconds` | 视频建议 | 视频时长 |
| `usage_note` | 是 | 教练什么时候用 |
| `customer_sendable` | 是 | yes/no |
| `copyright_status` | 是 | self_owned/authorized/unknown |

## 6. 清洗规则

### 6.1 脱敏

必须删除或替换：

1. 客户姓名、手机号、微信号、地址。
2. 真实病历、检查单编号、医院号。
3. 教练个人手机号、微信名片。
4. 群二维码、私域入口码。
5. 任何未授权的第三方客户截图。

推荐替换：

1. 客户名 → “客户”
2. 教练名 → “教练”
3. 具体日期 → “某天”或保留相对时间
4. 真实数值可保留区间，但高风险疾病细节需审核

### 6.2 去重

重复话术不需要全部保留。

保留优先级：

1. 更清晰、更短、更适合教练复制的版本。
2. 有明确适用场景的版本。
3. 有主管认可或历史效果好的版本。

### 6.3 改写

微信聊天里的优秀回复通常不适合原样进入知识库，需要改写：

1. 删除上下文依赖太强的称呼。
2. 删除临时活动、具体奖品、具体客户信息。
3. 保留表达结构和干预策略。
4. 改成可复用模板。

### 6.4 安全审查

以下内容必须标记为 `medical_sensitive` 或 `doctor_review`：

1. 涉及用药、停药、药量调整。
2. 涉及疾病诊断。
3. 涉及低血糖、高血糖危险处理。
4. 涉及孕期、肾病、肝病、儿童、老人等特殊人群。
5. 涉及“保证瘦”“保证降糖”等效果承诺。

## 7. 推荐工作流

### 第 1 步：建立收集池

建立一个飞书多维表格，命名为：

```text
RAG 语料收集池
```

视图建议：

1. 全部候选
2. 待清洗
3. 待标注
4. 待审核
5. P0 试点
6. 已通过
7. 暂不使用

### 第 2 步：按场景认领整理

每个教练或运营同事只认领 1-2 个场景，避免口径发散。

建议分工：

| 角色 | 负责内容 |
| --- | --- |
| 教练主管 | 定义标签口径、审核话术安全 |
| 一线教练 | 提供真实高频问题和优秀回复 |
| 运营人员 | 整理飞书文档、课程资料、素材 |
| 内容负责人 | 清洗改写、统一表达风格 |
| 项目负责人 | 决定 P0 范围和优先级 |

### 第 3 步：先做 10 个场景的样板

每个场景先整理：

1. 10 条话术
2. 5 条知识卡片
3. 3 个图片/视频素材
4. 5 条评估问题

先形成样板，再批量铺开。

### 第 4 步：主管审核

审核只看三件事：

1. 能不能发给客户。
2. 有没有安全风险。
3. 标签是否准确。

审核结果：

1. `approved`: 可以进入 P0。
2. `revise`: 需要改写。
3. `internal_only`: 只能内部参考。
4. `disabled`: 不进入 RAG。

### 第 5 步：交给开发入库

交付给开发时，不要直接给散文档，要给结构化表格和素材包。

交付包应包含：

1. 标注表格。
2. 素材文件或素材库 ID。
3. 飞书文档来源链接。
4. 审核人和审核状态。
5. P0 评估问题集。

---

## 8. P0 时间计划

### 第 1-2 天：盘点与建表

1. 建立语料收集池。
2. 确定 10 个 P0 场景。
3. 确认标签字典第一版。
4. 明确内容负责人和审核人。

### 第 3-5 天：集中整理

1. 每个场景收集 10-20 条候选话术。
2. 每个场景整理 3-5 条知识卡片。
3. 收集可用图片/视频素材。
4. 生成初版摘要和标签。

### 第 6-7 天：审核和试点集冻结

1. 教练主管审核。
2. 删除重复和不安全内容。
3. 冻结 P0 试点集。
4. 输出给开发入库。

### 第 8 天以后：根据召回效果补齐

1. 根据无召回问题补语料。
2. 根据误召回调整标签。
3. 根据素材误推荐补充禁忌和可见范围。

## 9. P0 交付物

业务侧需要交付：

1. `RAG 语料收集池` 表格。
2. `RAG 标签字典 v1`。
3. `P0 试点语料包`。
4. `P0 评估问题集`。
5. `素材文件夹或素材库 ID 列表`。
6. `审核记录`。

建议 P0 最小数量：

| 交付物 | 最小数量 |
| --- | ---: |
| 已审核话术 | 100 条 |
| 已审核知识卡片 | 50 条 |
| 已标注图片素材 | 30 个 |
| 已标注视频素材 | 10 个 |
| 评估问题 | 50 条 |

## 10. 评估问题集模板

评估问题用于检查“用户这样问时，系统是否能召回正确话术/知识/素材”。

| 字段 | 示例 |
| --- | --- |
| `question` | 客户说晚上总想吃零食怎么办？ |
| `expected_scene` | `obstacle_breaking` |
| `expected_goal` | `weight_loss` |
| `expected_resource_keywords` | 晚间零食、饱腹感、替代方案 |
| `must_not_include` | 药物、极端节食 |
| `ideal_asset_type` | image 或 script |
| `reviewer` | 教练主管 |

每条评估问题最好写清：

1. 应该召回什么。
2. 不应该召回什么。
3. 是否需要推荐素材。

---

## 11. 常见错误

1. 把整篇飞书文档直接丢给技术，不拆段、不标注。
2. 把微信聊天截图原样作为知识库，不脱敏。
3. 把图片/视频只上传文件，不写摘要和说明。
4. 只按“减重/控糖”大类打标签，缺少具体干预场景。
5. 没区分内部话术和可发客户话术。
6. 没做评估问题集，开发上线后不知道召回准不准。
7. 过早追求自动化标注，导致标签体系还没稳定就批量污染。

---

## 12. 推荐结论

业务侧现在最值得做的不是“尽快把所有资料搬进系统”，而是先做一批高质量、已脱敏、已标注、已审核的 P0 试点语料。

最通顺的链路是：

1. 用飞书多维表格做收集池。
2. 按 10 个高频场景整理小样本。
3. 每条内容补标题、摘要、标签、安全级别、可见范围。
4. 图片补 alt text，视频补摘要和转写。
5. 教练主管审核后冻结 P0。
6. 开发将 P0 语料导入话术管理、素材库和 RAG 索引。
7. 用评估问题集持续修正召回质量。

这条路比“先全量导入再慢慢修”慢一点点，但会让后面的 AI 教练回答更准、更稳，也更容易验收。
