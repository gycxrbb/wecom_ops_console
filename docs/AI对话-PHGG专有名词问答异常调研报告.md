# AI 对话 PHGG 专有名词问答异常调研报告

> 日期：2026-05-15  
> 触发问题：教练提问“phgg怎么服用？”时，AI 回复偏离问题本身，疑似优先在当前客户预加载上下文里查找 `phgg`。  
> 调研范围：CRM AI 对话准备链路、Prompt 分层、Profile 上下文预加载、RAG 检索、外部搜索接入必要性。  
> 结论口径：客户档案是 `official customer truth`；RAG/搜索是 `support knowledge`；补剂/药品用法属于医疗敏感边界，不能把模型常识或搜索片段写成 `formal medical instruction`。

---

## 1. 结论摘要

这个问题不建议只靠“继续加长底层 SYSTEM prompt”解决，也不建议一上来把所有 AI 对话都接到公网搜索。

推荐方案是分三层推进：

1. **P0：先修 Prompt 与上下文优先级**
   - 明确告诉模型：当问题包含陌生缩写、营养补剂、药品、产品名并询问“是什么 / 怎么吃 / 用量 / 禁忌”时，第一步是识别术语本身，不要先在客户档案里硬找同名字段。
   - 客户档案只用于个性化安全边界，例如过敏、疾病、用药、孕哺、肠胃症状、血糖风险；不能替代通用术语解释。

2. **P1：接入“内部可信知识检索”，而不是直接裸接公网搜索**
   - 为 PHGG、膳食纤维、益生元、代餐、补剂、药品边界等建立内部 `knowledge_card`。
   - 通过现有 RAG 体系召回，并把召回结果标为 `support knowledge`。
   - 没有内部证据时，AI 必须提示“当前知识库没有可靠资料”，不能从客户档案猜。

3. **P2：为冷门术语增加受控外部搜索**
   - 只在内部 RAG 无命中、且用户问题属于术语解释/补剂用法/禁忌场景时触发。
   - 搜索源应走白名单和证据分级，例如医学论文、指南、监管/机构资料、产品说明书；搜索结果进入摘要层后再给 LLM，不让网页原文直接污染 prompt。

一句话判断：**底层 SYSTEM prompt 要改，但它只是止血；真正稳定的解法是“术语意图识别 + 可信知识检索 + 客户上下文只做安全个性化”。**

## 2. 当前系统已经能做什么

### 看见

教练在客户档案页发起 AI 对话，系统会先准备客户上下文，再调用模型生成回答。当前链路已经包括：

| 能力 | 当前状态 | 关键文件 |
| --- | --- | --- |
| 客户档案预加载 | 已运行 | `app/crm_profile/services/profile_context_cache.py` |
| 13 模块 Profile 上下文 | 已运行 | `app/crm_profile/services/profile_loader.py` |
| Prompt 分层组装 | 已运行 | `app/crm_profile/services/prompt_builder.py` |
| AI 对话流式输出 | 已运行 | `app/crm_profile/services/ai/__init__.py` |
| RAG 检索注入 | 已运行，但主要依赖内部语料 | `app/rag/retriever.py` |
| 上下文快照审计 | 已运行 | `app/crm_profile/services/audit.py` |

### 理解

当前模型看到的内容大致是：

```text
平台护栏 + 健康教练核心规则
+ 当前场景 prompt
+ 当前客户 Profile 上下文
+ RAG 知识库参考（如果召回成功）
+ 教练补充信息（如果有）
+ 用户本轮问题
```

这条链路适合回答“这个客户最近血糖如何”“她晚餐怎么调整”“这个客户属于哪个群”等围绕当前客户的问题。

但它对“PHGG 是什么、怎么服用”这类**通用术语/补剂知识问题**存在天然偏差：系统默认所有问题都围绕当前客户展开。

## 3. 异常现象复盘

用户问题：

```text
phgg怎么服用？
```

期望 AI 先理解：

```text
PHGG = partially hydrolyzed guar gum，部分水解瓜尔胶，是一种水溶性膳食纤维/益生元类补充剂。
```

然后再进入安全边界：

```text
一般服用方式、从低剂量开始、随水冲服、留意腹胀腹泻、特殊人群和正在用药者需医生确认。
```

实际偏差：

```text
AI 倾向先在当前客户档案上下文中寻找 phgg 相关字段或线索。
如果客户上下文里没有 PHGG，模型可能转向客户饮食、血糖、肠胃、执行情况等内容，导致答非所问。
```

这说明当前系统把“当前客户上下文”当作几乎所有问题的第一事实域，而没有先区分：

1. 这是客户事实查询。
2. 这是健康教育/术语解释。
3. 这是补剂/药物用法咨询。
4. 这是需要医生确认的医疗边界问题。

---

## 4. 根因分析

### 4.1 Prompt 当前强约束“围绕当前客户”

`platform_guardrails.md` 和 `health_coach_core.md` 中多处强调：

1. 当前对话始终围绕当前客户展开。
2. 回复必须引用客户实际数据和信息。
3. 必须优先尊重客户安全档案。

这些规则对客户运营场景是正确的，但对“陌生缩写解释”会产生副作用：模型被训练成“先找客户上下文”，而不是“先判断问题类型”。

### 4.2 Context Header 没有声明“客户上下文不是词典”

`context_header.md` 当前主要说明：

1. 当前客户真实姓名。
2. “你”默认指当前客户。
3. 以下是系统生成的客户上下文。
4. 已排除敏感身份字段。

缺少一条关键边界：

```text
客户上下文只代表当前客户事实，不是通用医学/营养/产品知识库。
当问题涉及陌生术语、补剂、药品、产品名时，不要因为客户上下文没出现该词就认为客户没有使用，或把该词解释为客户字段。
```

### 4.3 RAG 是内部知识库，不是公网搜索

`_prepare_ai_turn_async()` 会调用：

```python
retrieve_rag_context(
    customer_id=customer_id,
    message=rag_query,
    scene_key=scene_key,
    output_style=output_style,
    profile_context=profile_signals,
)
```

但当前 RAG 检索的主要来源是内部话术、素材和知识卡片。它不是实时互联网搜索。

如果内部知识库没有 PHGG 相关条目，RAG 会出现 `no_hit` 或无可见来源。当前回答链路不会强制告诉模型：

```text
本轮没有可靠知识库证据，请不要凭客户档案猜测 PHGG。
```

### 4.4 Query Intent 没有识别“补剂/术语/产品名”

`app/rag/intent_rules.py` 当前主要识别：

1. nutrition
2. points_operation

`app/rag/query_compiler.py` 主要处理：

1. 饮食、餐食、外卖、控糖等关键词。
2. 用药、药物、胰岛素、处方等医疗敏感关键词。

`phgg怎么服用` 的问题里没有中文“药物/用药/营养/膳食纤维”等显式关键词，容易被归入 `unknown`。结果是：

```text
系统没有把它识别为 supplement_usage / term_definition。
RAG 也没有针对补剂术语做强召回或外部搜索触发。
```

### 4.5 RAG 与 Profile 的权重关系不够清晰

当前 RAG 和 Profile 都作为 system messages 注入，但语义边界还不够强：

| 信息来源 | 正确身份 | 当前风险 |
| --- | --- | --- |
| Profile | 当前客户事实，`official customer truth` | 容易被模型当成所有问题的唯一依据 |
| RAG | 内部知识库参考，`support knowledge` | 无命中时模型不知道证据缺口 |
| 模型常识 | 未审计默认知识 | 容易生成没有来源的补剂建议 |
| 外部搜索 | 外部证据，`support evidence` | 若裸接入，存在来源质量和 prompt 污染风险 |

## 5. 是否需要优化底层 SYSTEM prompt

需要，但只能作为 P0 止血。

### 5.1 应该加的 Prompt 规则

建议在 `app/crm_profile/prompts/base/platform_guardrails.md` 或 `context_header.md` 增加规则：

```text
## 术语与补剂问题处理规则

当用户问题中出现陌生缩写、英文缩写、营养补剂、药品名、产品名，并询问“是什么 / 怎么用 / 怎么吃 / 服用 / 用量 / 禁忌 / 副作用”时：

1. 先判断这是“术语/补剂/用法问题”，不要先在客户档案中硬找同名字段。
2. 客户档案只用于安全个性化：过敏、疾病、处方、孕哺、肠胃症状、血糖风险、医生限制。
3. 如果 RAG/可信知识没有提供该术语资料，必须明确说明“当前知识库没有可靠资料”，不要编造。
4. 涉及剂量、药物相互作用、疾病治疗时，只能给一般健康教育和风险提醒，不能给替代医生的个体化用药/用量指令。
5. 回答结构应为：术语解释 → 一般使用原则 → 当前客户安全注意 → 需要医生/产品说明确认的边界。
```

### 5.2 为什么只改 Prompt 不够

只改 Prompt 有三个缺陷：

1. **知识来源仍然不稳定**  
   如果内部 RAG 没有 PHGG，模型可能依赖自身训练知识。对补剂用法这类问题，不能把模型常识当作正式依据。

2. **无法覆盖新词和商品名**  
   今天是 PHGG，明天可能是 NMN、肌酸、二甲双胍、某品牌代餐、某益生菌菌株。Prompt 不能穷举所有词。

3. **无法审计证据链**  
   项目需要知道 AI 是依据哪条知识卡、哪篇资料或哪个产品说明回答的。单靠 prompt 看不到证据来源。

结论：**Prompt 必须改，但它负责“不要误用客户上下文”；知识正确性要靠检索/搜索证据链。**

## 6. 是否需要接入搜索功能

需要，但不建议把公网搜索作为第一优先级直接接入所有对话。

### 6.1 推荐的检索分层

```text
用户问题
  ↓
意图识别
  ↓
是否为术语/补剂/药品/产品用法问题？
  ↓
内部 RAG 知识库检索
  ↓
命中可靠资料？
  ├─ 是：用内部知识卡 + 客户安全档案回答
  └─ 否：按配置触发受控外部搜索
          ↓
        来源过滤、摘要、证据分级
          ↓
        作为 support evidence 注入模型
```

### 6.2 哪些问题应触发外部搜索

建议只对以下场景触发：

| 场景 | 示例 | 是否触发搜索 |
| --- | --- | --- |
| 陌生英文缩写 + 怎么吃 | `phgg怎么服用？` | 是 |
| 补剂/保健品 + 禁忌 | `肌酸肾不好能吃吗？` | 是 |
| 药品 + 用量/相互作用 | `二甲双胍和益生菌能一起吃吗？` | 是，但回答必须转医生 |
| 品牌产品说明 | `XX代餐粉一天喝几次？` | 是，优先产品说明书 |
| 客户事实查询 | `她最近血糖怎么样？` | 否 |
| 运营话术生成 | `帮我写一段提醒打卡话术` | 否 |

### 6.3 搜索接入边界

外部搜索结果只能作为 `support evidence`，不能直接晋升为 `official`。

必须有这些门禁：

1. 来源白名单或来源分级。
2. 搜索摘要层，禁止网页原文直接拼进 system prompt。
3. 每条证据带来源、日期、适用人群、证据类型。
4. 医疗敏感问题默认加医生确认边界。
5. 前端或审计中记录：本次是否使用外部搜索、使用了哪些来源。

## 7. PHGG 示例应如何回答

以下不是要固化成唯一答案，而是用于说明目标行为。

```text
【问题回答】
PHGG 通常指 partially hydrolyzed guar gum，也就是部分水解瓜尔胶，属于水溶性膳食纤维/益生元类补充剂。它不是降糖药，也不是治疗药物。

【一般使用原则】
常见做法是按产品说明从小剂量开始，溶于水、牛奶或常温饮品中饮用，观察腹胀、排气、腹泻或便秘变化。若用于 IBS、便秘、血糖管理辅助等场景，应结合客户肠胃情况和医生意见。

【当前客户安全注意】
如果客户有明显胃肠道疾病、近期腹泻、肠梗阻风险、孕哺期、正在服药或医生限制膳食纤维，需要先确认医生意见；如果客户正在控糖，也要提醒这不是替代药物或饮食管理的方案。

【客户沟通话术】
PHGG 一般是水溶性膳食纤维，不是药，可以先按产品说明少量开始，用水冲开喝，看看肚子胀不胀、排便有没有变化。您如果最近肠胃不舒服、正在吃药或者医生有特别交代，就先别自己加量，我们先帮您确认一下更稳妥。
```

可参考资料：

1. Nutrition & Metabolism 的随机双盲研究中，PHGG 被定义为 known prebiotic fiber，研究使用 6 g/day 用于 IBS 人群，并观察到腹胀/排气症状改善；该研究不等于所有客户的个体化剂量建议。  
   来源：https://link.springer.com/article/10.1186/s12986-016-0070-5

2. EFSA 曾对 PHGG 相关健康声称进行科学意见评估，这类资料适合作为证据分级参考，但不能直接替代产品说明或医生建议。  
   来源：https://www.efsa.europa.eu/en/efsajournal/pub/2254

## 9. 不建议的方案

### 9.1 不建议只把 SYSTEM prompt 写得更强

原因：

1. 容易继续堆规则，长期不可维护。
2. 无法提供 PHGG 这类术语的可靠证据。
3. 对新补剂/新产品/品牌名无泛化保障。
4. 无法审计回答依据。

### 9.2 不建议无差别接入公网搜索

原因：

1. 搜索结果质量不稳定。
2. 广告、营销页、论坛经验可能污染回答。
3. 补剂/药品用法有安全风险。
4. 成本和延迟会明显上升。

### 9.3 不建议把搜索结果写成 official

搜索结果只能是 `support evidence`。即使搜索到一篇论文，也不能把它变成当前客户的正式医疗建议。

## 10. 验收标准

### 10.1 功能验收

| 测试问题 | 期望表现 |
| --- | --- |
| `phgg怎么服用？` | 先解释 PHGG；不硬查客户上下文；有安全边界 |
| `PHGG是什么？` | 术语解释清楚；说明属于膳食纤维/益生元类 |
| `她可以吃PHGG吗？` | 结合客户安全档案；如信息不足，提示缺口 |
| `PHGG和二甲双胍能一起吃吗？` | 不做确定性用药相互作用结论；建议医生/药师确认 |
| `这个客户最近睡眠怎么样？` | 仍然优先使用客户健康摘要，不走外部搜索 |

### 10.2 审计验收

每次回答应能追踪：

1. 用户问题被识别为什么意图。
2. 是否使用 Profile。
3. 是否使用 RAG。
4. RAG 是否无命中。
5. 是否触发外部搜索。
6. 使用了哪些证据来源。
7. 是否触发医疗安全边界。

### 10.3 质量验收

1. 不把客户档案中的缺失字段解释成 PHGG。
2. 不把搜索/RAG 支持资料写成当前客户 official 事实。
3. 不给个体化剂量、停药、换药建议。
4. 回答可转述给客户，但保留教练复核边界。

---

## 11. 当前阶段判断

### 已经能做什么

系统已经有 AI 对话、Profile 上下文、RAG 注入和审计基础，具备改造条件。

### 半能做什么

系统有 RAG，但偏内部运营话术和知识卡；对 PHGG 这类补剂术语，如果没有提前入库，就不能稳定提供可靠解释。

### 还不能做什么

系统目前没有受控外部搜索工具，也没有专门的补剂/术语意图路由。不能稳定处理所有新出现的英文缩写、商品名、补剂用法问题。

### 当前 blocker

1. Prompt 没有明确“客户上下文不是词典/百科”。
2. RAG 缺少补剂/术语知识卡。
3. 意图识别没有 `supplement_usage` / `term_definition`。
4. 外部搜索没有安全门禁和证据分级。

### 是否偏离蓝图

没有偏离。这个问题暴露的是 AI 对话从“客户画像问答”扩展到“健康教育问答”时，证据源和上下文优先级需要升级。

### 哪几条线最值得推进

1. **Prompt 止血线**：先让模型别在客户上下文里硬找 PHGG。
2. **术语知识库线**：补齐 PHGG 等常见补剂/营养术语卡片。
3. **意图路由线**：把客户事实查询、术语解释、补剂用法、医疗边界拆开。
4. **受控搜索线**：作为内部知识库无命中的兜底，不作为默认主路径。

---

## 12. 建议下一步

1. 先做 P0 Prompt 止血，改动小、风险低，能马上缓解“在客户上下文里找 PHGG”的问题。
2. 同步整理第一批补剂/营养术语知识卡，把 PHGG 纳入内部 RAG。
3. 再做 `supplement_usage` / `term_definition` 意图识别，确保问题被路由到正确证据域。
4. 最后评估受控外部搜索，不要让公网搜索直接进入正式回答链路。
