## 二、v2.1 核心改动详情

### 2.1 L2 核心角色 — 新增"客户沟通话术规范"章节

这是本轮最重要的改动，在 L2 中新增了完整的话术规范，全局生效于所有场景。

#### 核心原则（新增）

```
- 回复必须引用客户的实际数据和信息，不能凭空分析
- 给出的建议必须具体可落地，不能泛泛而谈
- AI 要让教练感受到：这个平台是懂教练日常工作的、是懂客户状况的
```

#### 称呼规则（新增）

根据客户年龄和性别自动选择称呼：

| 客户年龄段 | 女性称呼 | 男性称呼 |
|------------|----------|----------|
| 60 岁以上 | XX阿姨 | XX叔叔 |
| 35-60 岁 | XX姐 | XX哥 |
| 25-35 岁 | XX姐 / XX（名字） | XX哥 / XX（名字） |
| 25 岁以下 | XX（名字） | XX（名字） |

#### 话术四大必须（新增）

1. **让客户感受到被关注**：话术中必须引用客户近期数据或变化，不能写成通用模板
2. **亲和力 + 专业度**：语气像关心你的朋友，但不是讨好；用大白话，不堆术语
3. **必须包含"怎么做"**：具体到客户能直接照做的指引，不说空话
4. **篇幅与结构**：80-200 字，一段话完成，正向期待收尾

#### 输出模板更新

```
【结论/判断】围绕用户的问题的核心发现是什么？
【关键依据】2-3 条支撑数据或事实，必须引用客户上下文中的实际数据
【行动建议】教练围绕用户的问题具体可执行的操作建议，需要详细的步骤指导
【客户沟通话术】教练可直接发给客户的沟通文案（必选项！不可省略！）
```

关键变化：话术从"可选"变为"必选"，分析依据明确要求引用上下文实际数据。

### 2.2 L3 场景策略 — 6 个场景全部更新

每个场景的输出结构统一增加【客户沟通话术】必选项，且 Few-shot 示例全部按 v2.1 标准重写：

| 场景 | 话术示例要点 | 称呼示例 |
|------|-------------|----------|
| 餐评 | 张姐，引用近期血糖数据→替代方案→明天帮配食谱 | 42 岁女性 → 张姐 |
| 数据监测 | 王哥，引用具体数值→归因于睡眠推迟→每天早睡 15 分钟 | 48 岁男性 → 王哥 |
| 异常干预 | 李阿姨，先关心有无症状→给睡前加餐方案→会同步医生 | 65 岁女性 → 李阿姨 |
| 问题答疑 | 陈姐，解释正常现象→30 分钟后再测→表扬观察力 | 38 岁女性 → 陈姐 |
| 周月复盘 | 赵姐，引用 4 周累计数据→肯定亮点→下周午餐微调 | 40 岁女性 → 赵姐 |
| 长期维护 | 刘哥，回顾 8 周进展→攻克午餐场景→讨论自主管理 | 50 岁男性 → 刘哥 |

每个示例都体现了三个核心要求：
- **引用上下文**：话术中引用了客户的具体数据或近期变化
- **具体"怎么做"**：不是"注意饮食"而是"米饭减到半碗，加一份炒青菜"
- **让客户感受到被关注**：不是通用模板，而是"我看到您的血糖从 6.3 降到了 5.9"

### 2.4 prompt_builder.py — message 结构优化

产品经理将动态上下文（RAG + profile_note + scene_hint）从 L4 context_header 中拆分为独立 message，优化 prompt cache 命中率：

```
改前：messages = [system(L1+L2+L3), system(L4+RAG+L5+L5.5), user(L6)]
改后：messages = [system(L1+L2+L3), system(L4), system(RAG+L5+L5.5), user(L6)]
```

L1+L2+L3 和 L4 相对稳定，cache 命中率高；动态部分独立成 message，变化时只需重算最后一层。

## 四、下一步建议

### 4.1 优先做：实际效果试跑

v2.1 是提示词密集改动，需要验证模型是否真正遵循了新增的规范：
- 每个场景用 5-10 个真实教练问题测试
- 重点验证：称呼是否正确？话术中是否引用了客户数据？"怎么做"是否具体？
- 根据试跑结果微调话术规范的措辞

### 4.2 中期做：DB 存储 + 热更新

当前改提示词需要改文件 + 重启服务。建议：
1. 新建 `prompt_templates` 表，启动时从 .md seed
2. registry.py 用 TTL 缓存替代 lru_cache
3. 加管理 API `POST /api/v1/admin/prompts/{key}` 保存到 DB

### 4.3 远期做：A/B 测试 + 评估框架

需要有足够使用量后再做。
