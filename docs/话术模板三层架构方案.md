# 话术管理：三级分类体系 + RAG 配置校验方案

> 日期：2026-05-03
> 文档状态：plan / candidate design

---

## 一、文档目标

将话术管理的侧边栏从当前 **2 级分类 + 场景标题** 改为 **3 级分类 + 具体话术**，并完善 RAG 配置编辑时的历史数据校验与覆盖逻辑。

本文档为方案真值，实际实现以最终代码为准。

---

## 二、当前状态判断

### 已经能做什么
- 两级分类树（L1 大类 > L2 子类）可浏览、可编辑（重命名/删除/新建/移动）
- 场景可归类到 L2 子类下
- 每个场景下的多风格话术可编辑保存
- RAG 元数据可编辑，保存后触发增量索引
- CSV 批量导入话术

### 半能做什么
- `SpeechCategory` 模型已支持 `level` 字段和自引用 `parent_id`，理论上可扩展为 3 级，但所有代码只处理 L1/L2
- RAG 配置可编辑，但打开时未校验历史数据是否存在、无覆盖确认机制

### 还不能做什么
- 无法创建/管理 L3（三级分类）
- 侧边栏无法展示 3 级嵌套
- 无法按 Excel 定义的标准 3 级分类批量 seed
- RAG 配置无 diff 预览、无历史追溯

### 当前 blocker
- `SCENE_CATEGORY_MAP` 硬编码了 `scene_key → L2 分类名` 的映射，与新 3 级体系不兼容
- 前端 `sidebarTree` 计算属性只构建 L1 > L2 > scenes 结构
- 后端 `/categories` 只返回 2 级树

## 三、目标层级结构

Excel 文件 `docs/shujubiaozhu/question-category.xlsx` 定义了标准分类：

```
一级分类（L1）  >  二级分类（L2）   >  三级分类（L3）     >  具体话术（叶子节点）
健康管理类       饮食管理            饮食计划咨询          [话术1, 话术2, ...]
健康管理类       饮食管理            食物选择建议          [话术1, 话术2, ...]
健康管理类       体重管理            减重速度咨询          [话术1, 话术2, ...]
系统操作类       功能操作            打卡功能操作          [话术1, 话术2, ...]
社区运营类       积分管理            积分获取咨询          [话术1, 话术2, ...]
服务支持类       用户协助            操作指导协助          [话术1, 话术2, ...]
```

共 **4 个 L1**、**16 个 L2**、**64 个 L3**（每 L2 下 4 个 L3）。

侧边栏最终效果：

```
▼ 健康管理类
  ▼ 体重管理
    ▼ 减重速度咨询          ← L3 可展开/折叠
      减重健康速度           ← 具体话术（叶子）
    减重原理咨询
    体重目标设定
    体重监测问题
  ▼ 饮食管理
    饮食计划咨询
    食物选择建议
    ...
▼ 系统操作类
  ...
```

## 四、方案设计

### 4.1 数据模型改动

**`SpeechCategory` 表**：已支持自引用 `parent_id` + `level`，无需 DDL 变更。只需新增 `level=3` 的行。

**`SpeechTemplate` 表**：`category_id` 当前指向 L2 子类。改为指向 **L3 子类**。

| 改动前 | 改动后 |
|--------|--------|
| `category_id → L2 (level=2)` | `category_id → L3 (level=3)` |

**兼容策略**：在 `category_id` 指向的行上检查 `level`，向后兼容旧的 L2 数据。

## 五、RAG 配置校验方案

### 5.1 当前问题

打开 RAG 配置对话框时，直接用模板的 `metadata_json` 填充表单。没有校验：
1. 当前模板是否已有 RAG 配置
2. 用户编辑的是覆盖还是追加
3. 保存时静默覆盖，无法追溯旧值

### 5.2 改进设计

#### 5.2.1 打开对话框时：校验已有配置

**文件**: `frontend/src/views/SpeechTemplates/composables/useSpeechTemplates.ts` → `openRagMetaDialog`

```
打开对话框 → 检查 metadata_json 是否非空
  → 已有配置：显示提示 "该话术已有 RAG 配置，编辑后保存将覆盖现有配置"
  → 无配置：正常空表单
```

在前端实现，不需要新的 API 调用。`openRagMetaDialog` 已有 `tpl.metadata_json`。

#### 5.2.2 保存时：diff 预览

在保存前展示变更对比：

```
原值 → 新值
  summary: "旧摘要" → "新摘要"
  tags: [A, B] → [A, C]
  safety_level: "low" → "medium"
```

实现方式：
- `openRagMetaDialog` 时快照 `originalMeta = JSON.parse(JSON.stringify(meta))`
- `handleSaveRagMeta` 前计算 diff
- 在确认对话框中展示 diff，用户确认后再提交

#### 5.2.3 后端：保存前校验 + 原子更新

**文件**: `app/routers/api_speech_templates.py` → `PATCH /{template_id}/rag-meta`

```python
@router.patch("/{template_id}/rag-meta")
def update_rag_meta(template_id: int, req: SpeechRagMetaReq, ...):
    tpl = db.query(SpeechTemplate).get(template_id)
    if not tpl:
        raise BizError("模板不存在")

    old_meta = json.loads(tpl.metadata_json) if tpl.metadata_json else {}

    # 合并策略：请求中提供的字段覆盖旧值，未提供的字段保留
    merged = {**old_meta, **req.dict(exclude_none=True)}

    tpl.metadata_json = json.dumps(merged, ensure_ascii=False)
    db.commit()

    # 增量索引
    index_single_speech_template(db, template_id)

    return {"metadata_json": merged}
```

**合并策略**：
- 请求中的非 None 字段覆盖旧值
- 请求中未提供的字段保留旧值
- 如果要删除某个字段，传空值（空字符串/空数组）

#### 5.2.4 前端实现细节

```typescript

  // 展示 diff 确认框

  await ElMessageBox.confirm(`以下配置将被更新：\n${diffText}`, '确认保存', { type: 'info' })
  // 调用 API...
}
```

## 八、风险与注意事项

1. **数据迁移**：现有 2 级分类数据需要迁移到 3 级。`category_id` 当前指向 L2，迁移后需指向 L3。需写一次性的迁移脚本或在 `schema_migrations.py` 中处理。
2. **SCENE_CATEGORY_MAP 精确度**：Excel 中 L3 名称需与实际话术场景精确匹配，映射可能需要多轮调整。
3. **向后兼容**：`/scenes` 和 `/categories` 的响应结构变化可能影响其他调用方（如 AI 对话场景选择）。需检查所有消费方。
4. **Excel 中 "细分分类" 列**：当前为可选项，本方案暂不使用该列。如果未来需要 4 级分类，可扩展 `level=4`。
5. **已有 RAG 数据**：阶段 3 的 diff 逻辑对已保存的 RAG 配置生效，不会丢失历史数据。
