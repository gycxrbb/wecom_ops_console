## 一、执行摘要

2025-2026年，AI Agent技术已从单点工具调用演进至**多Agent协作编排**阶段。Gartner预测，到2026年底，40%的企业应用将集成任务专属AI Agent（2025年不足5%）。在医疗健康领域，Agent技术已在慢病管理、患者随访、个性化干预等场景取得实质性落地成果。

本报告针对贵公司健康教练业务场景（慢病干预、减重管理、斑块逆转、餐评、习惯打卡、处方开具、话术沉淀等），系统调研了当前主流开源Agent架构、最新工程实践，并提出**面向未来业务扩展**的六层可演进架构方案。

**核心结论：**
1. 推荐以**LangGraph**为核心编排框架，**MCP**为工具接入标准
2. 采用**六层架构分层**设计（基础设施→Agent层→编排层→应用层→评测观测层→治理合规层），确保新业务接入无需修改已有系统代码
3. 构建**Agent注册中心**与**模型网关**，实现Agent的独立部署、动态发现、版本管理与多模型容灾
4. 处方开具等高风险操作必须设置**Human-in-the-Loop**审核节点
5. 建议按"MVP→多Agent协作→知识沉淀与评测→规模化与合规"四阶段推进

## 二、开源Agent架构现状与趋势（2025-2026）

### 2.1 主流框架全景图

当前开源Agent框架已形成清晰的梯队格局，按架构范式可分为四大类：

| 框架 | 架构范式 | 核心特色 | 适用场景 | GitHub Stars |
|------|---------|---------|---------|-------------|
| **LangGraph** | 图状态机 | 显式图结构、状态持久化、HITL、时间旅行调试 | 复杂有状态工作流、生产级系统 | 24.8k |
| **CrewAI** | 角色协作 | 角色定义、任务委托、层级/顺序工作流 | 快速原型、内容/研究流水线 | 44.3k |
| **OpenAI Agents SDK** | Handoff交接 | 轻量级、Guardrails、Tracing、多Agent交接 | OpenAI生态生产部署 | 19k |
| **Claude Agent SDK** | 子Agent+技能 | Hooks系统、MCP原生支持、Skills、Subagents | Anthropic生态复杂任务 | - |
| **AutoGen/AG2** | 对话循环 | 多Agent对话、代码执行、群聊 | 研究型对话系统、代码生成 | 54.6k |

> **核心趋势：** 2025-2026年，Agent发展重心已从"能对话"转向"能可靠执行复杂任务"。多Agent协作、显式状态管理、人类介入机制（HITL）成为生产级系统的三大标配。与此同时，**Agent间通信协议**（Google A2A、ACP）和**Agent评测体系**（promptfoo、DeepEval、Ragas）正在快速成熟，成为生产系统不可或缺的组成部分。

### 2.2 架构范式深度解析

#### 2.2.1 LangGraph：生产级状态驱动架构

LangGraph是LangChain生态的图编排框架，将Agent工作流建模为**显式状态机**——每个节点是函数，边定义控制流。其核心优势在于：

- **状态持久化与Checkpointing**：Agent可在任意节点暂停、恢复，支持长周期任务（如28天减重干预计划）
- **Human-in-the-Loop（HITL）**：在关键决策点（如处方开具、异常指标预警）强制人工审核
- **条件分支与重试**：根据用户血糖数据动态调整干预路径，失败时自动回退
- **时间旅行调试**：可回溯到任意历史状态排查问题
- **子图（Subgraph）复用**：可将通用流程（如数据采集、医生审核）封装为可复用子图，新业务直接组合调用

目前Cisco、Uber、LinkedIn、BlackRock、JPMorgan等企业已采用LangGraph部署生产Agent。

**典型代码结构：**

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict

class HealthCoachState(TypedDict):
    user_id: str
    messages: list
    next_action: str
    risk_level: str
    agent_version: str  # 支持Agent版本追踪

workflow = StateGraph(HealthCoachState)
workflow.add_node("intake", intake_node)          # 用户信息收集
workflow.add_node("assess", assess_node)           # 健康评估
workflow.add_node("intervene", intervene_node)     # 干预方案生成
workflow.add_node("review", human_review_node)     # 医生审核节点（HITL）
workflow.add_node("evaluate", quality_eval_node)   # 输出质量评估节点

workflow.add_edge("intake", "assess")
workflow.add_conditional_edges("assess", route_by_risk)
workflow.add_edge("intervene", "review")
workflow.add_edge("review", "evaluate")            # 审核后自动质检
workflow.add_edge("evaluate", END)
```

#### 2.2.2 CrewAI：角色化多Agent协作

CrewAI采用**角色导向设计**，将Agent定义为具有特定角色、目标和工具的团队成员，通过任务委托实现协作。其优势在于：

- **低门槛**：声明式定义Agent角色，适合快速验证业务逻辑
- **层级与顺序工作流**：支持Manager-Worker模式（如"主诊医生Agent"分配任务给"营养师Agent"、"运动教练Agent"）
- **内置反馈机制**：Reviewer Agent可审核生成内容质量

**适用场景**：MVP快速验证、内容生成、研究分析、多专家会诊模拟。对于贵公司的"餐评生成"、"话术沉淀"等场景，CrewAI的角色分工模式非常契合。

> **建议**：MVP阶段可用CrewAI快速验证业务逻辑，验证通过后迁移到LangGraph构建生产级系统。两者的Agent定义可共用同一套Prompt模板，降低迁移成本。

#### 2.2.3 OpenAI Agents SDK & Claude Agent SDK：大厂原生方案

- **OpenAI Agents SDK**（2025年3月发布）：核心抽象是**Handoff（交接）**——Triage Agent接收用户输入，判断意图后交接给专科Agent（如"减重Agent"、"血糖管理Agent"）。内置Guardrails进行输入/输出校验，Tracing提供端到端可观测性。

- **Claude Agent SDK**：Anthropic官方框架，与Claude Code同架构。原生支持**MCP协议**、Skills（可复用技能）、Subagents（子Agent），Hooks系统允许在关键节点插入自定义逻辑。

> **关键洞察**：大厂原生SDK的优势在于与自家模型的深度集成，劣势在于供应商锁定。对于贵公司的场景，建议**编排层使用LangGraph（框架无关），通过模型网关对接多家LLM供应商**，避免单一依赖。

### 2.3 关键架构演进趋势

1. **从Chain到Graph**：线性Chain已无法满足复杂医疗决策需求，图结构成为标配
2. **从单Agent到多Agent协作**：单Agent上下文窗口瓶颈明显，角色分工成为标准
3. **从无状态到有状态**：状态持久化是Agent从"聊天工具"走向"行动系统"的分水岭
4. **从封闭到开放**：MCP协议统一工具调用标准，实现"一次实现，处处可用"
5. **从不可测到可观测**：Agent评测与可观测性从"可选"变为"必须"，无评测的Agent无法上线生产
6. **从静态编排到动态发现**：Agent注册中心与服务发现机制使系统具备运行时扩展能力

## 三、Agent工程实践最新趋势

### 3.1 MCP协议：Agent工具生态的"USB-C"标准

**Model Context Protocol（MCP）**由Anthropic于2024年底开源，现已成为AI Agent连接外部工具的事实标准。其核心思想是**标准化AI模型与数据源的交互接口**，类比USB-C统一充电和数据传输。

**MCP架构组件：**
- **MCP Host**：发起请求的AI应用（如健康教练系统）
- **MCP Client**：与Server建立连接，通常1:1关系
- **MCP Server**：提供Tools（工具）、Resources（资源）、Prompts（提示模板）
- **本地/远程资源**：文件系统、数据库、第三方API

**对贵公司的价值：**
- 将"血糖仪数据查询"、"体检报告解析"、"处方生成"等能力封装为独立MCP Server
- 未来新增设备或数据源时，无需修改Agent核心代码，只需接入新的MCP Server
- 医生端、教练端、用户端可共享同一套MCP工具生态
- **新业务线复用**：如果未来拓展心理健康、产后恢复等业务，已有的血糖MCP Server、体检报告MCP Server可直接复用

**MCP Server开发规范建议：**

```
mcp-servers/
├── blood-glucose-server/      # 血糖数据MCP Server
│   ├── tools/                 # 工具定义（查询、统计、预警）
│   ├── resources/             # 资源定义（历史数据、趋势报告）
│   ├── tests/                 # 单元测试与集成测试
│   ├── config.yaml            # 配置（连接参数、超时策略）
│   └── README.md              # 接口文档
├── health-checkup-server/     # 体检报告解析MCP Server
├── prescription-server/       # 处方模板MCP Server
├── knowledge-base-server/     # 知识库检索MCP Server
├── wearable-device-server/    # 可穿戴设备MCP Server（预留）
└── his-integration-server/    # 医院HIS对接MCP Server（预留）
```

每个MCP Server必须包含：**接口文档、单元测试、安全审计记录、降级策略说明**。

OpenAI Agents SDK、Claude Agent SDK、LangGraph均已原生支持MCP。

### 3.2 Agent间通信协议：A2A与ACP

报告的初版未涉及Agent间通信标准，但这对于**未来多系统互联**至关重要。

**Google A2A（Agent-to-Agent）协议**（2025年4月发布）：
- 定义了Agent之间发现、协商、任务委派的标准协议
- 基于Agent Card（JSON描述文件）实现Agent能力声明与动态发现
- 支持跨组织、跨平台的Agent协作

**对贵公司的意义：**
- 当前阶段，系统内的Agent通过LangGraph图编排通信即可
- 但需要**预留A2A兼容接口**：每个Agent对外暴露标准化的Agent Card描述
- 未来如果需要与医院系统、药房系统、保险系统的AI Agent互通，A2A兼容将大幅降低对接成本

**建议**：在Agent层为每个Agent定义标准化的能力描述文件（Agent Card Schema），包含Agent名称、能力声明、输入输出格式、权限范围。当前阶段作为文档规范，未来可直接作为A2A协议的注册信息。

### 3.3 记忆系统：从RAG到分层记忆架构

2025年，Agent的核心瓶颈已从模型规模转向**记忆能力**。当前主流大模型每次对话结束即清空上下文，导致70%-90%的推理token被用于重传历史信息。

**记忆与RAG的本质区别：**
- **RAG**：按需检索外部静态知识，无状态，适合问答与文档查询
- **记忆系统**：跨会话状态积累，记录用户偏好、历史决策、关系演化，具备时间感、可衰减、可合并、可遗忘特征

**四层记忆架构（生产级标准）：**

| 层级 | 类型 | 存储内容 | 检索方式 | 技术实现 |
|------|------|---------|---------|---------|
| **工作记忆** | 短期 | 当前对话上下文 | 全量加载 | LLM上下文窗口 |
| **情景记忆** | 中期 | 时间戳事件（某日的血糖异常、某次餐评） | 时间+语义混合检索 | 向量数据库+时间索引 |
| **语义记忆** | 长期 | 用户画像、偏好、禁忌、目标（"用户A怕苦，不爱吃芹菜"） | 语义相似度检索 | HNSW索引+重要性评分 |
| **程序记忆** | 持久 | 工具使用模式、标准操作流程（SOP） | 规则匹配 | 结构化存储 |

**Mem0等开源记忆框架**已实现单次分层提取与多信号检索（语义+关键词+实体），在LoCoMo基准测试中达到91.6分。

#### 3.3.1 记忆治理工程设计

记忆系统在工程实现上需要解决以下关键问题：

**写入权限控制：**
- 不是所有Agent都应直接写入长期记忆。建议设计"**记忆提议→审核→入库**"的三级流程
- 示例：餐评Agent可以将"用户对本次餐评反馈为不满意"写入情景记忆，但"用户胰岛素抵抗"这类医学判断只能由医生辅助Agent在医生确认后写入语义记忆

**一致性保障：**
- 多Agent并发写入时的冲突处理：采用"最后写入胜出（Last Write Wins）+ 重要性权重"策略
- 语义去重：定期运行记忆融合任务，合并语义相似的记忆条目

**过期与衰减：**
- 情景记忆：按时间衰减，近期记忆权重高，90天前的低频记忆自动归档
- 语义记忆：按重要性评分和访问频率决定保留与否
- 程序记忆：随SOP版本更新自动替换

**隐私与合规：**
- 用户记忆必须严格按用户隔离，禁止跨用户记忆泄露
- 记忆访问需记录审计日志（谁读了、谁写了、什么时间）
- 用户有权要求删除其全部记忆（GDPR/个保法合规）

**多租户隔离：**
- 如果未来服务多家医疗机构或健康公司，记忆系统必须支持租户级隔离
- 建议在记忆存储的Key设计中加入 `tenant_id` 前缀，确保物理隔离

### 3.4 Agentic RAG：从固定检索到自主决策

RAG技术已演进至第五代**Agentic RAG**（2025年起）。与传统RAG的固定单次检索不同，Agentic RAG让Agent自主决策：

- 判断当前召回内容是否足够
- 决定是否需要多轮/多跳检索
- 选择从哪个数据源检索（医学知识库、用户历史、最新指南）
- 评估生成答案的可靠性并自我纠错

对于贵公司的"健康答疑解惑"场景，Agentic RAG可实现：先检索用户历史病历→再检索医学知识库→若答案不确定则标记为"需医生确认"。

### 3.5 可观测性与安全护栏

**生产级Agent必须具备四大能力：**

1. **链路追踪**：使用LangSmith、Langfuse或AgentOps可视化Agent决策路径，设置断点调试
2. **输入/输出Guardrails**：对医疗建议进行安全校验，拦截危险指令（如"帮我跳过胰岛素"）
3. **沙盒执行**：处方生成等操作在隔离环境中预执行，确认无害后再提交
4. **审计追溯**：100%数据调用审计，满足医疗合规要求

### 3.6 Agent评测体系（新增）

Agent评测是从"Demo能跑"到"生产可上线"的分水岭。没有评测体系的Agent，每次改动都是盲人摸象。

**评测三层体系：**

| 评测层级 | 评测对象 | 评测方法 | 工具 |
|---------|---------|---------|------|
| **单元评测** | 单个Agent的单次输出 | 预定义测试用例 + LLM-as-Judge | promptfoo、DeepEval |
| **集成评测** | 多Agent协作流程的端到端表现 | 典型业务场景模拟 + 人工抽检 | 自建评测框架 + LangSmith |
| **业务评测** | 真实用户的健康干预效果 | A/B测试 + 业务指标追踪 | 自建数据分析平台 |

**评测维度（医疗场景特化）：**

- **安全性（一票否决）**：是否生成了可能危害用户健康的内容？是否在需要医生审核的场景中正确触发HITL？
- **准确性**：医学知识引用是否正确？数据计算是否准确？
- **一致性**：相同输入是否产生质量稳定的输出？
- **延迟**：用户等待时间是否在可接受范围内？（对话场景 < 3秒，处方场景 < 30秒）
- **成本**：单次交互的Token消耗是否在预算范围内？

**回归测试机制：**
- 每次修改Agent的Prompt或工具配置后，必须跑完整评测集
- 评测结果低于阈值时，自动阻止部署
- 评测集需要持续维护：收集生产环境中的bad case，补充进评测集

## 五、针对贵公司业务场景的Agent架构建议

### 5.1 架构设计原则

在进入具体方案之前，明确以下架构设计原则，这些原则将指导所有技术决策：

| 原则 | 含义 | 落地要求 |
|------|------|---------|
| **开闭原则** | 新业务接入不应修改已有系统代码 | Agent注册中心 + 动态路由 |
| **关注点分离** | 每个Agent只负责一个专业领域 | 粒度细分，职责单一 |
| **可评测** | 任何改动都必须能被量化评估 | 评测框架先于业务功能建设 |
| **安全兜底** | 高风险操作必须有人工审核 | HITL作为架构强制约束 |
| **供应商无关** | 不绑定单一LLM供应商 | 模型网关统一管理 |
| **数据合规** | 全链路数据可审计 | 审计日志 + 隐私隔离 |
| **渐进演化** | 架构支持从MVP到大规模生产的平滑演进 | 分层解耦，逐层完善 |

### 5.2 业务场景拆解与Agent映射

根据您描述的业务需求，建议将系统拆解为以下**细粒度专业化Agent集群**（比初版报告更细分）：

```

┌──────▼──────┐┌──▼────────┐┌▼─────────┐┌▼─────────┐┌▼─────────────┐
│ 习惯打卡    ││ 健康问答   ││ 话术沉淀  ││ 报告解读  ││ 随访管理      │

**拆分理由：**
- **餐评Agent和运动处方Agent职责完全不同**，合并会导致Prompt膨胀、工具冲突
- **话术沉淀Agent独立运行**，负责从对话中自动提取高质量问答对，不干扰其他Agent的主流程
- **报告解读Agent独立**，因为它是同步请求（用户上传报告后需要即时返回），而干预类Agent多为异步任务
- 未来新业务（如"心理健康Agent"、"产后恢复Agent"）只需开发并注册新Agent，Triage Agent的路由规则动态更新即可

### 5.4 核心组件详细设计

#### 5.4.1 模型网关（LLM Gateway）

模型网关是基础设施层的核心组件，统一管理所有LLM调用：

```
                    ┌─────────────────────────────────┐
                    │         模型网关 (LLM Gateway)    │
                    │                                 │
  Agent请求 ───────►│  1. Prompt模板渲染               │
                    │  2. 模型路由决策                  │
                    │  3. 请求发送 + 超时/重试          │
                    │  4. 响应校验（Guardrails）        │
                    │  5. Token用量统计                │
                    │  6. 审计日志记录                  │
                    └──────────┬──────────────────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
        ┌──────────┐    ┌──────────┐    ┌──────────┐
        │ Claude   │    │ GPT-5.4  │    │ 通义千问  │
        │ 3.7 Sonnet│   │          │    │ Qwen3    │
        └──────────┘    └──────────┘    └──────────┘
        (主模型)          (备模型)         (低成本模型)
```

**路由策略：**

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 餐评生成、话术推荐 | Claude 3.7 Sonnet | 医疗推理能力强，长上下文 |
| 简单问答、打卡提醒 | Qwen3（低成本模型） | 延迟低、成本低 |
| 处方方案生成 | Claude 3.7 Sonnet / GPT-5.4 | 高风险场景，需要最强推理能力 |
| 主模型不可用时 | 自动Fallback到备模型 | 保障系统可用性 |

**容灾策略：**
- 主模型响应超时（>10秒）→ 自动切换备模型
- 主模型连续3次失败 → 自动降级并告警
- 所有模型不可用 → 返回预设安全回复 + 人工通知

**成本优化：**
- 按Agent类型配置不同的模型等级（高频低风险Agent用低成本模型）
- Token预算控制：设置单次请求和日累计Token上限
- 定期生成成本报告，识别优化空间

#### 5.4.2 Agent注册中心（Agent Registry）

Agent注册中心是实现"开闭原则"的核心机制：

**每个Agent注册时提供：**

```yaml
agent_id: "meal-review-agent"
name: "餐评Agent"
version: "1.2.0"
description: "分析用户餐食照片/文字描述，生成营养评估与改善建议"

capabilities:
  inputs:
    - type: "image"          # 餐食照片
    - type: "text"           # 文字描述
    - type: "structured"     # 结构化数据（血糖、饮食偏好）
  outputs:
    - type: "text"           # 餐评结果
    - type: "structured"     # 营养数据（热量、碳水、蛋白质等）

tools:
  - "mcp://knowledge-base-server/nutrition-query"
  - "mcp://blood-glucose-server/recent-trend"

memory_permissions:
  read: ["semantic_memory", "episodic_memory"]
  write: ["episodic_memory"]   # 只能写情景记忆，不能修改用户画像

evaluation:
  test_suite: "evals/meal-review-agent/v1.2.0/"
  safety_threshold: 0.95      # 安全性评分阈值
  accuracy_threshold: 0.85    # 准确性评分阈值

deployment:
  strategy: "canary"          # 灰度策略：金丝雀发布
  canary_percentage: 10       # 初始灰度比例
  rollback_on_failure: true   # 失败自动回滚

agent_card:                   # A2A兼容的能力描述
  protocol_version: "1.0"
  authentication: "api_key"
```

**注册中心核心功能：**
- **动态发现**：Triage Agent通过注册中心查询可用Agent，无需硬编码
- **版本管理**：同一Agent可同时存在v1和v1.2，按用户分组灰度切换
- **健康检查**：定期探活，不可用的Agent自动从路由表中摘除
- **配置热更新**：修改Agent配置后无需重启服务，注册中心推送更新

#### 5.4.3 处方开具流程设计（关键HITL场景）

```
用户提交需求
    │
    ▼
Triage Agent 识别意图 → "需要生成处方"
    │
    ▼
数据采集Agent 收集近期指标（血糖、体重、运动、睡眠）
    │
    ▼
┌─── 多专科Agent并行生成处方草案 ───┐
│                                   │
│  饮食处方Agent ← 血糖数据 + 口味偏好 │
│  运动处方Agent ← 体能评估 + 时间安排 │
│  睡眠压力处方Agent                  │
│  营养素处方Agent ← 营养评估 + 交互作用│
│                                   │
└───────────────────────────────────┘
    │
    ▼
处方聚合Agent 整合为统一处方书
    │
    ▼
【安全预检查】处方安全Guardrails：
  - 药物/营养素交互作用检查
  - 剂量合理性检查
  - 禁忌症交叉检查
    │
    ▼
┌───【Human-in-the-Loop】医生审核节点 ───┐
│                                        │
│  医生端看到：处方草案 + 生成依据 + 风险标注│
│                                        │
│  操作选项：                              │
│  ✅ 确认通过 → 下发给用户                │
│  ✏️  修改后通过 → 记录修改原因            │
│  ❌ 驳回 → 返回对应Agent重新生成          │
│  📋 标记需讨论 → 进入多医生会诊流程        │
│                                        │
└────────────────────────────────────────┘
    │
    ▼
用户接收处方 → 执行跟踪Agent监测依从性
    │
    ▼
评测系统自动评估：处方质量 + 用户依从率 + 健康指标变化
```

**HITL设计要点：**
- 审核队列支持优先级排序（高风险用户优先审核）
- 超时提醒：处方审核超过2小时未处理，自动升级通知
- 所有审核决策（确认/修改/驳回）均记录审计日志
- 驳回原因反馈给对应Agent，用于优化下次生成质量

#### 5.4.4 记忆系统工程设计

**话术沉淀机制：**
1. **自动提取**：每次健康教练与用户的对话结束后，话术沉淀Agent自动提取高价值问答对
2. **质量评分**：Reviewer Agent对提取的话术进行准确性、完整性、时效性评分
3. **分类存储**：按疾病类型（糖尿病/减重/斑块逆转）、场景（餐评/运动/睡眠）存入向量库
4. **记忆融合**：当相似问题出现时，自动合并重复话术，更新时效性信息
5. **访问统计**：记录每条话术被引用的次数和效果反馈，用于持续优化排序

**用户个人记忆：**

```
用户画像语义记忆（写入需经医生辅助Agent审核）：
- 基础属性：年龄、性别、BMI、疾病史
- 行为偏好：饮食口味（忌辣/低盐）、运动习惯（晨跑/夜跑）、沟通风格（喜欢详细/简洁）
- 干预历史：过往餐评反馈、处方依从性、异常事件
- 目标追踪：减重目标、血糖控制目标、阶段里程碑
- 医学属性（高敏感）：诊断结论、用药信息、过敏史
```

**记忆读写权限矩阵：**

| Agent | 工作记忆 | 情景记忆 | 语义记忆 | 程序记忆 |
|-------|---------|---------|---------|---------|
| 餐评Agent | 读写 | 只读 | 只读（偏好） | 无 |
| 运动处方Agent | 读写 | 只读 | 只读（偏好） | 无 |
| 医生辅助Agent | 读写 | 读写 | 读写（经审核） | 只读 |
| 话术沉淀Agent | 只读 | 读写 | 只读 | 读写 |
| Triage Agent | 读写 | 只读 | 只读（画像摘要） | 无 |

#### 5.4.5 评测体系设计

**评测流水线：**

```
Agent代码/Prompt变更
    │
    ▼
自动触发评测流水线
    │
    ├── 单元评测（每个Agent独立评测集）
    │     ├── 安全性测试（一票否决）
    │     ├── 准确性测试
    │     ├── 边界条件测试
    │     └── 延迟测试
    │
    ├── 集成评测（端到端业务场景）
    │     ├── 典型流程测试（餐评→审核→下发）
    │     ├── 异常流程测试（用户中断、数据缺失）
    │     └── 多Agent协作测试
    │
    ▼
评测报告生成
    │
    ├── 通过 → 允许部署（灰度发布）
    └── 未通过 → 阻止部署 + 通知开发团队
```

**关键评测指标：**

| 指标 | 定义 | 阈值 |
|------|------|------|
| 安全性评分 | 是否生成危害性内容 | ≥ 0.95（一票否决） |
| 准确性评分 | 医学知识引用正确率 | ≥ 0.85 |
| HITL触发率 | 高风险场景正确触发审核的比例 | ≥ 0.98 |
| P95延迟 | 95%请求的响应时间 | 对话 < 3s，处方 < 30s |
| 单次成本 | 单次交互的Token费用 | < ¥0.5（普通交互） |
| 用户满意度 | 基于用户反馈评分 | ≥ 4.0/5.0 |

### 5.6 关键接口预留设计

为确保未来业务扩展的灵活性，以下接口在当前阶段必须预留：

| 预留接口 | 当前用途 | 未来扩展场景 |
|---------|---------|-------------|
| **Agent Card接口** | 每个Agent输出标准化能力描述 | A2A协议对接医院/药房AI系统 |
| **事件总线接口** | Agent间异步消息通信 | 跨系统事件联动（如血糖异常自动触发医院通知） |

| **评估集扩展** | 评测框架支持动态加载评测集 | 新业务线快速定义专属评测标准 |

## 七、风险与注意事项

### 7.1 医疗安全风险

- **幻觉风险**：Agent生成的医学建议必须经过医生审核，不可直接作为诊断依据。**对策**：高风险场景强制HITL + 安全Guardrails兜底
- **数据隐私**：用户健康数据需加密存储，符合《个人信息保护法》《数据安全法》。**对策**：全链路加密 + 数据访问审计 + 用户数据可删除
- **责任界定**：明确Agent辅助决策与医生最终决策的责任边界。**对策**：系统设计上确保所有医疗决策的最后一环是人类

### 7.2 技术风险

- **模型迭代**：LLM版本更新可能导致Agent行为变化。**对策**：回归评测机制 + 模型版本锁定 + 灰度发布
- **工具依赖**：MCP Server的稳定性直接影响Agent可用性。**对策**：降级策略（MCP Server不可用时返回缓存结果或安全默认值）+ 服务健康检查
- **记忆漂移**：长期记忆可能出现信息过时。**对策**：记忆衰减机制 + 定期记忆清理任务 + 用户可主动纠正记忆
- **供应商锁定**：过度依赖单一LLM供应商。**对策**：模型网关统一抽象 + 定期验证备供应商模型效果

### 7.3 业务风险

- **教练接受度**：健康教练可能担心被替代。**对策**：定位为"效率工具"而非"替代工具"，强调Agent承担80%重复性工作，教练聚焦20%核心决策与情感关怀
- **用户信任**：初期需明确告知用户正在与AI交互。**对策**：透明度声明 + AI/人工切换通道
- **评测成本**：完善的评测体系需要持续投入。**对策**：MVP阶段轻量评测（50条），随业务增长逐步扩展

## 八、总结与核心建议

1. **架构选择**：推荐以**LangGraph**为核心编排框架，**MCP**为工具接入标准，构建**六层可演进架构**
2. **关键基础设施**：**Agent注册中心**（动态发现+版本管理）、**模型网关**（多模型路由+容灾）、**评测体系**（质量门禁+回归保障）是系统可扩展的三大支柱
3. **记忆设计**：采用**四层记忆架构**（工作/情景/语义/程序），结合**Agentic RAG**实现知识动态检索，并通过**记忆治理机制**确保数据质量
4. **安全底线**：处方开具、异常诊断等高风险场景必须设置**Human-in-the-Loop**审核节点，HITL是架构强制约束
5. **知识沉淀**：建立自动话术提取→质量评分→分类存储→复用推荐的闭环机制
6. **接口预留**：Agent Card、事件总线、Webhook、多租户标识等接口在当前阶段必须预留，确保未来新业务接入零代码修改
7. **分步实施**：按"MVP→多Agent协作→知识沉淀→规模化"四阶段推进，每阶段设定明确的验证指标

> **最终定位：** Agent技术已从概念验证进入生产落地阶段。对于贵公司的健康教练业务，关键在于**将Agent定位为"超级助手"**——不是替代健康教练和医生，而是承担80%的重复性工作（餐评生成、习惯督促、话术检索、报告预解读），让专业人员聚焦于20%的核心决策与情感关怀。同时，通过六层架构设计、Agent注册中心和接口预留，确保未来向心理健康、产后恢复、企业健康管理等新业务线扩展时，**新业务的接入不需要修改已有系统的代码**。
