# 飞书文档信息架构与数据结构升级 Plan

> 日期：2026-04-23
>
> 结论口径：本文档描述的是“飞书文档”模块的推荐升级方向，不代表功能已经正式完成。

## 0. 边界共识与非目标

这次方案有一个非常重要的前提：

**“飞书文档”只承担入口、维护、归档、关系管理和使用引导，不承担文档内容托管。**

也就是说：

1. 系统内不承载飞书文档正文。
2. 系统内不做飞书文档的内嵌预览。
3. 系统内不做飞书文档的正文编辑。
4. 用户真正查看和编辑文档时，仍然通过点击链接跳转到飞书。

这意味着“飞书文档”模块的正式定位不是 CMS，而是：

**外部文档的运营入口与上下文管理中心。**

### 0.1 这次明确不做的事

1. 不把飞书文档正文同步进本地数据库。
2. 不在系统内做富文本预览器。
3. 不尝试替代飞书自身的权限、评论、协同编辑能力。
4. 不把本模块做成“另一套文档系统”。

### 0.2 这次真正要解决的问题

要解决的是：

1. 入口太多，不知道该点哪个。
2. 文档很多，不知道哪个是当前正式使用版本。
3. 项目前中后期切换后，文档关系混乱。
4. 模板、正式文档、参考材料、历史归档混在一起。
5. 运营同学要找文档时，只能靠记忆和人工命名约定。

## 1. 先说结论

当前“飞书文档”更像一个按固定分类维护的外链库，而不是一个真正服务运营流程的文档中台。

它现在适合“有一些常用链接，按分类找一找”的场景，但不适合下面这类真实运营工作：

1. 同一个项目前期、中期、后期会不断生成新表单和新文档。
2. 同一篇文档可能同时属于“某个项目”“某个阶段”“某个职能动作”。
3. 运营同学找文档时，脑子里想的通常不是“它属于哪个功能分类”，而是：
   - 我现在在推进哪个项目？
   - 现在处于哪个阶段？
   - 我要找的是定位表、复盘表、日报，还是交付清单？

所以这块的核心不应该继续围绕“单一分类字段”设计，而应该升级成：

**资源层 + 工作台层 + 多维标签层 + 关系层** 的结构。

一句话概括：

**不要把飞书文档建成“分类目录”，而要建成“围绕运营事项组织的文档关系网”。**

---

## 2. 当前阶段判断

### 2.1 已经能做什么

当前正式实现已经支持：

1. 维护文档标题、分类、链接、描述。
2. 按固定分类筛选。
3. 按标题/描述搜索。
4. 管理员或具备 `sop` 权限的用户新增、编辑、删除。

当前正式真值位于：

1. 后端模型：`app/models.py` 中的 `SopDocument`
2. 后端接口：`app/routers/api_sop.py`
3. 前端页面：`frontend/src/views/SopDocuments.vue`

### 2.2 半能做什么

目前勉强能通过这些方式“凑合使用”：

1. 把项目名塞进标题。
2. 把阶段信息塞进描述。
3. 用固定分类粗略归档。

但这些都属于人工补丁，不是正式结构化能力。

### 2.3 还不能做什么

当前还不支持：

1. 一个文档同时挂多个业务语义。
2. 围绕“项目/专项/客户/活动”形成文档工作台。
3. 按阶段查看当前该用哪些文档。
4. 区分 `official / support / candidate / archive` 这类文档角色。
5. 管理模板文档、正式文档、历史归档文档之间的关系。
6. 识别重复链接、失效链接、同一文档的多个入口。
7. 给运营同学提供“最近在用 / 我负责的 / 当前阶段必填清单”这种高频视图。

### 2.4 当前 blocker

当前 blocker 不是页面样式，而是**底层对象建模过于简单**：

`SopDocument` 只有：

1. `title`
2. `category`
3. `url`
4. `description`
5. `sort_order`
6. `created_by`

这套结构只能表达“一个链接属于一个分类”，表达不了真实运营协作关系。

### 2.5 是否偏离蓝图

如果蓝图是“方便运营同学快速找到、判断、复用、沉淀文档”，那当前实现已经偏离：

1. 它更像静态链接卡片，不像运营工作台。
2. 它以系统分类为中心，不以运营推进过程为中心。
3. 它只能存“文档是什么”，不能存“文档在当前业务里扮演什么角色”。

## 3. 设计原则

这块升级时建议坚持 6 条原则：

1. **文档资源和业务上下文分离**
   - 先把“这是哪篇文档”存清楚，再把“它属于哪个项目/阶段/事项”单独建关系。

2. **允许一篇文档拥有多个归属**
   - 同一篇文档可能既是“项目 A 的中期定位表”，也是“营养干预类模板的正式实例”。

3. **默认围绕运营事项找文档，不围绕功能目录找文档**
   - 运营同学首先想到的通常是“我要推进哪个项目”，不是“我要点哪个分类”。

4. **结构化区分文档角色**
   - 必须能区分 `official / support / candidate / template / archive`，避免把补充材料当正式真值。

5. **阶段是字典，不是硬编码死枚举**
   - 系统可以内置“前期/中期/后期”，但要允许不同业务线扩展更细的阶段节点。

6. **前台视图服务任务，后台结构服务真值**
   - 页面可以给运营同学呈现简单入口，但底层数据结构必须足够表达复杂关系。

7. **系统只管理上下文，不管理正文**
   - 正式真值是“链接、归属、角色、阶段、责任人、状态”，不是飞书正文内容本身。

8. **平台能力要可扩展，不把模型写死在飞书单一类型上**
   - 当前虽然叫“飞书文档”，但底层最好支持未来接入 `sheet / bitable / wiki / folder`，甚至其他外部文档平台。

9. **高频属性结构化，低频属性弹性扩展**
   - 常用筛选维度必须结构化，零散差异字段不要把主模型撑肿，建议预留 `extra_meta` 一类扩展区。

## 4. 推荐数据结构

建议不要继续只扩一个 `category` 字段，而是改成四层结构。

### 4.1 资源层：`doc_resources`

这一层只回答一个问题：

**这到底是哪一篇文档？**

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `title` | 文档标题 |
| `canonical_url` | 标准链接 |
| `open_url` | 实际跳转链接，允许与标准链接分离 |
| `source_platform` | 来源平台，当前主要是 `feishu` |
| `source_doc_token` | 飞书文档唯一标识，便于去重 |
| `doc_type` | 文档类型，如 `doc / sheet / bitable / folder / wiki` |
| `summary` | 简短说明 |
| `access_scope` | 访问范围，如 `public_in_team / restricted / unknown` |
| `owner_user_id` | 文档负责人 |
| `maintainer_user_id` | 当前维护人 |
| `status` | `active / archived / invalid` |
| `verification_status` | `verified / unverified / broken / need_check` |
| `last_verified_at` | 最近一次校验链接可用时间 |
| `last_opened_at` | 最近一次被系统跳转打开时间 |
| `created_by` | 创建人 |
| `updated_by` | 最近更新人 |
| `extra_meta_json` | 预留扩展字段，承接低频属性 |

这一层不存“属于哪个项目阶段”，因为那不是资源本体属性。

这里额外建议两点：

1. `canonical_url` 和 `open_url` 分开
   - 未来可能会遇到短链、分享链接、跳转链接、移动端 scheme，不建议把“用于去重的标准地址”和“实际打开地址”混成一个字段。
2. `source_platform + source_doc_token + doc_type`
   - 这三个组合比单纯保存 URL 更稳，后续做去重、迁移、兼容多平台时会更轻松。

### 4.2 工作台层：`doc_workspaces`

这一层回答：

**这些文档是围绕哪条业务推进线组织的？**

可以把它理解成运营视角的“项目工作台 / 专项工作台 / 客户项目 / 活动计划”。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `name` | 工作台名称，如“CCM 减脂营专项” |
| `workspace_type` | `project / campaign / customer / internal_topic` |
| `biz_line` | 业务线 |
| `client_name` | 客户或品牌名，可选 |
| `owner_user_id` | 负责人 |
| `status` | `planning / running / reviewing / archived` |
| `start_date` | 开始日期 |
| `end_date` | 结束日期 |
| `description` | 工作台说明 |

这层是运营同学最自然的入口。

建议额外补两个思路：

1. `workspace` 不是“系统项目表”的技术复制，而是运营视角的使用容器。
2. 如果未来系统里已经有正式项目主实体，也建议先通过关联或映射接入，不要在一开始把飞书文档强耦合到唯一项目模型上。

### 4.3 字典层：`doc_taxonomy_terms`

这一层回答：

**从哪些维度理解一篇文档？**

建议先内置 5 类维度：

1. `stage`
   - 如：线索接入、方案设计、启动准备、执行中、复盘交付
2. `deliverable_type`
   - 如：定位表、排期表、日报、周报、复盘表、素材清单、话术表
3. `function_topic`
   - 如：营养、训练、社群运营、用户分层、复购转化
4. `audience_role`
   - 如：运营、教练、销售、项目经理、市场
5. `channel_or_scene`
   - 如：私聊、群发、直播、训练营、朋友圈

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `dimension` | 维度名 |
| `code` | 稳定编码 |
| `label` | 显示名称 |
| `sort_order` | 排序 |
| `is_active` | 是否启用 |

这层的好处是：阶段、文档类型、场景不再写死在前端数组里。

同时建议支持：

1. 系统默认字典
2. 业务线级别字典
3. 工作台自定义补充字典

也就是说，字典本身也应该有作用范围，例如：

1. `global`
2. `biz_line`
3. `workspace`

### 4.4 关系层：`doc_bindings`

这一层最关键，它回答：

**这篇文档在某条业务推进线里，到底扮演什么角色？**

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `resource_id` | 对应 `doc_resources.id` |
| `workspace_id` | 对应 `doc_workspaces.id`，可为空 |
| `primary_stage_term_id` | 当前主要阶段 |
| `deliverable_term_id` | 当前主要产物类型 |
| `relation_role` | `official / support / candidate / template / archive` |
| `is_primary` | 是否该工作台下的主文档 |
| `due_at` | 该文档在流程中的应完成时间 |
| `sort_order` | 同一视图下排序 |
| `remark` | 备注 |

这里的 `relation_role` 非常重要：

1. `official`
   - 当前工作台在该阶段正式采用的文档
2. `support`
   - 参考材料、补充资料，不等于正式真值
3. `candidate`
   - 备选方案或待确认版本
4. `template`
   - 标准模板，不等于某个实际项目的正式执行文档
5. `archive`
   - 历史归档，仅供追溯

这一步能直接解决现在“链接很多但不知道哪个才是当前该用的”问题。

这里建议把 `doc_bindings` 当成最核心真值，而不是附属表。

因为对运营同学来说，真正重要的不是“这个链接存在”，而是：

1. 它现在挂在哪个工作台里
2. 它在当前阶段算正式还是参考
3. 它是不是这个阶段的主入口
4. 它是否已经过期或待替换

### 4.5 多维挂载层：`doc_term_bindings`

如果一篇文档不止一个阶段、不止一个话题，就不要继续塞单字段。

建议单独做多对多绑定：

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `resource_id` | 文档资源 |
| `term_id` | 某个维度值 |
| `binding_type` | `primary / secondary / recommended` |

这样同一篇文档就可以：

1. 主阶段是“执行中”
2. 次级话题是“营养 + 训练”
3. 面向角色是“运营 + 教练”

### 4.6 使用痕迹层：`doc_usage_logs` 或统计字段

这一层不是第一优先级，但很有运营价值。

建议至少支持：

1. `last_opened_at`
2. `opened_count`
3. `last_opened_by`
4. `favorited_count`

有了这层，后续才能做：

1. 最近使用
2. 高频文档
3. 长期无人打开的脏链接清理

### 4.7 弹性扩展层：`extra_meta_json` 与自定义字段能力

为了不把模型写死，建议从一开始就接受“会有新需求不断冒出来”这个事实。

但扩展不能靠持续加散字段硬撑，否则后面会很乱。

推荐原则：

1. 高频检索字段进正式列
   - 如阶段、角色、状态、负责人、产物类型
2. 低频描述字段进 `extra_meta_json`
   - 如客户简称、投放区域、活动批次、备注标识、外部编号
3. 真正成为高频筛选条件后，再考虑从 `json` 升级为正式字段或字典表

这个策略能避免两种极端：

1. 一开始字段太死，后续一变需求就要大迁移
2. 一开始全塞 JSON，后续无法稳定筛选和排序

## 5. 运营同学真正需要的，不是“分类”，而是这些视图

如果按上面的数据结构建，前台建议优先提供 6 个视图：

### 5.1 最近在用

适合高频使用者回到上次推进现场。

### 5.2 我负责的

按负责人快速收敛自己名下文档与工作台。

### 5.3 按项目工作台看

进入某个项目后，直接看到：

1. 当前阶段
2. 当前阶段的 `official` 文档
3. 补充 `support` 文档
4. 历史 `archive`

### 5.4 按阶段清单看

适合运营同学在推进节点里找“这个阶段通常要用哪些表”。

例如：

1. 启动准备
2. 训练营执行
3. 中期复盘
4. 结项交付

### 5.5 按产物类型看

不是看“功能分类”，而是看真正交付物：

1. 定位表
2. 方案表
3. 素材清单
4. 复盘表
5. 数据回收表

### 5.6 模板中心

把 `relation_role = template` 的内容单独收进模板视图，避免模板和真实项目文档混在一起。

### 5.7 待补齐与待清理

这个视图很适合负责人或 admin 使用。

建议收口：

1. 缺少当前阶段 `official` 文档的工作台
2. 链接失效或待校验的文档
3. 长期无人打开但仍挂在活跃工作台里的文档
4. 同一工作台同一阶段存在多个 `official` 的异常项

这类视图很重要，因为它决定这个模块后续能不能真正承担“管理”和“治理”职责。

## 6. 推荐页面交互，不要继续停留在“卡片列表”

推荐把“飞书文档”改造成“两级入口”：

### 一级入口：工作台视图

用户先选：

1. 我负责的项目
2. 当前推进中的项目
3. 模板中心
4. 最近使用

### 二级入口：文档列表视图

进入后再用筛选器收缩：

1. 阶段
2. 产物类型
3. 文档角色
4. 负责人
5. 关键字

每张文档卡片上建议直接展示：

1. 文档标题
2. 所属工作台
3. 当前阶段
4. 文档角色
5. 负责人
6. 最近更新时间
7. 是否当前主文档

这样运营同学在点开前就能判断：

1. 这是不是当前项目的正式文档
2. 这是模板还是实际执行表
3. 这篇是不是历史版本

### 6.1 前端不应该让运营同学先做“信息管理”，而应该先做“找得到”

运营同学的典型心智通常是：

1. 我现在在推进哪个项目
2. 我当前卡在哪个阶段
3. 我需要哪类表
4. 哪一篇是当前该打开的

所以前端首页不建议先给一个复杂筛选器，也不建议先给一棵很重的目录树。

更顺的做法是：

1. 先给“我在推进的工作台”
2. 再给“当前阶段推荐入口”
3. 再给“最近打开/最近更新”
4. 最后才给高级筛选和全量检索

### 6.2 推荐的首页模块

建议首页按下面顺序组织：

1. `我负责的工作台`
2. `最近打开`
3. `当前阶段必用文档`
4. `模板中心`
5. `待清理 / 待补齐`

这样组织的好处是：

1. 先服务高频动作
2. 再服务治理动作
3. 不需要运营同学先理解复杂结构才能开始使用

### 6.3 推荐的“新增文档”交互

新增时不应该让运营同学手填很多字段。

推荐最小流程：

1. 粘贴飞书链接
2. 系统自动解析或预填标题、类型、标准链接
3. 选择所属工作台
4. 选择当前阶段
5. 选择文档角色
6. 可选补充负责人、摘要、备注
7. 保存

也就是说，新增动作要尽量从“填表”变成“绑定关系”。

### 6.4 推荐的“打开文档”交互

点击文档时，系统只做三件事：

1. 记录一次打开行为
2. 可选展示“你即将离开系统前往飞书”
3. 新窗口或新标签跳转

不要在这条链路里引入预览器、权限代理或内容同步，否则交互会变重，也会偏离这次方案边界。

### 6.5 推荐的工作台详情页

工作台详情页建议按阶段组织，而不是按文档类型平铺。

例如：

1. 顶部：工作台概览、负责人、状态、当前阶段
2. 中部：按阶段分区展示 `official / support / candidate`
3. 右侧或下方：最近更新、模板推荐、历史归档

这样更符合运营推进节奏。

---

## 7. 推荐落地路径

不建议一步到位做成很重的知识系统，建议分三期。

### 第一期：先从“单分类链接库”升级到“多维元信息链接库”

目标：先把数据表达能力补起来，不先追求复杂页面。

建议动作：

1. 保留现有页面和接口骨架。
2. 先把 `SopDocument` 升级为“文档资源”模型。
3. 增加：
   - `source_doc_token`
   - `doc_type`
   - `owner_user_id`
   - `status`
   - `last_verified_at`
4. 把当前 `category` 降级为 `deliverable_type` 或 `function_topic` 的过渡字段。
5. 增加 `relation_role`、`workspace_name`、`stage_label` 这类过渡元信息字段。

这期重点是先把“模板 / 正式 / 候选 / 归档”区分开。

### 第二期：补工作台与多对多关系

目标：让文档真正围绕运营事项组织，而不是围绕目录组织。

建议动作：

1. 新增 `doc_workspaces`
2. 新增 `doc_bindings`
3. 新增 `doc_taxonomy_terms`
4. 支持一个文档挂多个维度标签
5. 支持同一工作台下有“当前主文档”

这期完成后，系统才真正具备“项目中台”能力。

### 第三期：补运营效率能力

目标：让它从“可管理”变成“好用”。

建议动作：

1. 最近使用
2. 收藏
3. 失效链接巡检
4. 重复文档识别
5. 阶段清单推荐
6. 模板一键复制到工作台

## 8. 未来很可能会发生的扩展需求

为了不提前写死，建议从现在就把下面这些扩展场景考虑进去。

### 8.1 文档类型扩展

未来很可能不止是“飞书文档链接”，还会包括：

1. 飞书表格
2. 飞书多维表
3. 飞书知识库节点
4. 文件夹集合入口
5. 其他外部文档平台

所以底层一定要用：

1. `source_platform`
2. `doc_type`
3. `source_doc_token`

而不是只存一个 URL 字符串。

### 8.2 工作台之外的挂载需求

未来可能会出现：

1. 一个模板不属于具体工作台，但属于某条业务线
2. 一篇文档同时挂多个项目
3. 一篇文档既服务运营，又服务教练

所以关系层必须允许：

1. 多对多
2. 多维度
3. 同一资源存在多个绑定

### 8.3 文档治理需求

文档中心做起来后，运营侧很快会提这些要求：

1. 哪些链接已经失效
2. 哪些项目没有补齐正式文档
3. 哪些工作台挂了重复入口
4. 哪些文档长期没人打开
5. 哪些文档负责人已经离岗或需要交接

所以从现在开始就应该预留：

1. 校验状态
2. 使用记录
3. 负责人
4. 归档状态

### 8.4 视图与权限扩展

未来可能会有：

1. 普通运营只看自己相关工作台
2. 项目经理看整条业务线
3. admin 看全局治理看板
4. 不同角色对“模板中心”和“归档区”有不同可见范围

所以前端不要把视图逻辑写死在一个页面里，后端也不要把筛选逻辑只做成前端本地过滤。

### 8.5 自动化与提醒扩展

后面很可能还会加：

1. 阶段切换提醒
2. 缺少正式文档告警
3. 失效链接巡检任务
4. 常用模板推荐
5. 文档交接提醒

这些能力现在不必做，但数据结构要能托住。

---

## 9. 推荐的最小可用模型

如果这轮只做一个“够用但不重”的版本，我建议最少做下面 3 张表：

1. `doc_resources`
2. `doc_workspaces`
3. `doc_bindings`

其中：

1. `doc_resources` 负责“文档本体”
2. `doc_workspaces` 负责“项目/专项上下文”
3. `doc_bindings` 负责“这篇文档在这个上下文里扮演什么角色”

而 `stage / deliverable_type / topic` 先可以过渡成：

1. 字典表
2. 或 JSON 数组字段

但长期仍建议回到字典 + 绑定表。

## 10. 一个具体例子

比如“减脂营项目”在执行过程中可能有这些文档：

1. 项目定位表
2. 开营排期表
3. 每周复盘表
4. 教练话术协同表
5. 结营复购方案

当前模型下，这 5 篇文档大概率只能被塞进：

1. 运营流程
2. 话术库
3. 经验库

运营同学之后很快就会乱。

按新模型，它们可以表达为：

1. 工作台：`2026Q2 减脂营专项`
2. 阶段：
   - 启动准备
   - 执行中
   - 中期复盘
   - 结营转化
3. 文档角色：
   - `official`
   - `support`
   - `template`
4. 产物类型：
   - 定位表
   - 排期表
   - 复盘表
   - 话术协同表
   - 转化方案

这时运营同学进入工作台后，一眼就能知道：

1. 当前阶段该用哪几篇正式文档
2. 哪些只是参考资料
3. 哪些是历史版本
4. 哪些模板可以复制复用

## 11. 风险与开放问题

### 10.1 阶段要不要全系统统一

建议：

1. 系统内置默认阶段字典
2. 允许工作台按业务线定制阶段

不要把所有项目都强压成一套死枚举。

### 10.2 项目和模板是否共用一套结构

建议共用结构，但在 `relation_role` 和 `workspace_type` 上区分。

原因是模板和实际项目文档会互相引用，如果完全拆成两套，后面反而更难维护。

### 10.3 是否要直接接飞书开放平台

这期不建议把范围扩大到“直接同步飞书内容”。

先把本地文档关系网建对，比先做同步更重要。

### 10.4 要不要把所有字段都做成必填

不建议。

如果首屏字段太多，运营同学会把这个模块当成负担。

推荐做法：

1. 首次新增只填最小必要信息
2. 治理字段分层补齐
3. 只有真正影响检索和推荐的字段才设为硬性必填

### 10.5 要不要先做复杂树状结构

也不建议。

树状结构看起来完整，但实际会把运营同学重新拖回“记住目录”的负担里。

更推荐：

1. 工作台入口
2. 阶段分区
3. 角色标签
4. 搜索和筛选补充

让系统帮用户定位，而不是要求用户先理解树。

---

## 12. 最值得继续推进的三条线

### 线 1：先把数据模型从“分类卡片”升级为“资源 + 关系”

这是基础盘，不做这步，前端再怎么改也只是更好看的混乱。

### 线 2：把首页入口从“按分类浏览”改成“按工作台/阶段找文档”

这是直接影响运营使用体感的一步，价值非常高。

### 线 3：把新增流程做成“粘贴链接 -> 绑定上下文”，而不是“打开一个大表单”

这是降低使用负担的关键一步。

---

## 13. 最终建议

“飞书文档”这块最适合的正式方向不是：

1. 单纯继续扩功能分类
2. 也不是只按项目建树状目录

而是：

**以工作台为主入口，以文档资源为真值，以阶段/产物/角色为多维索引，以关系表表达文档在业务中的职责。**

这样设计之后，运营同学面对的就不再是“很多链接”，而是：

1. 当前在推进什么
2. 当前阶段要看什么
3. 哪篇是正式的
4. 哪篇是参考的
5. 哪些可以复用

这才是真正贴近运营流程的“飞书文档”能力。

同时也要始终记住它的边界：

**它负责把人准确送到该去的飞书文档，不负责把飞书搬进系统。**
