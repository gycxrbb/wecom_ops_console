# 定时群发队列超时与重复发送问题排查报告

## 1. 现象还原与问题概括
用户在执行定时队列群发任务时（消息数量 > 20条），发生以下现象：
1. **发送失败与重试**：当发送到第 21 条时，任务出现 `{"errcode": 45009, "errmsg": "api freq out of limit..."}` 报错，并伴随 “(已重试3次)” 的提示日志。
2. **部分成功交替发生**：限制期过后，后续消息继续由系统发送并成功。
3. **重复发送（Duplicate Messages）**：群组内收到了重复的内容。

经深入排查，此问题是**系统批量发送逻辑中的频控设计缺陷**，叠加上**UI 手动重试体验导致的用户误操作**两者共同造成的。

---

## 2. 根因详细分析

### 2.1 产生 45009 限流的原因
企微对单个群机器人的发送频率限制是 **20条/分钟（即 60 秒的滑动窗口）**。
在 `app/routers/api.py` 的队列发送逻辑 `perform_job_send` 中：
```python
for item in batch_items:
    try:
        results = await do_send_to_groups(...)
    ...
    # 限速：条目间等待 2 秒
    await asyncio.sleep(2)
```
- **速率过快**：相邻两条消息的间距为 `2秒`。在没有任何阻挡的情况下，1分钟能发 `60/2 = 30条`。
- **必然触发封控**：因为 30条 > 20条，对于超过 20 条的大队列，在发送到第 21 条时，时间刚过去大约 40 秒，必然会触发企微的 60 秒速率墙（45009 错误）。
- **内部拦截器失效/误差**：代码中虽然有 `_check_rate_limit` (20条/分钟的内存拦截算法)，但内存限流在多进程部署、系统刚重启、或在精准的边缘滑动窗口下常常会产生漏网之鱼，直接把请求打到企微服务器，获得真实的 `45009` HTTP 响应码。

### 2.2 为什么“(已重试3次)”毫无作用而直接判定失败？
系统 `_send_to_single_group` 带有自动重试逻辑：
```python
retry_delay = settings.send_retry_delay_seconds  # 默认仅 1.0 秒
for attempt in range(1, max_retries + 1):
    try:
        # WeCom API Request
    except Exception as exc:
        await asyncio.sleep(retry_delay)
```
当企微的限流 45009 发生时，系统捕获到异常并会睡眠 1 秒后重试。由于**“60秒滑动窗口”不会在短短的 1.0 秒重试间隔内恢复**，因此系统的 3 次重试会在 3 秒内飞速消耗掉且全部碰壁 45009，随后记录为一次彻头彻尾的发送失败，继续流转到下一条。后续的消息会一直重复 “碰壁 -> 等1秒并失败 3 次”，直到距离第一条消息被发送总共过去了 60 秒（滑动窗口终于开始清空历史记录）。这就解释了为何一开始失败，后来某段时间窗口又“自动”恢复成功。

### 2.3 为什么群内出现了“重复发送”的消息？
重复发送本质是一次**用户视觉残留引发的重试误操作**：
1. **自动队列没有重发成功过**：如上所述，在当时触发失败的那几条消息，在系统的底层流程记录中是死路一条，直接被标注为 `status: 'failed'` 存入日志，系统并未在底层偷偷将其“晚一点再发”。
2. **UI 列表残留原失败记录**：运维人员在前端 UI 的日志列表中，看到了这条标记为“失败”的数据行，并在焦急等待中点击了行尾的“重试”按钮。
3. **接口的新产生设计**：`api.py` 的 `/logs/{log_id}/retry` 接口在响应用户的点击时，确实**真实执行了一次向企微补充发送消息**，并且**往 DB 写入了一行全新的成功日志**（这也是 UI 截图中那些 `source` 为 `retry` 并且状态标注为 `成功` 的长条目来源）。
4. **原始失败记录未掩盖**：非常遗憾的是，重试接口**并未修改或隐藏掉最初的那一根“失败”状态的记录行**。因此，用户刷新页面后，原本的“失败”记录依然矗立在那里，并在行尾堂而皇之地继续展示着“重试”按钮。用户很容易产生直觉错觉：“刚才点重试是不是没反应/失败了？”，进而**连续反复点击重试**。从而向群内输送了大量重复内容。

## 3. 修复方案建议（Action Items）

为了彻底解决超过 20 条的任务发送稳健性问题与杜绝人员误操作现象，需推进以下三阶段修复：

### 第一部分：核心发送限流的治本对齐 (后端逻辑)
1. **调整队列发送步长**：将 `perform_job_send` 处的 `await asyncio.sleep(2)` 限定时间更改为安全值：如果是单群推送，**必须设置 `await asyncio.sleep(3.1)` 以上（确保 60 秒的滑动窗口内无法积累超过 20 条）**，最佳做法是动态侦测 `batch_items` 并对单一接收群做动态流控（例如累计20个暂停10-20s）。
2. **优化自动重试机制针对限流的降级**：
   在 `_send_to_single_group` 检测到 `45009` (Freq out of limit) 错误时，不应当进行短促无意义的 1s x 3 次重试。对于 45009 这种特定的 429 类型错误，系统应对该队列指令**临时挂起休眠（Sleep for 10~30s）**以等待滑动窗口释放后再补发，或投入长时等待队列。

### 第二部分：手动重试 UX 修复 (前端及后端日志)
1. **隐藏失效重试按钮**：更改前端界面的判定逻辑：当发现某条 `/logs` 最新日志内该 `group_id` 与 `message_id` 的相同投递如果已经出现了 `success=1`（不论通过何种 attempt），那么列表中关于这条消息涉及的所有“失败”记录，其重试按钮都应该**变成置灰/隐藏/变为“已补发”字样**。
2. **修改重试 API 并发锁保护**：在 `/logs/{log_id}/retry` 接口追加并发锁或是 `is_resolved` 标记（为 Message 添加一个补救成功后的字段），防止短连接抖动造成的用户多次连点疯狂写入企微群聊。
