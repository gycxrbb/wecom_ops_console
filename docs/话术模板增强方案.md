# 话术管理增强方案

## 当前状态评估

### 已经能做什么
- CSV 批量导入话术，自动解析场景/风格/元数据
- 两级分类树浏览（L1 大类 > L2 子类 > 场景列表）
- 分类树可编辑（重命名、删除、新建、移动）
- 场景可归类到子类下
- 每个场景下的多种风格话术可编辑保存
- RAG 批量索引（导入时可选）

### 半能做什么
- CSV 导入可用，但存在多个风险（见下方审查）
- 分类自动回填可用，但依赖硬编码映射表

### 还不能做什么
- 无法在前端手动新建话术（必须 CSV 导入）
- 无法在前端编辑话术的 RAG 元数据（summary、标签、安全级别等）
- 无法在前端查看话术的 RAG 索引状态
- 无法对单个话术触发重新索引

## 一、CSV 导入审查结论

### 1.1 去重机制

**当前逻辑**：按 `label == title` 匹配 `is_builtin == 0` 的模板，找到则更新，找不到则新建。

**问题**：

| 风险 | 等级 | 说明 |
|------|------|------|
| 去重键不合理 | **高** | 用 `label` 做去重键，但同一 `label` 可属于不同 `(scene_key, style)` 组合。两个不同场景但标题相同的话术会互相覆盖 |
| 与手动创建不一致 | **中** | 手动创建接口 (`POST /speech-templates`) 按 `(scene_key, style)` 做唯一校验，CSV 导入按 `label` 做去重，两套逻辑 |
| 并发导入竞态 | **低** | 两个 CSV 同时导入同一 label 的行，可能都查不到已有记录，都插入，产生重复 |
| 无审计追踪 | **低** | 更新时直接覆盖 `content`、`metadata_json`，不记录旧值，无版本历史 |

**建议修复**：
1. 去重键改为 `(scene_key, style, is_builtin)` 组合，与手动创建逻辑对齐
2. 更新前记录变更日志（至少打一条日志，理想情况写入版本记录）
3. 并发保护：在更新时加 `SELECT ... FOR UPDATE`

### 1.2 字段映射缺失

| CSV 字段/概念 | 是否映射到 DB | 影响 |
|------|------|------|
| `category_id` / 分类 | **未映射** | 导入的话术无分类，依赖 `/scenes` 的懒回填 |
| `owner_id` / 导入人 | **未映射** | 无法追踪谁导入了什么 |
| `is_builtin` | 硬编码为 `0` | 正确，避免覆盖 seed 数据 |
| `source_id` | 仅返回在结果中 | 不入库，无法追溯到源数据 |
| `tags` | 写入 `metadata_json` | 正确 |

### 1.3 验证缺口

| 缺口 | 影响 |
|------|------|
| 无 `title` 长度校验（DB 上限 128） | 超长标题会导致 DB 错误 |
| 无 `scene_key` 长度校验（DB 上限 64） | `normalize_scene_key` 的 fallback 截断只覆盖部分路径 |
| 无行数上限 | 超大 CSV 可能 OOM |
| 非 `approved`/`active` 状态的行静默跳过 | 用户不知道为什么某些行没导入 |
| `style` 不识别时静默默认为 `professional` | 用户不知道风格被改了 |

### 1.4 RAG 索引风险

- 导入触发的是**全量索引**（扫描所有话术），不是增量索引
- 数据量大时耗时长，且没有进度反馈
- `category_id` 为空时，RAG 语义文本缺少分类信息，影响检索质量

### 1.5 修复优先级

| 优先级 | 修复项 |
|--------|--------|
| P0 | 去重键改为 `(scene_key, style)`，与手动创建对齐 |
| P1 | CSV 导入时自动按 `SCENE_CATEGORY_MAP` 回填 `category_id` |
| P1 | `title` 长度校验（截断或报错） |
| P2 | 非 approved/active 行的跳过原因写入结果 |
| P2 | 导入时记录 `owner_id` |
| P3 | 并发保护（`SELECT FOR UPDATE`） |
| P3 | 行数上限 |

## 二、手动新建话术 + RAG 元数据编辑

### 2.1 需求描述

1. 在侧边栏的场景列表区域，每个 L2 子类下增加"新建话术"入口
2. 点击后弹出对话框，填写场景标识、风格、标题、内容
3. 保存后自动分配到当前子类的 `category_id`
4. 话术编辑区增加"RAG 配置"按钮，点击弹出对话框，可编辑：
   - `summary`（摘要）
   - `customer_goal`（客户目标，多选标签）
   - `intervention_scene`（干预场景，多选标签）
   - `question_type`（问题类型，多选标签）
   - `safety_level`（安全级别）
   - `visibility`（可见性）
   - `tags`（自定义标签）
   - `usage_note`（使用说明）
5. 保存 RAG 配置后触发单条话术的增量索引（非全量）

### 2.2 涉及文件

| 操作 | 文件 | 说明 |
|------|------|------|
| 修改 | `app/routers/api_speech_templates.py` | 新增 `POST /` 创建话术、`PATCH /{id}/rag-meta` |
| 新增 | `app/schemas/speech_template.py` | Pydantic schema: 创建请求、RAG 元数据更新请求 |
| 修改 | `app/rag/resource_indexer.py` | 新增 `index_single_speech_template()` 增量索引 |
| 修改 | `frontend/src/views/SpeechTemplates.vue` | 新建话术入口 + RAG 配置对话框 |
| 重构 | `frontend/src/views/SpeechTemplates.vue` | 文件已接近红线，需拆分为 composable + 子组件 |

#### 2.3.2 RAG 元数据更新 `PATCH /{id}/rag-meta`

```python
class SpeechRagMetaReq(BaseModel):
    summary: str | None = None
    customer_goal: list[str] | None = None
    intervention_scene: list[str] | None = None
    question_type: list[str] | None = None
    safety_level: str | None = None
    visibility: str | None = None
```

流程：
1. 读取当前 `metadata_json`，解析为 dict
2. 用请求中的非 None 字段覆盖
3. 序列化回 JSON，写入 DB
4. 调用 `index_single_speech_template(template_id)` 增量索引
5. 返回更新后的 `metadata_json`

### 2.4 前端设计

#### 2.4.1 文件拆分

当前 `SpeechTemplates.vue` 已有 820+ 行，按 AGENTS.md 红线需拆分：

### 2.5 实施步骤

#### Phase 1：CSV 导入修复（P0+P1）

1. 修改 `speech_template_import.py` 去重键为 `(scene_key, style)`
2. 导入时按 `SCENE_CATEGORY_MAP` 回填 `category_id`
3. 加 `title` 长度校验
4. 加 `owner_id` 记录
5. **验证**：导入测试 CSV，确认去重、分类、RAG 索引正常

#### Phase 2：后端新建话术 + RAG 元数据 API

1. 新增 `app/schemas/speech_template.py`（Pydantic schema）
2. 修改 `POST /speech-templates` 支持前端创建
3. 新增 `PATCH /{id}/rag-meta` 端点
4. 新增 `index_single_speech_template` 增量索引
5. `GET /scenes` 返回 `metadata_json`
6. **验证**：API 测试，创建话术 + 编辑 RAG 元数据 + 索引

#### Phase 3：前端拆分 + 新建话术

1. 拆分 `SpeechTemplates.vue` → composable + 子组件
2. L2 子类下增加"新建话术"按钮 + 对话框
3. **验证**：前端创建话术 → 出现在对应子类下 → 可编辑

#### Phase 4：前端 RAG 配置

1. 新增 `SpeechRagMetaDialog.vue` 组件
2. 编辑区加"RAG 配置"按钮
3. 对话框加载/保存 RAG 元数据
4. **验证**：编辑 RAG → 保存 → 索引成功 → AI 对话可检索到

### 2.6 注意事项

1. **前端文件红线**：拆分后每个文件不超过 400 行
2. **后端三层分离**：RAG 元数据保存逻辑放 service 层，路由只做参数校验和调用
3. **标签复用**：RAG 标签体系与素材库共用，不新建标签表
4. **增量索引**：只索引单条话术，不触发全量扫描
5. **向后兼容**：`metadata_json` 为 NULL 时，对话框各字段显示空值
6. **CSV 导入 P0 修复优先于前端功能开发**
