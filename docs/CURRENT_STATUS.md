# 企微群消息运营后台当前状态

- 更新时间：2026-03-31
- 对照依据：`README.md`、`app/`、`frontend/`、`docs/`
- 结论口径：本文档描述的是当前仓库代码状态，不等于最终规划态

---

## 1. 项目现在在做什么

这个项目正在把“企业微信群机器人消息发送”从脚本化、配置化操作，升级成一套教练和运营可直接使用的后台系统。

从业务链路看，当前产品已经覆盖了以下闭环：

1. 管理群和 Webhook
2. 管理模板和素材
3. 在发送中心选择消息类型并预览
4. 发起立即发送或创建定时任务
5. 对部分任务走审批
6. 查看发送日志并支持重试

---

## 2. 当前真实技术形态

### 2.1 后端

- FastAPI 单体应用
- SQLAlchemy ORM
- 默认 SQLite，本地数据库位于 `data/app.db`
- 本地上传目录位于 `data/uploads`
- 认证采用 JWT，代码中保留了 session 兼容逻辑
- 发送逻辑通过企业微信群机器人 Webhook 直接调用
- 调度使用 APScheduler 雏形
- Celery 代码入口文件已存在，但未形成完整生产链路

### 2.2 前端

- Vue 3 + TypeScript + Element Plus + Pinia + Vue Router + Vite
- 已有登录、看板、群管理、模板中心、素材库、发送中心、定时任务、审批中心、日志、用户管理页面
- 发送中心已拆出 `message-editor` 组件和 `useSendLogic.ts` 组合式逻辑

### 2.3 部署形态

- 当前更接近本地开发 / 单机部署形态
- `Dockerfile`、`docker-compose.yml` 已存在
- Redis、Celery、Next.js 仍处于规划态，不是当前真值

---

## 3. 已经能做什么

### 3.1 看见层

- 管理员和教练可以登录
- 可以看到看板、群、模板、素材、任务、审批、日志等页面

### 3.2 理解层

- 模板支持变量渲染
- 消息支持 `text`、`markdown`、`news`、`image`、`file`、`voice`、`template_card`
- 图片和文件消息会在后端补素材路径，再构造企微 payload

### 3.3 决策层

- 可以立即发送
- 可以发送到测试群
- 可以创建一次性和周期性任务
- 可以发起审批并由管理员审批

### 3.4 闭环层

- 有消息落库和日志查询入口
- 有失败日志重试入口
- 有基础统计接口和看板页面

---

## 4. 半能做什么

以下能力已经有代码骨架，但还没有完全收口：

### 4.1 定时任务

- 前后端都已有任务创建、启停、复制、审批、手动执行入口
- 但模型、路由和调度服务之间存在字段名不一致
- 调度器在 `lifespan` 中没有完整启动和同步逻辑

### 4.2 审批流

- 已有审批表、审批列表、审批通过/拒绝接口
- 但审批状态字段在不同层命名不完全统一
- “立即发送是否必须审批”这条链路还不完整，当前更偏向任务审批

### 4.3 统一接口规范

- 前端请求层已经按 `/api/v1` 和统一响应在写
- 但实际接口返回结构并不完全一致
- 一部分接口是直接返回列表或对象，不是完整 `{ code, message, data, request_id }`

---

## 5. 当前主要真值漂移

这是当前最需要优先收口的部分。

### 5.1 数据模型与路由字段漂移

- `app/models.py` 中 `Schedule` 仍是 `name`、`group_id`、`content_snapshot`、`require_approval`
- `app/routers/api.py` 中调度相关逻辑使用的是 `title`、`group_ids_json`、`content`、`approval_required`
- 这说明任务模型正在从旧结构向新结构迁移，但尚未完成

### 5.2 调度服务与模型名漂移

- `app/services/scheduler_service.py` 仍引用 `models.MessageJob`
- 但当前模型文件中并没有 `MessageJob`，而是 `Schedule`
- 这会直接影响调度链路稳定性

### 5.3 日志与消息模型字段漂移

- `api.py` 中日志序列化使用 `job_id`、`group_id`、`initiated_by_id`、`request_json`
- `models.py` 中 `MessageLog` 当前字段是 `message_id`、`request_payload`、`response_payload`
- 这说明日志真值也没有完全统一

### 5.4 文档与实现技术栈漂移

- 文档多处写了 MySQL、Redis、Next.js、Celery 是当前方案
- 当前代码真实落地的是 SQLite、APScheduler、本地上传、FastAPI 单体

## 6. 当前阶段判断

项目当前更准确的定位是：

- 已完成从“概念设计”到“可见产品骨架”的阶段
- 正处于 V2 前后端分离版本的联调与真值统一阶段
- 距离“稳定可上线的生产版”还差一轮后端模型/接口/调度链路收口

一句话概括：

当前不是从 0 到 1 的阶段，而是从“功能都长出来了”走向“真值统一、稳定可运行”的阶段。

---

## 7. 当前 blocker

### blocker 1：后端模型与接口没有统一

这是当前最大的结构性 blocker，会影响：

- 定时任务能否真正创建、查询、执行
- 审批和日志能否稳定关联
- 前端页面是否能持续联调

### blocker 2：调度执行链路未完全闭环

即便页面和 API 已存在，如果调度器、任务模型和发送任务模型不一致，定时发送就无法视为正式完成。

### blocker 3：统一响应和接口契约未完全收口

这会让前端在不同页面产生兼容分支，后续维护成本会上升。

---

## 8. 建议优先级

### P0：先做真值统一

1. 统一 `Schedule` 模型字段
2. 统一 `Message` / `MessageLog` 字段
3. 统一审批字段命名
4. 统一 `/api/v1` 接口返回结构

### P1：再打通调度闭环

1. 修复 `scheduler_service.py` 对旧模型的引用
2. 在应用启动时完成调度器启动和数据库任务同步
3. 校验一次性任务、Cron 任务、跳过周末、跳过日期

### P2：补完稳定性和运维能力

1. 补文件上传限制和素材删除校验
2. 补失败重试策略
3. 补 request_id、结构化日志和审计日志
4. 补最小测试集和启动验证脚本

### P3：再进入生产化演进

1. SQLite 升级到 MySQL / PostgreSQL
2. 内存频控升级到 Redis
3. APScheduler 演进为 Celery + Redis
4. 引入 Next.js 预览 / 对外页面能力

---

## 9. 下一步最值得做什么

如果只做一件最值回票价的事情，建议先做：

**后端任务、消息、日志三条模型真值统一，并同步修正 API 返回结构。**

原因很简单：

- 这会直接决定发送、调度、审批、日志四条主链路是否真的能闭环
- 这也是当前文档、前端、后端三方漂移的共同根因
