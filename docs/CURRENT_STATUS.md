## 1. 项目现在在做什么

这个项目已经从"企微群机器人消息发送后台"升级为**企业微信运营后台**，覆盖消息发送、客户档案管理、AI 教练助手、运营计划编排、积分运营等完整运营链路。

当前产品已覆盖的闭环：

1. **消息闭环**：群管理 → 模板 → 发送中心 → 定时调度 → 审批 → 日志 → 重试
2. **运营计划闭环**：计划编排 → 日程节点 → 批量发布 → 自动创建定时任务
3. **客户档案闭环**：CRM 数据接入 → 7 模块档案 → AI 教练对话 → 消息选发
4. **积分运营闭环**：CRM 群绑定 → 积分排名预览 → 话术模板 → 批量发送

## 2. 当前真实技术形态

### 2.1 后端

- **FastAPI** 单体应用，按资源拆分 14 个路由文件
- **SQLAlchemy ORM**，15 个核心模型 + 6 个外部文档模型 + 5 个 CRM Profile 模型
- 默认 **SQLite**（开发），支持 **MySQL 8.0**（生产）
- **APScheduler** 调度（AsyncIOScheduler），Celery 作为可选异步任务通道
- **Redis** 作为 Celery Broker 和缓存中间件
- 认证：JWT（24h 过期）+ Session 兜底 + RSA 密码加密 + Fernet Webhook 加密
- 权限：10 项细粒度权限（admin/coach 角色）
- AI/LLM：OpenAI 兼容 API，支持 aihubmix / DeepSeek 双 provider
- 存储：本地 / 七牛云双存储，StorageFacade 自动降级

### 2.2 前端

- **Vue 3 + TypeScript + Element Plus + Pinia + Vue Router + Vite**
- 20+ 页面视图，按功能模块拆分到 7 个视图目录
- 发送中心：消息编辑器（8 种类型）+ 预览 + 队列 + 调度
- CRM Profile：客户列表 + 360 档案 + AI 教练面板（流式 SSE）+ 消息选发
- 模板中心：模板管理 + 运营计划工作台
- 群管理：内部群 / CRM 外部群 / 排行榜三合一
- 外部文档：工作空间 + 资源管理 + 治理队列
- 暗色主题支持

### 2.3 部署形态

- **Docker Compose** 多阶段构建（Node 构建 + Python 运行，单容器交付）
- 开发环境：MySQL + Redis + App + Celery Worker + Celery Beat
- 生产环境：单容器 + 宿主机 MySQL/Redis/Nginx（宝塔面板）
- GitHub Webhook 自动部署

## 3. 已经能做什么

### 3.1 核心消息发送

- 8 种消息类型：text、markdown、news（图文）、image、file、voice（AMR）、template_card、raw_json
- 单条/批量发送、测试群验证、消息队列、预览
- 一次性 + Cron 周期任务，支持跳过周末/指定日期、日历视图
- 教练提交 → 管理员审批 → 自动执行
- AI 文案润色（LLM 优化）
- 内置频控（单机器人 20 条/分钟）

### 3.2 运营计划

- Plan → Day → Node 三级结构，节点关联模板
- 支持 Excel (.xlsx) 和 JSON 格式导入/导出
- 一键发布：将计划各天节点批量创建为定时任务
- 积分活动模式：专属活动阶段编排与批量建任务

### 3.3 CRM 客户档案与 AI 教练

- 7 模块客户 360 档案（基础档案、体成分、目标偏好、健康摘要、积分活跃、安全档案、服务关系）
- AI 教练助手：流式 SSE 对话，基于客户档案回答问题，可见思考链
- 6 个 AI 场景 Prompt（餐评、数据监测、异常干预、答疑、周期复盘、长期维护）
- 安全门禁：医疗内容过滤、需医生确认标记
- 教练可编辑沟通风格、阶段重点、执行障碍等长期备注
- 上下文预加载：进入客户详情页后自动预热 AI 上下文缓存
- 消息选发：从 AI 对话中选择消息一键跳转发送中心

### 3.4 CRM 积分运营

- CRM 外部群 ↔ 本地微信群映射绑定
- 积分排名：12 种话术场景、多种风格模板，支持单人/群排名
- 全局排名：跨群 TOP10 汇总
- 话术模板：按场景/风格管理可复用话术

### 3.5 外部文档管理

- 飞书等外部链接统一管理
- 工作空间：项目级文档组织，阶段跟踪
- 术语体系：可配置的阶段/交付物/分类词汇
- 治理队列：过期/未验证文档追踪

### 3.6 系统管理

- 看板：群/模板/任务统计、发送成功率、趋势图
- 用户管理：admin/coach 角色、10 项精细权限、CRM 账号联动登录
- 素材存储：本地/七牛云双存储，支持迁移和降级回退
- 系统教学文档：Markdown 文档系统，草稿/发布模式
- 暗色主题

## 4. 当前架构特征

### 4.1 后端分层已落地

```
路由层（routers/）  → 14 个路由文件，按资源拆分
服务层（services/） → 30+ 个服务文件，业务逻辑封装
数据层（models.py） → 26 个 ORM 模型
```

路由层严格遵守"不写超过 5 行业务逻辑"的规则。`api.py`（~1465 行）是历史遗留的最大文件，后续新增接口已全部拆分到独立路由。

### 4.2 前端模块化已落地

```
views/        → 7 个功能目录，每个目录含 index.vue + components/ + composables/
components/   → 全局组件（Markdown 渲染、消息编辑器、Cron 构建器等）
composables/  → 可复用逻辑（useAiCoach、useSendLogic、useCrmProfile 等）
```

### 4.3 Feature Gate 机制

CRM Profile 和 AI Coach 通过环境变量 `CRM_PROFILE_ENABLED` / `AI_COACH_ENABLED` 控制，条件加载路由和模块。

### 4.4 存储抽象

`StorageProvider` → `LocalStorageProvider` / `QiniuStorageProvider`，`StorageFacade` 管理活跃/降级 provider。

## 5. 已知的技术债务

### 5.1 `api.py` 仍然过大（~1465 行）

这是历史遗留的核心路由文件，包含群、模板、素材、任务、发送、日志、用户、看板、审批等所有早期接口。后续新接口已拆分到独立文件，但旧接口未迁移。

### 5.2 统一响应结构未完全收口

部分接口直接返回数组或对象，不是完整的 `{ code, message, data }` 结构。新接口基本已统一，但旧接口未批量改造。

### 5.3 Celery 与 APScheduler 并存

APScheduler 是实际调度引擎，Celery Worker/Beat 配置已存在但使用有限。两者职责边界不清。

### 5.4 缺少自动化测试

当前无单元测试和集成测试覆盖。
