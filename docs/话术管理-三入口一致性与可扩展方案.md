

# 话术管理：三入口一致性与可扩展方案

> 日期：2026-05-11

> 前置参考：`docs/话术模板三层架构方案.md`、`docs/话术模板CSV导入与对齐调研报告.md`、`docs/话术模板分类管理方案.md`、`docs/话术模板增强方案.md`、`docs/shujubiaozhu/话术标注规范.md`、`docs/shujubiaozhu/RAG实现链路说明.md`

## 一、这份文档解决什么问题

话术管理目前有三个入口：

1. 前端侧边栏**手动创建**一条话术（`POST /api/v1/speech-templates`）
2. 前端**单条 RAG 配置**对话框（`PATCH /api/v1/speech-templates/{id}/rag-meta`）
3. CSV **批量导入**（`POST /api/v1/speech-templates/import-csv`）

三个入口目前字段集合不一致、去重规则不一致、分类落地逻辑不一致，带来下面 4 个痛点：

- **痛点 A**：CSV 是全字段，但单条 RAG 配置只收一半字段，手动创建完全不收 RAG 字段，导致同一条话术在不同入口填出不一样的口径
- **痛点 B**：CSV 里没有"分类"字段，当前靠 `SCENE_CATEGORY_MAP[scene_key] → (L2, L3)` 硬编码映射来归类，一旦 L3 改名/新增就漂移
- **痛点 C**：前端点侧边栏 L3 创建话术时，表单里还要让用户手填 `scene_key` 英文标识，这跟"自动识别当前所在层级"的直觉相反
- **痛点 D**：L1/L2/L3 以后会越来越细（Excel 预定义了 4×18×72，未来还会扩），三条入口的字段和词表如果不收敛到同一中心，后面每增加一类分类都要改三处

本文给出一份**可落地、可扩展**的收敛方案。不推翻已有 3 级分类架构，而是在它上面补齐"单一事实源 + 一致性门禁"。

## 二、当前现状（基于代码真值的审查）

### 2.1 数据真值

**`SpeechTemplate`（`app/models.py:326`）**

```
id, scene_key(64), style(32), label(128), content(Text),
metadata_json(Text|null), is_builtin, owner_id, category_id → speech_categories.id
```

注意：当前模型里**没有**独立的 `summary / customer_goal / intervention_scene / question_type / safety_level / visibility / tags / usage_note / reviewer / review_note` 字段，这些全部以 JSON 形式塞在 `metadata_json` 里。这是目前"字段不对称"的根本结构原因。

**`SpeechCategory`（`app/models.py:316`）**

```
id, name(64), parent_id(self→), level(1|2|3), sort_order
```

没有 `slug / code / code_path` 字段。目前系统以**中文 `name`** 作为分类的稳定标识在 `SCENE_CATEGORY_MAP` 中流转——一旦运营在界面上改了 L3 的中文名，映射立刻失效。

**数据库现状**（上一份调研的真值抽查）：
- `speech_categories` 分布：`{level=1: 4, level=2: 18, level=3: 72}`
- `speech_templates = 41`，全部有 `metadata_json`
- `rag_resources(source_type='speech_template') = 41`
- `rag_resources.visibility` 中仍有 4 条 `customer_sendable`（旧值残留）

### 2.2 三个入口的字段集合对比

| 字段 | CSV 导入 | 手动创建 | 单条 RAG 配置 | 落点 |
|------|:-:|:-:|:-:|------|
| `title / label` | ✔ | ✔ | ✗（另一处编辑） | `speech_templates.label` |
| `content / clean_content` | ✔ | ✔ | ✗（另一处编辑） | `speech_templates.content` |
| `scene_key` | 从 `intervention_scene/question_type` 推导 | ✔ 手填英文 | ✗ | `speech_templates.scene_key` |
| `style / tone` | ✔ | ✔ | ✗ | `speech_templates.style` |
| `category_id` | ✗ 靠 `SCENE_CATEGORY_MAP` 回填到 L3 | ✔ 从点击的 L3 带入 | ✗ | `speech_templates.category_id` |
| `status` | ✔ 作为门禁 | ✗ 默认 active | ✗ | 仅门禁，不落业务字段 |
| `summary` | ✔ | ✗ | ✔ | `metadata_json.summary` |
| `customer_goal` | ✔ | ✗ | ✔ | `metadata_json.customer_goal` |
| `intervention_scene` | ✔ | ✗ | ✔ | `metadata_json.intervention_scene` |
| `question_type` | ✔ | ✗ | ✔ | `metadata_json.question_type` |
| `safety_level` | ✔ 走词表归一 | ✗ | ✔ 但当前未校验，前端还残留 `caution/risk` 旧值 | `metadata_json.safety_level` |
| `visibility` | ✔ | ✗ | ✔ | `metadata_json.visibility` |
| `tags` | ✔ | ✗ | ✔ | `metadata_json.tags` |
| `usage_note` | ✔ | ✗ | ✔ | `metadata_json.usage_note` |
| `content_kind` | ✔（被忽略，固定 `script`） | ✗ | ✗ | 固定 `script` |
| `reviewer` | ✔（当前丢弃） | ✗ | ✗ | 无落点 |
| `review_note` | ✔（当前丢弃） | ✗ | ✗ | 无落点 |
| `owner_id` | ✔（但参数漏传——**P0 blocker**） | ✔ | ✗ | `speech_templates.owner_id` |

可以看出：

- CSV 是唯一"全字段入口"，但 `reviewer / review_note / content_kind` 三个字段进来之后就丢了
- **手动创建入口**除了基础 4 件套什么 RAG 字段都不收集——用户必须"先创建，再切换到 RAG 配置对话框"才能补标签。对"直接点击 L3 新建话术"这个动线非常不友好
- **RAG 配置对话框**反过来只管 8 个 RAG 字段，不涉及业务主字段——单独打开时用户看不到话术的全貌，容易覆盖
- **RAG 配置对话框当前不走词表校验**。CSV 导入会通过 `resolve_code / resolve_tag_values` 过滤词表值，但 `PATCH /rag-meta` 当前直接 `meta.update()` 写入；前端还提供 `caution/risk`，与现行 `safety_level` 词表不一致
- 手动创建**能继承 `category_id`**（前端传的是点击的 L3 id），但 CSV 不能传 `category_id`，只能靠硬编码映射——两条入口走的是不同的分类落地链路

### 2.3 scene_key 的双重角色冲突（关键结构问题）

`scene_key` 在系统里被当成两样东西用：

**角色 A：业务触发 Key（official）**

硬编码在以下位置，改一个都会破业务：

```text
app/services/crm_1v1_actions.py:ACTION_TYPES['praise_top3'] = {..., 'scene_key': 'top_leader'}
app/services/auto_ranking_push.py: must_scene_keys_json default=['top_leader', 'top_six']
app/services/crm_speech_templates.py: _SEED_TEMPLATES, SCENE_LABELS, SCENE_CATEGORY_MAP
app/routers/api_speech_templates.py: (scene_key, style) 唯一键
app/services/speech_template_import.py: 去重键 (scene_key, style)
```

这些是**不可改**的业务 ID，属于 official truth。

**角色 B：CSV 导入时的"分类依据"（side-effect）**

```text
speech_template_import.py::normalize_scene_key
  从 intervention_scene / question_type 里猜一个 KNOWN_SCENES 里的 code
  猜不到 → fallback 用 question_type 前 64 字节
  再猜不到 → 'rag_import'
然后 SCENE_CATEGORY_MAP[scene_key] → (L2名, L3名) → 查 L3 id → 写 category_id
```

这条链路把一个**本应由 L3 决定的"分类归属"**，强塞给一个**业务触发 Key**。带来三个问题：

1. 运营导入一批饮食科普话术，CSV 里 `intervention_scene=meal_review,qa_support`。`normalize_scene_key` 会挑到 `meal_review`（第一个 KNOWN_SCENES 命中），自动归到"饮食管理/食物选择建议"。如果这批其实是"饮食计划咨询"，运营在 CSV 里没办法表达
2. 一条 scene_key 只能归一个 L3。同一个 `meal_review` 下面真实业务场景可能要横跨"饮食计划 / 食物选择 / 饮食记录"三个 L3，但当前只能选一个
3. 未来 L3 再细分、再扩 → `SCENE_CATEGORY_MAP` 表体积爆炸，运营改 L3 中文名又会让它静默失效

**本质：`scene_key` 是业务 ID，不是分类 ID；当前代码用它承担了分类 ID 的职责，这是不可扩展的根源。**

### 2.4 已确认的 P0 Blocker（上一轮调研已定位、至今未修）

`app/routers/api_speech_templates.py`：

```python
stats = import_speech_templates_csv(db, decode_csv_bytes(raw), dry_run=dry_run, owner_id=user.id)
```

但 `app/services/speech_template_import.py::import_speech_templates_csv` 签名：

```python
def import_speech_templates_csv(db, csv_text, *, dry_run=False) -> dict:
    return import_speech_template_rows(db, parse_csv_text(csv_text), dry_run=dry_run)
```

**没有 `owner_id` 参数**。前端点"导入 CSV"会直接抛：

```
TypeError: import_speech_templates_csv() got an unexpected keyword argument 'owner_id'
```

这是当前话术管理最硬的 blocker。

### 2.5 其它已确认的中等问题

- **全量索引函数 `index_speech_templates` 和单条增量函数 `index_single_speech_template` 都还是 L2 口径**。当前话术的 `category_id` 已经指向 L3，但两处代码都只在 `category_id` 指向 L2 时才写 `category_l1/category_l2`，所以 L3 分类下无论全量重建还是 `PATCH /rag-meta` 后单条索引，分类 payload 都会缺失
- `resource_indexer._build_semantic_text_speech` 只拼 `大类/子类`（L1/L2），没有把 L3 名字写进 `semantic_text`——L3 信息没有被向量化
- Qdrant payload 也没有 `category_l3` 字段——检索时无法按 L3 做过滤或加权
- `SCENE_CATEGORY_MAP` 中文名硬编码了（如 `('饮食管理', '食物选择建议')`），运营界面上一旦重命名 L3，这份映射静默失效。没有任何检测
- 前端手动创建对话框里让用户填 `scene_key` 英文 ID——违反用户直觉
- CSV 里没有任何"指定 category"的字段；运营无法绕过 `SCENE_CATEGORY_MAP` 指定归属
- 页面 CSV 导入会因为 `owner_id` 参数失败；脚本 `scripts/import_speech_templates_csv.py --dry-run` 能跑通只说明 service 内层逻辑可用，不代表前端 API 入口可用

## 三、核心矛盾与目标

### 3.1 一句话总结当前矛盾

> **"话术的业务身份（scene_key / style）、内容（label / content）、分类归属（L1/L2/L3）、RAG 标注（customer_goal / intervention_scene / safety_level / …）"这四层被混在一起了，三个入口各自只收集其中一部分，又各自走不同的字段落地路径，所以不对齐。**

### 3.2 目标

把话术数据模型按"职责"拆成四层，三个入口共享同一套字段定义、同一套落地规则：

```
┌────────────────────────────────────────────────────────┐
│ Layer 1 · 业务身份 (official)                          │
│   scene_key  style  is_builtin  owner_id               │
│   → 不可轻易变；决定"这条话术是谁触发的"                │
├────────────────────────────────────────────────────────┤
│ Layer 2 · 内容 (official)                              │
│   label  content                                       │
├────────────────────────────────────────────────────────┤
│ Layer 3 · 分类归属 (official)                          │
│   category_id  → speech_categories.id (L3)             │
│   通过 L3 自动反推 L1/L2 路径                          │
├────────────────────────────────────────────────────────┤
│ Layer 4 · RAG 标注 (support knowledge)                 │
│   summary  customer_goal  intervention_scene           │
│   question_type  safety_level  visibility  tags        │
│   usage_note                                           │
│   → 存在 metadata_json 里；只影响 RAG support 索引      │
└────────────────────────────────────────────────────────┘
```

三个入口共同遵守：

- **共享字段集合** — 由一个 Pydantic schema 文件统一定义（`app/schemas/speech_template.py`），所有入口 reuse
- **共享词表中心** — 所有 RAG 标注的 code 都走 `app.rag.vocabulary.resolve_code / resolve_tag_values`
- **共享分类落地策略** — 所有入口都要有"显式 category_id"的能力，硬编码 `SCENE_CATEGORY_MAP` 降级为"兜底 fallback"
- **共享保存链路** — 统一进 `speech_template_service.upsert(...)`，该 service 内部统一写 `speech_templates` + `metadata_json` + 触发增量 RAG 索引

## 四、目标架构

### 4.1 分层数据模型（DDL 小改，不破坏兼容）

#### 4.1.1 `speech_categories` 加一个稳定标识列

```sql
ALTER TABLE speech_categories ADD COLUMN code VARCHAR(128) NULL;
ALTER TABLE speech_categories ADD UNIQUE INDEX uq_speech_categories_code (code);
```

- `code` 是**英文 slug**，如 `health.diet.food_choice`（点号分隔 L1.L2.L3）
- 运营在前端改中文 `name` 不影响 `code`
- `SCENE_CATEGORY_MAP`、CSV 导入、前端路由、RAG payload 都改成以 `code` 作为稳定键
- 迁移脚本：按现在 4×18×72 一次性生成 code（L1 `health`/`system`/`community`/`service`，L2 英文化，L3 英文化）

> 这是"L3 以后越分越细"的**第一个抗漂移基建**——没有这一列，未来一切依赖中文名的代码都有漂移风险。

#### 4.1.2 `speech_templates` 不改表结构，只约束 `metadata_json` 的 schema

所有 RAG 标注继续存 `metadata_json`，但必须符合下面的 JSON schema（通过 `SpeechMetadata` pydantic 模型强约束）：

```python
# app/schemas/speech_template.py
class SpeechMetadata(BaseModel):
    summary: str | None = None
    customer_goal: list[str] = []
    intervention_scene: list[str] = []
    question_type: list[str] = []
    safety_level: Literal['general','nutrition_education','medical_sensitive','doctor_review','contraindicated'] | None = None
    visibility: Literal['coach_internal','customer_visible'] | None = None
    tags: list[str] = []
    usage_note: str | None = None
    # 以下三字段当前被 CSV 丢弃，补齐
    reviewer: str | None = None
    review_note: str | None = None
    content_kind: Literal['script'] = 'script'
```

> 为什么不为每个字段建一列：当前只有 41 条、未来几百条规模，Qdrant payload 已经承担了过滤/加权，MySQL 单独建列没收益；JSON schema 约束就够了。如果以后要在业务表上直接按 `safety_level` 建索引（比如批量导出），再单独提列。

### 4.2 词表（vocabulary）作为全局唯一真值源

现状 `app/rag/vocabulary.py` 已经是受控词表，覆盖了 `customer_goal / intervention_scene / question_type / visibility / safety_level / status / source_type / content_kind`。三个入口必须全部通过它做校验。

**新增**一个维度：

```python
VOCABULARY["category_code"] = [
    ("health.diet.food_choice", "健康管理-饮食管理-食物选择建议", None, 1),
    ...
]
```

或者直接暴露 `speech_categories.code` 作为"动态词表"。建议后者 —— 因为 L3 未来会在数据库侧动态扩，而不是代码侧。

**语义：一切涉及代码层面 scene/goal/safety/visibility/category 的判定，都 route 到 vocabulary。三个入口 0 重复。**

### 4.3 统一的服务层（消除三入口逻辑漂移）

新增 `app/services/speech_template_service.py`（<=400 行），成为唯一写入 `speech_templates` 的通道：

```python
def upsert_speech_template(
    db: Session,
    *,
    # Layer 1: 业务身份
    scene_key: str,
    style: str,
    # Layer 2: 内容
    label: str,
    content: str,
    # Layer 3: 分类
    category_id: int | None = None,
    category_code: str | None = None,
    # Layer 4: RAG 标注
    metadata: SpeechMetadata | None = None,
    # 归属
    owner_id: int | None = None,
    # 控制
    index_rag: bool = True,
) -> SpeechTemplate: ...

def validate_safety_rules(metadata: SpeechMetadata) -> list[str]:
    """安全组合硬规则：
       - medical_sensitive / doctor_review 必须 coach_internal
       - contraindicated 禁止进入 active RAG
    """
    ...

def resolve_category_id(
    db: Session,
    *,
    category_id: int | None,
    category_code: str | None,
    scene_key: str,
) -> int | None:
    """分类解析规则：
       1. 同时传 category_id 和 category_code → 必须指向同一个 L3，否则报错
       2. 只传 category_id → 校验其存在且 level=3 后使用
       3. 只传 category_code → 查 speech_categories.code → id
       4. 都没传 → 退化到 SCENE_CATEGORY_MAP[scene_key] → category_code/L3 name → id (兼容层)
       5. 再没有 → 返回 None（UI 显示"未分类"）
    """
    ...
```

三个入口都调这同一个 `upsert_speech_template`。差异只剩"字段从哪儿来、默认值怎么填"，不再有第二套逻辑。

### 4.4 三入口字段策略收敛

改造后三个入口都收同一组字段，只是 UI 上用不同组合：

| 入口 | Layer 1 业务身份 | Layer 2 内容 | Layer 3 分类 | Layer 4 RAG 标注 | UI 行为 |
|------|:-:|:-:|:-:|:-:|------|
| 手动创建 | 自动填充（见下） | ✔ | 自动从点击的 L3 带入 | 表单默认展开、可跳过 | 一步搞定，可选"稍后补 RAG" |
| 单条编辑/RAG 配置 | 只读展示 | 可改 | 可改（弹分类 cascader） | ✔ | 统一成一个大编辑抽屉，不再分两个对话框 |
| CSV 导入 | 从标注字段推导 or 显式 | ✔ | 新增 `category_code` 列；留空则退化到老逻辑 | ✔ | 增加 strict dry-run + 错误报告 |

**手动创建的"自动填充"规则**：

- 点击 L3 创建时，前端知道 `category_id` 和分类路径，**不再要求用户手填 `scene_key` 英文**。
- 默认规则：`scene_key = '{l2_code}_{l3_code}'`（如 `diet_food_choice`），`label` = 用户输入的标题，`style` = 默认 `professional`。
- 用户依然可以高级展开区改 `scene_key`，但**一般用户看不见**——这解决了痛点 C。
- 在同一 L3 下，同一 `(scene_key, style)` 唯一；如果用户再点"新建话术"，自动加后缀 `_2`、`_3`，避免撞键。

**CSV 新增 `category_code` 列**：

新的标注规范里增加一列：

```
title,clean_content,summary,status,customer_goal,intervention_scene,content_kind,visibility,safety_level,question_type,tags,usage_note,tone,reviewer,review_note,category_code
```

- `category_code` 为空时：老逻辑（`SCENE_CATEGORY_MAP`）
- 非空时：直接查 `speech_categories.code`；查不到报错行级错误
- 这样运营可以**精确指定**"这条话术归到 `health.diet.food_choice`"，不再依赖模糊推导
- 老 CSV（没这列）完全兼容

**三字段补齐**：`reviewer / review_note / content_kind` 统一落到 `metadata_json`，不再丢。

### 4.5 RAG 索引链路补齐

修复 `app/rag/resource_indexer.py`：

1. 把 `index_speech_templates` 和 `index_single_speech_template` 的分类解析都改成 L3→L2→L1，消除当前两处都按 L2 处理的缺口
2. `_build_semantic_text_speech` 加入 `三级分类: {l1} / {l2} / {l3}` 一行，让 L3 名称进入向量
3. `_build_payload_from_metadata` 增加 `category_l3` 和 `category_code` 两个字段写入 Qdrant payload
4. `retriever.py` 支持按 `category_code` 做可选过滤（比如 AI 在明确当前话题时，把检索范围锁定到对应 L3）

### 4.6 可扩展性保证

| 变化 | 目标架构如何应对 |
|------|---|

## 七、风险与注意事项

1. **`scene_key` 是业务 ID，不要试图跟 L3 强绑定**。积分运营、CRM 1v1、auto ranking 仍然以 `scene_key` 硬编码命中话术，这是 official 契约；Phase 4 的自动生成只覆盖"手动新建的非运营类话术"，运营类 scene_key 照样走 `_SEED_TEMPLATES`
2. **不要把 `metadata_json` 一次性拆成列**。当前数据规模下没收益，反而限制未来灵活性；需要索引某字段时单独提列即可
3. **`category_code` 设计要一次到位**。点号分隔的英文 slug 可以 L4 延伸、可以跨层级唯一、可以写在 URL 里，比层级整数 ID 更抗漂移
4. **词表是全局真值**。任何新字段（比如未来加 `language`、`audience_age_band`）都必须先进 `vocabulary.py`，再进 `SpeechMetadata`；不允许直接写裸字符串。当前例外是 `PATCH /rag-meta` 还没有校验，这是 Phase 1 必须修掉的漂移点
5. **向后兼容**：`/scenes` 和 `/categories` 的消费方除了话术页面，还有 AI 对话场景选择、auto ranking 推送、1v1 动作生成。所有改动要保证这几个接口的响应结构 **只增不减**
6. **历史脏数据不要在重构中带走**。Phase 1~5 都不改历史 `metadata_json` 的内容，只新增字段；`customer_sendable` 清洗单独放在 Phase 6
7. **增量索引优先，全量索引兜底**。单条 save 只触发 `index_single_speech_template`；定期（每日）才走 `index_speech_templates` 做一次全量 reconcile
8. **AGENTS.md 文件红线**：`useSpeechTemplates.ts` 已接近 600 行，Phase 4/5 扩展时要同步拆分到 `components/SpeechRagMetaDialog.vue` 等子组件

## 九、架构健壮性补丁（方案之外必须补齐的细节）

本章不是可选润色，是"不加的话后面一定会踩坑"的结构性约束。前面 1~8 章描述理想状态，本章给这个理想状态上硬约束，避免实施过程中重新发明轮子或引入静默不一致。

### 9.1 事务边界：业务写入 vs RAG 索引

当前 `PATCH /{id}/rag-meta` 的写法是 `db.commit()` 之后 `await index_single_speech_template()`，这带来一个健壮性漏洞：

- 如果 RAG 索引失败（Qdrant 宕机、embedding 超时、网络抖动）
- 业务表已经写进去了，但 `rag_resources.source_hash` 还是旧的
- API 可能返回失败，但业务表实际已经变更；也可能未来被包装成 200，让用户以为索引已同步
- AI 检索仍然命中旧内容——**业务数据与 RAG support 层静默不一致**

**Phase 1 必做约定**（写进 `speech_template_service.py` 的 docstring 和测试里）：

1. 业务表写入 + `rag_resources` 状态更新 `source_hash` 在**同一个 DB 事务**里完成（即便 Qdrant upsert 失败，下次 reconcile 仍能识别）
2. Qdrant upsert 作为**事务后 side-effect**，失败时把 `rag_resources.semantic_quality = 'stale'`，同时记录一条 failure 日志
3. 定时任务每日扫一次 `semantic_quality='stale'`，自动重试
4. API 返回体里带 `rag_index: 'ok' | 'stale' | 'error'`，前端能提示"已保存，但索引同步中"

### 9.2 权限模型显式化

当前三入口权限不一致，但文档里没写：

| 动作 | 当前权限 | 是否需要调整 |
|---|---|---|
| `POST /speech-templates` | `speech_template` 权限 | ✔ 合理 |
| `PATCH /{id}/rag-meta` | `speech_template` 权限 | ✔ 合理 |
| `POST /import-csv` | `speech_template` 权限 | ⚠ 建议升级为 admin 或专门的 `speech_import` 权限，批量导入影响面大 |
| `DELETE /{id}` | admin | ✔ |
| `POST /categories` | admin | ✔ |
| `PUT /categories/{id}/move` | admin | ✔ |

**新增一条约束**：`PATCH /rag-meta` 的 `safety_level` 字段，如果从低升到 `medical_sensitive / doctor_review / contraindicated`，应该要求二次确认 + 审计日志——涉及合规边界。

### 9.3 `is_builtin` 边界

seed 数据 `is_builtin=1`。当前：

- `DELETE` 已拦 builtin
- `POST /import-csv` 的去重只查 `is_builtin=0`（正确，避免被 CSV 覆盖）
- 但 `PUT /{id}` 和 `PATCH /{id}/rag-meta` **没有**拦截 builtin

**建议**：在 service 层 `upsert_speech_template` 里加一条：

```python
if row.is_builtin == 1 and not allow_builtin_update:
    raise BizError('内置模板不可修改')
```

保留 `allow_builtin_update=True` 作为管理员显式覆盖的开关（偶尔要修 seed 文案）。

### 9.4 前端自由标签 vs 后端词表约束的静默丢失

`el-select allow-create` 允许用户输入任意 `customer_goal` 值（比如"孕期营养"）。当前有两种不一致行为：

- CSV 导入会走 `resolve_tag_values`，非法值会被过滤或归一，但导入结果没有把"被忽略的值"明确反馈给运营
- `PATCH /rag-meta` 当前不走词表校验，会把任意字符串直接写进 `metadata_json`

这两种都不应该继续存在：CSV 不能静默丢，单条配置不能静默收。

**三个修复方向**（优先级从高到低）：

1. **后端返回"被忽略的值"**：`PATCH /rag-meta` 响应体加 `dropped_values: {customer_goal: ['孕期营养']}`，前端弹 warning
2. **前端限制 allow-create**：只让不在词表里的值进入"建议词表扩展"流程，不直接写入
3. **长远**：后台加一个"词表申请"审批表，运营提交新 code → admin 审批 → 自动加进 `rag_tags` 表

### 9.5 `category_code` 命名约束与一致性

4.1.1 说 `code = health.diet.food_choice`，但要补三条**硬约束**避免漂移：

1. **层级前缀约束**：L3 的 `code` 必须以其 L2 的 `code + "."` 开头；L2 同理。这个约束在 `create_category` / `move_category` 的 service 层强校验
2. **字符集约束**：`^[a-z][a-z0-9_]{0,31}(\.[a-z][a-z0-9_]{0,31}){0,3}$`（最多 4 段，单段 ≤32 字符，数据库列使用 `VARCHAR(128)`）——给 L4 留位
3. **中文名 → code 的 slug 规则**：建一个 `slugify()` 辅助函数（拼音 or 人工指定），放在 `services/speech_category_slug.py`，避免各处各写一套

### 9.6 API 响应向后兼容清单（Phase 2 前必须做）

改 `code` 之前，先把所有"以中文 name 作判定依据"的消费方列一份清单。当前已知：

```typescript
// frontend/src/views/SpeechTemplates/composables/useSpeechTemplates.ts
const isPointsScene = computed(() => scene?.category_l2 === '积分管理')
```

**这一行是地雷**。Phase 2 落地后，运营只要把"积分管理"重命名，积分话术的占位符提示（`{name} {rank} {detail} {activity}`）就会消失。

**Phase 2 必做的前置审查**：

```bash
# 后端
rg "category_l1\s*==\s*['\"]" app/
rg "category_l2\s*==\s*['\"]" app/
rg "category_l3\s*==\s*['\"]" app/
# 前端
rg "category_l[123]\s*===?\s*['\"]" frontend/src/
```

所有命中点必须改成 `category_code === 'community.points'`（或 startsWith）这种稳定判定。

### 9.7 scene_key 冲突处理的精确定义

4.4 里说"自动加 `_2`、`_3` 后缀"，描述不够精确。实际数据库约束是 `(scene_key, style)` 唯一（没有 scene_key 单独唯一），所以：

- 同一个 `scene_key` 下可以有 3 个 style（professional / encouraging / competitive）——这是正常情况
- 冲突只会发生在：用户在同一 L3 下新建了第 2 条话术，且系统生成的 `scene_key = {l2_code}_{l3_code}` 与第 1 条撞了，style 又一样

**精确规则**：

```python
def generate_scene_key(l2_code: str, l3_code: str, style: str, db) -> str:
    base = f"{l2_code}_{l3_code}"
    if not exists(scene_key=base, style=style):
        return base
    # 找同 base 前缀的最大序号
    for n in range(2, 1000):
        candidate = f"{base}_{n}"
        if not exists(scene_key=candidate, style=style):
            return candidate
    raise BizError('同一分类下话术过多')
```

注意：如果用户只是想加一个新 style，不应该生成新 scene_key——前端要优先提示"是否要为现有话术增加一种风格"。

### 9.8 CSV 导入的事务 + 批次审计（提前到 Phase 1）

上一份调研把导入批次审计放 Phase 6，但"事务 + 批次 ID"其实是 Phase 1 就该做的，原因：

- Phase 3 让运营可以用 `category_code` 精确指定分类 → 导入规模会上来
- 没有批次 ID，一旦导入 50 条错归类，没法一次性撤销

**建议 Phase 1 顺带做**：

```sql
-- 新表 speech_import_batches
id, filename, operator_id, dry_run(bool),
created_count, updated_count, skipped_count, error_count,
error_report_json(Text), created_at

-- speech_templates 表加一列
import_batch_id INT NULL  -- 指向 speech_import_batches.id
```

这样：

- 每次导入 → 写一条 batch + 所有新增/更新的 `speech_templates` 打上 `import_batch_id`
- 出错后管理员一条 SQL 撤销：`UPDATE speech_templates SET status='disabled' WHERE import_batch_id=?`

### 9.9 Phase 间的"单向门"标注

不是所有 Phase 都可回滚。必须明确标注：

| Phase | 单向门？ | 回滚成本 |

| Phase 1 | ✗ 可回滚 | 路由层直接走老逻辑 |

### 9.10 RAG 检索 Gold Set 回归测试

Phase 2 把 L3 名称写进 `semantic_text` 之后，向量会发生变化。可能的副作用：

- 某些问法原本命中 A 话术，重建索引后命中 B 话术
- 分类文字权重变大，可能让原本"内容相关"的话术被"分类相关"的压下去

**必须建立 gold set**（放在 `tests/rag/gold_set_speech.yaml`）：


- **Phase 2 落地前**：跑一次 → 记下基线分数
- **Phase 2 落地后**：再跑一次 → 分数下降 >10% 则回滚
- **常态**：CI 里跑，每次改 `semantic_text` 构造、改 retriever 过滤逻辑必跑

### 9.11 话术 vs 素材 RAG Schema 共享（长期结构健壮性）

目前话术 `metadata_json` 的 schema 是 ad hoc 的，素材 `rag_meta_json`（`material_rag_service.py`）也是 ad hoc 的。两套字段有大量重叠（`customer_goal / safety_level / visibility / usage_note`）但独立演化。

**长期建议**（不用立刻做，但要在 Phase 1 的 `SpeechMetadata` 设计时预留接口）：

抽一个公共基类 `app/schemas/rag_annotation.py::RagAnnotationBase`，话术和素材各自继承并加自己的字段。词表也是共享的 `rag.vocabulary`。这样未来"新增一个 RAG 维度"只改一处，话术 + 素材同时生效。

## 十、shujubiaozhu 目录文档修订计划

### 10.1 现在就改（与代码真值对齐，与 Phase 无关）

#### 10.1.1 `话术标注规范.md` — 5 处纠偏

**偏差 1：`status` 跳过行为描述不完整**

当前文档："当前导入逻辑会跳过 `status` 不是 `approved` 或 `active` 的行。"

真值补充（`speech_template_import.py`）：跳过时会在 `stats.errors` 里记录 `"第 N 行状态为 xxx，跳过"`，不是静默跳过。规范里要告诉运营"跳过原因会在导入结果错误列表里看到"。

**偏差 2：`reviewer / review_note / content_kind` 描述不实**

当前文档把这三个字段写成"有就填，没有可以空着"。但 `_build_metadata_json` 实际上**不写这三个字段**（代码里完全没引用）。

加一行醒目提示：

> ⚠️ 当前已知限制：`reviewer / review_note / content_kind` 三个字段填了也会被导入逻辑丢弃，不会进入 `metadata_json`。计划在本目录同级 `docs/话术管理-三入口一致性与可扩展方案.md` 的 Phase 1 补齐。

**偏差 3：`tone` 值域不完整**

当前文档列了"专业温和 / 温柔安抚 / 实用落地 / 鼓励共情 / 客观专业"。但 `STYLE_ALIASES` 实际接受的更多（还有"鼓励 / 共情 / 专业 / 竞争 / 冲刺"）。补全并给出完整映射表：

| 填写值 | 落点 code |
|---|---|
| 专业 / 专业温和 / 客观专业 / 实用落地 | `professional` |
| 鼓励 / 共情 / 温柔安抚 / 鼓励共情 | `encouraging` |
| 竞争 / 冲刺 | `competitive` |

**偏差 4：`intervention_scene` 词表两种分工不明**

当前词表列了 12 个 code，但 `speech_template_import.py::KNOWN_SCENES` 只认 8 个。剩余 4 个（`abnormal_intervention / data_monitoring / device_guidance / points_operation`）写进 CSV 后**不会参与 scene_key 推导**，只会进 `metadata_json` 作 RAG 标签。

加一段"词表分两类"说明：

- **8 个推导 code**：会被用来推导 `scene_key`，决定话术归类
- **另外 4 个辅助 code**：只用于 RAG 标签加权，不影响归类

**偏差 5：`data_monitoring` 在 `question_type` 和 `intervention_scene` 两个维度同名**

当前文档没说清楚。补一句：

> 这两个维度各自独立，同名 code 允许共存；`resolve_tag_values('question_type', 'data_monitoring')` 只会命中 `question_type` 维度。

#### 10.1.2 `RAG实现链路说明.md` — 4 处纠偏

**偏差 1：`semantic_text` 包含 L1/L2 的说法不完整**

当前文档："索引时会把这些信息拼成 `semantic_text`：... 话术分类 L1/L2。"

真值：`resource_indexer._build_semantic_text_speech` 确实拼了 `大类 / 子类`，但 `index_speech_templates`（全量）和 `index_single_speech_template`（单条增量）解析时都只处理 `cat.level == 2` 口径，而 `category_id` 已经指向 L3，所以**全量索引和单条增量索引时 L1/L2 payload 都会缺失**。

改为：

> 当前 `semantic_text` 字段**应该**包含 L1/L2 名称，但由于全量索引函数 `index_speech_templates` 和单条增量函数 `index_single_speech_template` 的分类解析都还是 L2 口径（而模板 `category_id` 已指向 L3），批量重建和 `PATCH /rag-meta` 触发的单条增量索引都会缺失 L1/L2。此问题将在 Phase 0 修复。

**偏差 2：Qdrant payload 字段清单缺 `category_l3`**

当前文档说 payload 保存 `customer_goal / intervention_scene / question_type / visibility / safety_level / content_kind / status`。真值核对：

- ✔ 以上字段确实有
- ✗ `index_single_speech_template` 在 L3 分类下也不会写出 `category_l1 / category_l2`
- ✗ **没有 `category_l3`**（当前遗漏）

补一句：

> 当前 payload 不含 `category_l3 / category_code`，因此 retriever 无法按 L3 做精确过滤，将在 Phase 2 补齐。

**偏差 3："当前已知历史数据风险"第 4 条措辞不准**

当前文档："`question-category.xlsx` 的三级、细分分类尚未进入 RAG payload，当前只落了话术分类 L1/L2。"

真值：**连 L1/L2 在全量和单条增量索引时也会丢**（见偏差 1）。改为：

> 三级分类 L3 以及细分分类（`question-category.xlsx` 里的第 4 列）均未进入 RAG payload；当前 L1/L2 在全量重建和单条增量索引时都会缺失，需在 Phase 0 统一修复。

当前文档说"当前为空，这是实现现状"。真值：表存在但未写入。补一句避免被误解为"表已废弃"：

> 未来方案可能把标签迁移到这张关系表以便做多选过滤，但当前仍以 `metadata_json` + Qdrant payload 为准。参见 `docs/话术管理-三入口一致性与可扩展方案.md` 9.11 节。
