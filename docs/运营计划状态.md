# 运营编排中心当前实现说明

- 更新时间：2026-04-04
- 对照依据：`app/routers/api_operation_plans.py`、`app/models.py`、`app/schema_migrations.py`、`frontend/src/views/Templates/`
- 结论口径：本文档描述的是当前仓库里已经实现的“运营编排中心”能力，以及下一步最值得继续优化的方向

---

## 1. 这个功能现在解决什么问题

运营编排中心不是传统意义上的“模板列表页”，而是把运营内容按真实业务流程组织起来：

`主题 / 阶段 -> 第 N 天 -> 当天固定流程节点 -> 实际发送内容`

它主要服务这类场景：

1. 先定义一个运营主题，例如“第一阶段：调整血糖·代谢优化”
2. 一个主题下拆成 30 天或更多天
3. 每天固定有一套流程节点
4. 运营同学在节点里填写当天真正要发送的内容

这比单纯维护零散模板更贴近实际工作方式，也更方便后续批量复用。

---

## 2. 当前已经实现了什么

## 2.1 数据和接口层

后端已经新增了运营编排的正式数据模型：

1. `Plan`：运营主题
2. `PlanDay`：某个主题下的第几天
3. `PlanNode`：某一天里的流程节点

并且已经有完整的基础接口：

1. 运营主题的创建、查询、更新、删除
2. 运营天的创建、查询、更新、删除
3. 流程节点的创建、查询、更新、删除
4. 预置流程节点元数据查询
5. 按天复制编排内容

数据库迁移也已经补齐，并做了 SQLite / MySQL 的兼容修正。

## 2.2 业务预置流程

系统已经内置了与你们当前运营流程对应的 6 个默认节点：

1. 早安 / 早餐提醒
2. 积分相关发布
3. 午餐前内容发布
4. 午餐提醒发布
5. 晚餐提醒发布
6. 晚安 / 总结 / 次日预告

新建运营主题时，系统会自动按运营天数生成每天的固定节点骨架，不需要运营同学自己逐条新建。

## 2.3 前端主工作台

模板中心已经升级成双入口结构：

1. `运营编排`
2. `模板库`

其中“运营编排”已经具备以下主视图：

1. 左栏看主题 / 阶段
2. 中栏看天数展开
3. 右栏看当天流程节点
4. 下方编辑区直接编辑节点标题、说明、消息类型、发送内容、自定义变量
5. 右侧上下文区编辑当天标题和当天重点

也就是说，运营同学已经可以在一个页面里顺着“主题 -> 天 -> 节点”完成主要编排工作。

## 2.4 已实现的提效能力

当前已经补了几条非常关键的效率链路：

### A. 从模板库快速套用到当前节点

当前节点编辑区已经支持直接选择模板并套用：

1. 套用模板内容
2. 套用模板消息类型
3. 套用模板变量
4. 记录模板来源 `template_id`

这让模板库不再只是一个独立页面，而是真正成为运营编排的内容来源。

### B. 从某一天复制到当前天

已经支持选择一个来源天，把它的内容复制到当前选中的天：

1. 复制当天重点
2. 复制每个流程节点的内容
3. 复制消息类型、变量、状态等信息

这适合“先写一天，再微调后续天”的场景。

### C. 批量复制到多天

已经支持把当前天作为来源，批量覆盖多个目标天。

这适合：

1. 先编好 Day 1
2. 再铺到 Day 2 ~ Day 7
3. 然后逐天细调差异内容

### D. 同步同类节点

已经支持把当前节点同步到其他天的同类节点。

例如：

1. 只同步“午餐提醒”
2. 只同步“晚安 / 总结 / 次日预告”
3. 不影响这些天的其他流程节点

这对于“某个固定时段的内容结构一致，只改少量信息”的场景非常有价值。

---

## 3. 当前适合怎么用

基于当前实现，推荐运营同学这样使用：

1. 新建一个运营主题，设置阶段名、主题名、天数
2. 先完整编好第一天或某个代表日
3. 使用“复制到多天”把骨架快速铺到整周或整阶段
4. 对于固定时段内容，用“同步同类节点”统一结构
5. 对于单个节点，再通过“套用模板”快速填充标准内容
6. 最后逐天微调当天重点和具体文案

这套方式已经能明显减少重复输入。

## 4. 当前边界和已知不足

虽然这条线已经进入可用 MVP，但还不是最终形态。

当前主要边界如下：

## 4.1 保存体验还偏技术向

节点编辑目前仍偏“即时回写”，还没有完全收成更稳的运营体验。

这会带来两个潜在问题：

1. 长文案编辑时反馈偏碎
2. 用户不容易判断“当前内容是否已经完整保存”

## 4.2 复制能力还以当前主题内复用为主

当前的复制和同步更偏向：

1. 同一主题内的跨天复用
2. 同类节点的横向复用

但还没有做到：

1. 整周批量生成
2. 跨主题复用
3. 预设流程版本化

## 4.3 积分相关发布还没有做成专用节点编辑器

目前“积分相关发布”已经在流程层有位置，但本质上仍然在通用消息编辑器里处理。

离理想状态还差一步：

1. 录入积分数据
2. 触发 AI 生成表单
3. 表单截图回填
4. 自动进入发送内容

这条线还没有做成专属工作流。

## 4.4 还没有完成最终发送链路的深度联动

当前运营编排中心已经可以组织内容，但还没有完全打通为：

1. 从编排直接生成发送计划
2. 按节点生成定时任务
3. 自动进入审批流
4. 自动落到发送中心做最后校验

所以目前更像“运营编排工作台”，而不是“完整执行闭环”。

---

## 5. 未来最值得优化的方向

下面这部分不只是“可以继续做什么”，而是按优先级整理成一份更完整的产品和研发路线图。每项都包含：

1. 业务价值
2. 实现方向
3. 前后端影响
4. 适合作为验收的落地结果

### 5.1 P1：把编辑体验收成真正可用的运营工作台

- **业务价值**：
  当前节点和当天信息仍偏“即时回写”，运营同学编辑长文案时容易感知混乱，也不容易判断到底有没有保存成功。这会直接影响日常使用信心。
- **实现方向**：
  1. 增加显式保存按钮
  2. 增加防抖自动保存
  3. 展示“保存中 / 已保存 / 保存失败”状态
  4. 离开页面前做脏数据提醒
  5. 对节点编辑区补局部草稿缓存
- **前端影响**：
  需要把 `useOperationPlans.ts` 里当前“编辑即提交”的逻辑改成 draft + save 的状态模型；`Templates/index.vue` 需要补按钮、状态提示和离开拦截。
- **后端影响**：
  后端接口可以保持现有 `PUT` 能力不变，但建议补 `updated_by`、`updated_at` 的使用场景，便于前端展示最后保存信息。
- **验收结果**：
  运营同学可以明确看到当前编辑内容是否保存成功，长文案编辑过程中不会被频繁打断。

### 5.2 P1：补齐批量运营能力，减少重复劳动

- **业务价值**：
  固定流程型运营的核心不是“逐条写”，而是“批量铺开后微调”。如果批量能力不完整，运营编排中心只能算半成品。
- **实现方向**：
  1. 复制整周到下一周
  2. 跨主题复制天内容
  3. 按节点类型批量套模板
  4. 某个节点批量替换变量
  5. 批量修改节点发送时间、排序或启停状态
  6. 支持“仅复制结构”与“复制结构 + 内容”两种模式
- **前端影响**：
  需要补批量操作弹窗、目标范围选择器、覆盖确认提示、执行结果反馈。
- **后端影响**：
  建议补正式的批量接口，而不是完全依赖前端循环调用；同时要补覆盖策略、幂等和错误回滚策略。
- **验收结果**：
  运营同学可以在几步内完成整周或整阶段骨架铺排，而不是逐天重复操作。

### 5.3 P1：把编排结果往发送执行层打通

- **业务价值**：
  当前运营编排中心更像内容工作台，还不是执行中台。只要编排结果不能直达发送、审批、定时，运营同学就仍然要跨多个页面来回跳。
- **实现方向**：
  1. 从某一天直接生成发送计划
  2. 从某个节点直接进入发送中心预览
  3. 按天或按节点生成定时任务
  4. 对需要审批的节点直接挂审批状态
  5. 支持“编排完成后一键下发到执行层”
- **前端影响**：
  模板中心需要增加“生成发送计划 / 去发送中心 / 创建定时任务 / 提交审批”入口与结果回显。
- **后端影响**：
  需要补“运营编排 -> 消息/任务/审批”的桥接服务，避免把转换逻辑堆到前端。
- **验收结果**：
  运营同学可以从编排直接走到执行，不需要重复录入同一份内容。

### 5.4 P1：增强运营状态管理与多人协作感知

- **业务价值**：
  当一个主题有 30 天、每天 6 个节点时，没有状态视图就很难知道哪里配完了、哪里还没配、谁在改什么。
- **实现方向**：
  1. 每天完成度
  2. 每个节点的完成状态、审核状态、执行状态
  3. 未完成项筛选
  4. 最近修改记录
  5. 谁修改了什么
  6. 当前主题的整体进度看板
- **前端影响**：
  需要补状态徽标、筛选器、完成度条、最近变更面板。
- **后端影响**：
  建议在 `Plan` / `PlanDay` / `PlanNode` 维度补操作人、操作时间、状态流转字段或事件日志。
- **验收结果**：
  任何运营或管理员进入页面后，都能快速判断当前主题的配置进度和风险点。

### 5.5 P2：做积分节点的专用 AI 工作流

- **业务价值**：
  “积分相关发布”不是普通消息节点，而是典型的任务型节点。如果继续塞进通用消息编辑器，后续 AI 接入会越来越别扭。
- **实现方向**：
  1. 护士教练录入积分数据
  2. AI 自动生成表单文案或结构
  3. 自动生成截图或关联图片
  4. 一键回填到节点内容
  5. 一键发群或生成待发送消息
- **前端影响**：
  需要把 `score_publish` 做成专用编辑器，而不是继续复用通用 Markdown 编辑器。
- **后端影响**：
  需要为积分节点补专用 payload 结构、AI 处理链路、文件回填能力和失败回退逻辑。
- **验收结果**：
  积分节点可以从“手工拼内容”升级成“结构化录入 -> AI 生成 -> 一键发送”。

### 5.6 P2：把流程骨架和节点模板产品化

- **业务价值**：
  现在系统已经有 6 个默认节点，但还停留在“硬编码预置流程”。如果后面要支持不同训练营、不同主题阶段，流程骨架必须可配置。
- **实现方向**：
  1. 流程模板 / 节点模板单独管理
  2. 新建主题时选择流程骨架
  3. 支持不同主题使用不同默认节点顺序
  4. 支持自定义新增或停用某类节点
- **前端影响**：
  需要新增流程骨架管理入口，或在新建主题弹窗中增加流程模板选择。
- **后端影响**：
  需要把当前 `NODE_PRESETS` 从硬编码提升成可配置数据源。
- **验收结果**：
  运营团队可以针对不同业务线复用不同的“日运营剧本”，不再只有一套固定骨架。

### 5.7 P2：让模板库真正变成“母版层”

- **业务价值**：
  模板库目前已经能给节点套用，但还没有形成真正的母版体系。后面如果节点多、团队多人协作，模板库必须更像底层资产层。
- **实现方向**：
  1. 区分系统母版 / 我的常用模板 / 节点类型模板 / AI 生成模板
  2. 模板与节点类型建立正式关联
  3. 支持模板版本与引用提示
  4. 支持从节点反查模板来源
- **前端影响**：
  模板库需要增强筛选、来源标识、引用关系展示。
- **后端影响**：
  建议补模板类型、版本、来源、适用节点类型等元数据。
- **验收结果**：
  模板库不再只是一个列表，而是可持续复用的母版资产层。

### 5.8 P3：补版本管理、发布门禁和回滚能力

- **业务价值**：
  当运营编排开始真正承载 30 天内容时，就会出现“改错一天”“上线前不敢动”“不知道谁改坏了”的问题。
- **实现方向**：
  1. 天级和节点级版本快照
  2. 发布前校验
  3. 回滚到上一版本
  4. 差异对比
- **前端影响**：
  需要补“查看历史版本”“对比变化”“恢复版本”界面。
- **后端影响**：
  需要引入版本快照或操作日志存储策略。
- **验收结果**：
  运营编排可以安全修改、安全发布、安全回退。

### 5.9 P3：补更完整的分析、搜索和运营视图

- **业务价值**：
  当主题数和天数增加后，单纯靠浏览列表会很难管理。需要增强“找内容”和“看整体”的能力。
- **实现方向**：
  1. 按主题、天、节点、消息类型搜索
  2. 最近修改、最近发送、待完善、待审批视图
  3. 按主题统计完成率、发送率、失败率
  4. 未来可补日历视图或看板视图
- **前端影响**：
  需要补筛选、视图切换、统计组件。
- **后端影响**：
  需要补聚合接口、搜索接口和更明确的状态字段。
- **验收结果**：
  运营编排中心从“编辑器”升级成“运营驾驶舱”。

### 5.10 推荐的实施顺序

如果按投入产出比来排，最建议的顺序是：

1. `P1-编辑体验`
2. `P1-批量运营能力`
3. `P1-执行层打通`
4. `P1-状态管理`
5. `P2-积分节点 AI 工作流`
6. `P2-流程骨架产品化`
7. `P2-模板库母版层`
8. `P3-版本管理与分析能力`

这个顺序的原因是：

1. 先把日常使用体验和操作效率做扎实
2. 再把编排结果真正接到执行层
3. 最后再补更强的治理和分析能力

---

## 6. 当前阶段结论

一句话总结当前状态：

运营编排中心已经不是纯概念，而是一个可启动、可编辑、可按天复用、可按节点复用的 MVP。

它现在已经能帮助业务同学完成：

1. 按主题组织 30 天运营内容
2. 按固定流程节点管理每天内容
3. 通过复制和模板套用降低重复劳动

但离“完整的运营执行中台”还差几步：

1. 保存体验更顺
2. 批量能力更强
3. AI 节点更专用
4. 与发送 / 审批 / 定时任务联动更深

如果后续继续沿这条线推进，最建议优先把“保存体验 + 执行联动 + AI 节点专项化”这三件事做掉。
