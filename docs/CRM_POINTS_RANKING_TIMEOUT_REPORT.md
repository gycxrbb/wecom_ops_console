# CRM 积分排行预览超时分析报告

## 1. 背景

本次问题发生在“发送中心 -> 积分推送 -> 生成排行消息”链路。

已观测到的关键日志为：

```text
CRM 积分排行预览较慢: total_ms=97495
slow_groups=[
  {'crm_group_id': 1017, 'crm_group_name': '首钢集团健康领航群', 'member_count': 20, 'elapsed_ms': 4925},
  {'crm_group_id': 14, 'crm_group_name': '3首钢京唐群 3（甲丙）', 'member_count': 65, 'elapsed_ms': 4857},
  {'crm_group_id': 15, 'crm_group_name': '4首钢京唐群 4（ 乙丁）', 'member_count': 51, 'elapsed_ms': 4686},
  {'crm_group_id': 13, 'crm_group_name': '1首钢京唐群 1（白）', 'member_count': 68, 'elapsed_ms': 4644},
  {'crm_group_id': 1000, 'crm_group_name': '首钢矿业-群 1', 'member_count': 58, 'elapsed_ms': 4526}
]
```

同时，前端体感是在 60 多秒时直接失败、退出或刷新。

## 2. 结论摘要

### 2.1 核心结论

当前超时的主因，不是“单条 SQL 极慢”，而是：

1. 一次请求选中了多个群。
2. 每个群在生成排行文案时，都会额外执行一轮 `point_logs` 洞察分析。
3. 这轮洞察分析是“按群串行”执行的。
4. 单群耗时约 `4.5s ~ 5s`，二十多个群串行累计后，整体被拉到 `97s`。

换句话说，真正的瓶颈是：

`多群串行 + 每群单独查询 point_logs + 每群单独做洞察计算`

而不是“群成员查询本身”。

### 2.2 次要结论

“前端 60 多秒就退掉”与后端 `97s` 日志同时存在，说明：

1. 后端请求实际上还在继续跑，直到 97 秒才结束。
2. 前端没有等到后端返回。
3. 这更像是前端/网关/旧页面缓存里的超时阈值先触发了，而不是后端主动中断。

当前代码里新的前端请求超时已经改为 `180000ms`，如果你现场仍然稳定在 60 秒左右退出，需要优先排查：

1. 当前浏览器页面是否仍是旧前端代码，未刷新到新 bundle。
2. 是否存在反向代理 / 网关 / ingress 的 60 秒超时。
3. 是否现场访问链路不是当前本地开发环境，而是部署环境中的旧版本。

## 3. 当前生成链路复盘

当前 `/api/v1/crm-points/preview-ranking` 的主路径如下：

1. 路由层按所选群 ID 查询 CRM 群名称。
2. 批量查询所选群的成员信息。
3. 批量查询这些成员的周积分 / 月积分。
4. 然后开始逐群生成排行消息。
5. 每个群生成消息时，会调用 `point_logs` 洞察逻辑。
6. 洞察结果再转成“亮点话术”附加到 markdown 文案中。
7. 最后再批量补充本地发送群绑定关系。

其中第 1、2、3、7 步都已经做过一轮优化，属于“必要的查询”。

当前真正重的是第 4、5、6 步，尤其是第 5 步。

## 4. 当前 `point_logs` 洞察到底在干什么

当前洞察逻辑位于：

- `app/services/crm_points_ranking.py`
- `app/services/crm_points_insights.py`

### 4.1 入口行为

每个群在生成排行消息时，都会执行：

1. 从当前群里筛出“当前积分 > 0”的成员。
2. 按排序指标排序。
3. 取排行前 `top_n` 的成员用于展示。
4. 再从这些上榜成员里截取前 `20` 人，作为洞察分析对象。
5. 对这 20 人去查 `point_logs` 最近 14 天积分流水。
6. 为每个人生成场景标签，再拼成“亮点话术”。

### 4.2 `point_logs` 查询内容

当前查询是：

```sql
SELECT customer_id, type, num, des, category, created_at
FROM point_logs
WHERE customer_id IN (...)
  AND type = 0
  AND created_at >= 最近14天
ORDER BY customer_id, created_at DESC
```

含义是：

1. 只看最近 14 天。
2. 只看正向积分流水 `type = 0`。
3. 只看当前要分析的这批用户。

### 4.3 查回流水后做的分析

对每个成员，当前逻辑会尝试识别这些运营场景：

1. `头部领先`
   - 当前排名前 3 / 前 6 / 前 10。

2. `连续活跃`
   - 近 14 天内有正向流水的活跃天数是否达到阈值。

3. `积分暴涨`
   - 比较最近 3 天与最近 7 天的积分节奏，判断是否突然上涨。

4. `强势回归`
   - 最近 3 天重新有积分，但更早之前一段时间没有活跃。

5. `活动类型解析`
   - 从 `category` 和 `des` 里识别“体重打卡 / 习惯打卡 / 餐食上传 / 看课学习 / 提交阻碍”等活动类型。

最终，这些洞察再交给话术模板模块，生成诸如“谁最近连续活跃”“谁最近积分暴涨”的个性化文案。

### 4.4 这套洞察的业务价值

这套逻辑不是装饰性的，它确实在做一件运营上有价值的事：

它试图把“纯排行榜”升级成“带运营解释的排行榜”。

也就是说，它不只是在说“谁多少分”，还在试图回答：

1. 谁最近明显在冲刺。
2. 谁是连续活跃型。
3. 谁是突然爆发型。
4. 谁是沉寂后回归型。
5. 这些变化可能和什么行为有关。

所以问题不是“洞察没意义”，而是“现在的实现方式太贵”。

## 5. 为什么现在会慢到 97 秒

### 5.1 从日志可以推断出的事实

已知：

1. 整体 `total_ms = 97495`
2. 单个慢群 `elapsed_ms ≈ 4.5s ~ 4.9s`

这说明一个非常明确的模式：

不是单个群突然跑了 90 多秒，而是多个群都在 4 到 5 秒之间，然后顺序累加。

如果一次选了 20+ 个群，`20 x 4.5s ≈ 90s`，和本次日志高度一致。

### 5.2 当前最可疑的耗时段

基于当前代码路径，最重的环节不是成员批量加载，而是：

`逐群 -> 逐群调用 detect_individual_insights() -> 逐群查询 point_logs`

原因如下：

1. 群成员和周/月积分已经做成“批量查一次”了。
2. 但洞察仍然是每个群单独查询一次 `point_logs`。
3. 每个群都要新开 CRM 连接、跑一次近 14 天流水查询、再做一次内存分析。

### 5.3 现有实现的放大效应

当前实现有四层放大：

1. `按群串行`
   - 22 个群就查 22 次 `point_logs`。

2. `每群单独建连接`
   - `_fetch_recent_logs()` 每次都会重新 `_get_connection()`。

3. `每群单独扫近 14 天流水`
   - 即使这次请求里有很多群，本质仍是拆成若干次小查询。

4. `每群单独做 Python 侧二次遍历`
   - 活跃天数、3 天 / 7 天积分、回归判断、行为类型解析，都是在 Python 里再扫一轮。

### 5.4 当前缓存帮助有限

虽然 `_fetch_recent_logs()` 有 120 秒缓存，但它对本次问题帮助不大，原因是：

1. 它只对“相同客户集合”的重复请求有效。
2. 这次是第一次冷请求，缓存没有命中。
3. 即使第二次再点，如果群组合略有变化，缓存 key 就变了。
4. 所以它更像“二次点击提速”，不是“首轮请求提速”。

## 6. 当前实现里还有哪些具体性能问题

### 6.1 `point_logs` 查询粒度仍然太细

这是当前最大的性能问题。

现在是：

`一个群 -> 一次 point_logs 查询`

更合理的方式应该是：

`一次请求里所有群的洞察候选人 -> 一次 point_logs 查询`

### 6.2 洞察计算与“消息成功生成”被强绑定

当前只有在洞察也算完后，整条群排行消息才算生成完成。

这意味着：

1. 排行本体已经可以发了。
2. 但因为“亮点话术”没算完，整个消息仍然不能返回。

这是典型的“高价值增强能力阻塞主链路”。

### 6.3 查询里有 `ORDER BY`，但洞察本身并不严格依赖全局排序

当前 `point_logs` 查询使用：

```sql
ORDER BY customer_id, created_at DESC
```

但后续逻辑只是为了：

1. 按客户分组
2. 统计活跃天数
3. 求 3 天 / 7 天积分
4. 识别活动类型

这些并不一定必须依赖数据库排序完成。排序会增加额外负担，尤其在 `IN (...) + 时间范围` 的查询上。

### 6.4 索引设计仍可能不够贴合洞察 SQL

当前已经补了：

`point_logs(customer_id, created_at)`

但当前洞察 SQL 实际过滤条件是：

1. `customer_id IN (...)`
2. `type = 0`
3. `created_at >= 某时间`

因此更值得评估的索引是：

1. `(customer_id, type, created_at)`
2. 或 `(type, customer_id, created_at)`

哪个更优，要基于真实 MySQL `EXPLAIN` 决定，不能只凭感觉。

### 6.5 Python 侧重复遍历仍偏多

单个群内虽然人数不大，但当前对每个成员会做多次扫描：

1. `_count_active_days()`
2. `_sum_period_points(..., 3)`
3. `_sum_period_points(..., 7)`
4. 回归判断时再构造 `older_logs`
5. `_parse_activity_types()`

这部分通常不是 97 秒的第一瓶颈，但在数据库查回数据之后，它仍然会继续放大 CPU 耗时。

## 7. 最值得做的优化点

下面按收益 / 风险 / 落地难度排序。

### 7.1 P0：把 `point_logs` 洞察从“逐群查询”改成“整次请求批量查询”

这是当前最值得做、也是预期收益最大的优化。

建议新流程：

1. 先完成所有群的排行排序。
2. 从每个群里抽取洞察候选人，例如前 10 或前 20。
3. 合并成“这次请求的全部唯一 customer_id”。
4. 对这些客户一次性查询最近 14 天 `point_logs`。
5. 在内存里按 `customer_id` 建索引。
6. 再回填到每个群的洞察结果里。

这样会把：

`22 次 point_logs 查询`

压缩成：

`1 次 point_logs 查询`

如果当前 97 秒里大部分时间都花在这里，这一项通常就能把总耗时直接砍掉一大截。

### 7.2 P0：把洞察变成“可降级能力”，不要阻塞排行主链路

建议给积分推送增加两级模式：

1. `标准排行模式`
   - 只生成群总分 + 成员排行榜
   - 不做 `point_logs` 洞察
   - 用于多群批量快速推送

2. `运营增强模式`
   - 额外生成亮点话术
   - 用于少量群 / 精细化运营 / 单独复核

或者更稳一点：

1. 默认先返回排行正文。
2. 如果洞察超时或超预算，自动跳过亮点，不阻塞主结果。

从产品价值上看，这是最现实的折中。

### 7.3 P1：把洞察统计从“扫原始流水”改成“先聚合后判断”

当前洞察很多指标，本质上都可以先做成“客户 14 天画像”：

1. 近 14 天活跃天数
2. 近 3 天积分
3. 近 7 天积分
4. 最近一次活跃时间
5. 活动类型计数

这些可以有两种实现：

1. 请求时用 SQL 一次性按 `customer_id` 聚合出来
2. 更进一步，做成每日汇总表或缓存表

如果能做到“先按客户聚合，再按规则判断”，就不必每次都把原始 `point_logs` 全量拉回 Python 再算。

### 7.4 P1：去掉洞察 SQL 里的 `ORDER BY`

如果后续逻辑不依赖数据库侧排序，建议去掉。

当前只是为了后续 Python 分组统计，并不真的要数据库帮我们排好顺序。

这一点单独未必是决定性优化，但属于低风险、值得做的减负项。

### 7.5 P1：针对洞察 SQL 做 `EXPLAIN`，重新确定索引

这一步不能跳。

当前应重点验证：

```sql
SELECT customer_id, type, num, des, category, created_at
FROM point_logs
WHERE customer_id IN (...)
  AND type = 0
  AND created_at >= ?
```

建议在真实 CRM 库上对以下索引方案做 `EXPLAIN` 对比：

1. `(customer_id, created_at)`
2. `(customer_id, type, created_at)`
3. `(type, customer_id, created_at)`

然后再决定正式索引口径。

### 7.6 P2：把 Python 侧统计改成一次预聚合

当前对同一个客户日志会被反复扫描。

更优方式是第一次分组时就直接算出：

1. `active_days_14d`
2. `points_3d`
3. `points_7d`
4. `older_active_days`
5. `activity_type_counts`

后续场景判断直接读这个中间结果，而不是重复扫 `cust_logs`。

### 7.7 P2：连接复用或连接池化

当前每个洞察查询都单独建 CRM 连接。

如果后续仍保留多次查询模式，至少应减少“每群一次连接”的往返成本。

但这项收益大概率不如“批量查一次 point_logs”明显。

## 8. 建议的优化实施顺序

### 第一阶段：先把 97 秒打下来

目标：

把“多群积分排行预览”从接近 100 秒压到可接受范围。

建议动作：

1. 洞察候选人按整次请求合并，批量查询一次 `point_logs`
2. 洞察改为可降级，不阻塞排行主链路
3. 去掉无必要 `ORDER BY`

### 第二阶段：再把洞察本身做轻

目标：

让“带亮点话术”的积分排行也能稳定运行。

建议动作：

1. 对洞察 SQL 做 `EXPLAIN`
2. 确认正式索引方案
3. 把 Python 侧重复扫描改为一次预聚合

### 第三阶段：再考虑长期运营能力

目标：

把积分运营从“临时实时计算”升级成“可持续的运营基础设施”。

建议动作：

1. 建客户近期积分画像缓存 / 汇总表
2. 把运营洞察做成可复用服务
3. 为运营编排提供“是否附带积分洞察”的显式配置项

## 9. 关于当前 `point_logs` 洞察是否值得保留

我的判断是：

值得保留，但不应该继续以“同步阻塞主链路”的形式保留。

原因：

1. 业务价值是真实存在的
   - 它确实能把“排行”升级为“带解释的运营话术”。

2. 当前问题主要是实现方式，不是需求方向错误
   - 现在慢，是因为“逐群串行扫流水”，不是因为洞察本身不应该做。

3. 更合适的产品策略应是“排行优先，洞察增强”
   - 批量推送场景优先保证稳定生成
   - 精细化运营场景再附加洞察

## 10. 建议结论

如果要尽快解决你当前的问题，我建议按下面这个判断执行：

### 如果目标是“先让积分推送稳定可用”

优先做：

1. 多群请求的 `point_logs` 批量查询合并
2. 洞察降级为可选或超时自动跳过

### 如果目标是“保留高质量运营话术”

在上面两项之后继续做：

1. 洞察 SQL 的 `EXPLAIN`
2. 正式索引调整
3. 画像聚合或汇总表方案

## 11. 一句话判断

当前这次 97 秒超时，本质上是：

`point_logs 洞察仍按群串行实时计算，导致 20+ 个群的单群 4~5 秒耗时被线性累加。`

所以最该优化的不是“排行榜展示本身”，而是：

`point_logs 洞察的查询粒度、执行时机、以及是否必须同步阻塞。`
