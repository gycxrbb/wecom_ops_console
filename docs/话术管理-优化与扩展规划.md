## 二、当前存在的问题（按优先级排序）

### P1：分类 code 管理未闭环

**现象**：管理员在页面新建分类时，只能填 `name`，无法指定 `code`。新建出来的分类 `code` 为空。

**影响**：
- CSV 导入无法用 `category_code` 指向这个新分类
- `isPointsScene` 等基于 code 的判断对新建分类失效
- 新分类与种子分类在数据结构上不平等

**涉及文件**：
- `app/routers/api_speech_templates.py` — `CategoryCreateReq` / `CategoryUpdateReq`
- `frontend/src/views/SpeechTemplates.vue` — 分类新建 UI

### P2：CSV 导入 unknown tag 静默丢失

**现象**：CSV 中填了词表里不存在的 `customer_goal` / `intervention_scene` / `question_type` 值时，后端 `resolve_tag_values` 会静默丢弃，不报告给用户。

**影响**：
- 数据标注同学以为填了，RAG 实际没生效
- 需要人工抽查才能发现遗漏

**涉及文件**：
- `app/services/speech_template_import.py` — `_build_metadata_json`

### P3：KNOWN_SCENES 与词表不同步

**现象**：`speech_template_import.py` 中硬编码了 8 个 `KNOWN_SCENES`，但 `vocabulary.py` 中 `intervention_scene` 维度有 12 个值。新增的 4 个场景（`abnormal_intervention`、`data_monitoring`、`device_guidance`、`points_operation`）在 CSV 导入时永远走 fallback。

**影响**：
- CSV 导入时新增的干预场景话术无法正确归类
- 每次词表新增场景都需要同步改代码

**涉及文件**：
- `app/services/speech_template_import.py` — `KNOWN_SCENES`

### P4：`allow-create` 误导用户

**现象**：前端下拉框配置了 `allow-create`，用户可以输入任意值，但保存时后端会丢弃不在词表中的值。用户体验是"填了但丢了"。

**影响**：
- 标注同学心智混乱——"为什么我填的不见了？"
- 没有"申请新标签"的入口

**涉及文件**：
- `frontend/src/views/SpeechTemplates.vue` — el-select allow-create

### P5：标签管理缺乏运营入口

**现象**：后端已有 `POST /api/v1/rag/tags`，但前端没有标签管理界面。数据标注同学需要新增标签时，只能找管理员走后端接口或改代码。

**影响**：
- 词表扩展依赖开发人员
- 标注效率低

## 三、优化方案

### 3.1 分类 code 动态管理（P1）

**策略**：预设定 + 动态派生

管理员新建分类时：
- 可手动指定 code（如 `health.diet.new_item`）
- 也可留空，系统自动派生：`parent.code + '.' + pinyin_slug(name)`

#### 3.1.1 后端改动

**`CategoryCreateReq`** 增加 `code` 字段：

```python
class CategoryCreateReq(BaseModel):
    name: str
    code: str | None = None       # 手动指定，如 health.diet.new_item
    parent_id: int | None = None
    sort_order: int = 0
```

**`CategoryUpdateReq`** 增加 `code` 字段：

```python
class CategoryUpdateReq(BaseModel):
    name: str
    code: str | None = None       # 管理员可修改 code
    sort_order: int = 0
```

**`create_category`** 增加自动派生逻辑：

```python
# 如果没传 code，自动派生
if not req.code and req.parent_id:
    parent = db.query(SpeechCategory).get(req.parent_id)
    if parent and parent.code:
        req.code = f"{parent.code}.{pinyin_slug(req.name)}"
elif not req.code and not req.parent_id:
    # L1 没有 parent，用名字首字母
    req.code = pinyin_slug(req.name)
```

新增 `pinyin_slug` 工具函数（放在 `app/services/crm_speech_templates.py`）：

```python
def pinyin_slug(name: str) -> str:
    """中文名转拼音 slug。如 '新分类' → 'xin_fen_lei'。"""
    from pypinyin import lazy_pinyin
    parts = lazy_pinyin(name)
    return '_'.join(parts).lower()[:64]
```

> 不引入 pypinyin 时的 fallback：直接用 `name` 的 MD5 前缀或序号，如 `l3_auto_42`。

**code 校验规则**：
- 唯一性：`SpeechCategory.code` 有 unique 索引
- 格式：仅小写字母、数字、点、下划线
- 层级前缀：如果 parent 是 `health.diet`，子 code 应以 `health.diet.` 开头（建议而非强制，允许跨级重排）

#### 3.1.2 前端改动

分类新建对话框增加 code 输入框：

```html
<el-form-item label="分类标识">
  <el-input v-model="newCategoryCode" placeholder="如 health.diet.new_item（留空自动生成）" />
</el-form-item>
```

分类编辑时显示 code（只读或 admin 可改）。

#### 3.1.3 涉及文件

| 文件 | 改动 |
|------|------|
| `app/routers/api_speech_templates.py` | CategoryCreateReq/UpdateReq 加 code |
| `app/services/crm_speech_templates.py` | 新增 `pinyin_slug` 函数 |
| `frontend/src/views/SpeechTemplates/composables/useSpeechTemplates.ts` | 新建/编辑传 code |
| `frontend/src/views/SpeechTemplates.vue` | 新建/编辑分类 UI 加 code |

### 3.2 CSV 导入预检闭环（P2）

#### 3.2.1 unknown tag warning

`_build_metadata_json` 改为同时返回 `(metadata_json, dropped_tags)`：

```python
def _build_metadata_json(row: dict[str, str]) -> tuple[str, list[str]]:
    """Return (json_str, dropped_warnings)."""
    warnings = []
    # ... existing logic ...

    # Track dropped values
    customer_goal_raw = split_tag_values(row.get("customer_goal"))
    if customer_goal_raw:
        resolved = resolve_tag_values("customer_goal", '|'.join(customer_goal_raw))
        dropped = [v for v in customer_goal_raw if v not in resolved]
        if dropped:
            warnings.append(f"customer_goal: {', '.join(dropped)} 未命中词表")

    # ... similarly for intervention_scene, question_type ...

    return json_str, warnings
```

导入循环收集所有 warnings，追加到 `stats["warnings"]`（新字段）：

```python
stats = {
    "created": 0, "updated": 0, "skipped": 0,
    "errors": [], warnings: [], "rows": [],
}
```

前端导入结果增加 warnings 展示：

```html
<el-alert v-if="importResult.warnings?.length" type="warning" :closable="false">
  <template #title>{{ importResult.warnings.slice(0, 10).join('；') }}</template>
</el-alert>
```

### 3.3 KNOWN_SCENES 动态化（P3）

**方案**：去掉硬编码 `KNOWN_SCENES`，改为从 `rag_tags` 中读取 `intervention_scene` 维度的所有 code。

```python
# Before
KNOWN_SCENES = {
    "meal_checkin", "meal_review", ...
}

# After: import 时动态加载
def _load_known_scenes() -> set[str]:
    from ..rag.tag_cache import tag_cache
    return set(tag_cache.get_codes("intervention_scene"))
```

## 五、CSV `category_code` 最佳实践

### 5.1 策略：预设定 + 动态派生

- **种子分类**（4 L1 × 18 L2 × 72 L3）的 code 在 `CATEGORY_SEED` 中预设，部署时自动 seed 到 DB
- **运营新建分类**：管理员在页面新建时，手动指定语义化 code（如 `health.diet.postpartum`），或留空自动派生
- **自动派生规则**：`{parent.code}.{pinyin_slug(name)}`，如 `health.diet.新饮食法` → `health.diet.xin_yin_shi_fa`
- **CSV 使用**：始终优先用 `category_code` 列，不再依赖 `SCENE_CATEGORY_MAP` 中文 fallback

### 5.2 预设定 vs 动态的配合

| 场景 | 谁来定 code | 例子 |
|------|-----------|------|
| 核心业务分类 | 开发时在 CATEGORY_SEED 预设 | `health.weight.loss_speed` |
| 运营新增分类 | 管理员手动或自动派生 | `community.activity.new_year` |
| CSV 批量导入发现缺失 | dry-run 报错 → 管理员先建分类 | 报错 `code 'x.y.z' 未找到` |

### 5.3 code 格式规范

```
L1:  {一级英文}                     health, system, community, service
L2:  {一级}.{二级英文}               health.diet, community.points
L3:  {一级}.{二级}.{三级英文}        health.diet.food_choice
```

- 全小写，单词用下划线连接
- 每层不超过 3 个单词
- 不使用中文、空格、特殊字符

## 六、词表管理机制

### 6.1 标签生命周期

```
新增 → 启用 → [被话术引用] → [可能需要禁用/合并]
```

| 阶段 | 操作者 | 入口 |
|------|--------|------|
| 新增 | 标注同学 / admin | 编辑抽屉内"管理"按钮 |
| 编辑（改名、别名） | admin | 同上 |
| 禁用 | admin | 同上 |
| 合并 | admin | 独立管理页（后续） |

### 6.2 新增标签规范

| 字段 | 规则 | 示例 |
|------|------|------|
| dimension | 必须是已有维度 | `customer_goal` |
| code | 英文小写蛇形，唯一 | `postpartum_weight` |
| name | 中文名，给运营看 | `产后体重管理` |
| aliases | 中文别名列表，方便 CSV 和搜索 | `["产后减重", "产后管理"]` |
| sort_order | 排序权重 | 20 |

### 6.3 标注同学操作流程

```
1. 编辑话术 → 客户目标下拉
2. 发现需要的标签不在列表中
3. 输入标签名 → 系统提示"不在词表"
4. 点击"新增标签" → 填写 code + 中文名
5. 标签立即加入下拉，可选中
6. 保存话术
```

## 八、风险与注意事项

1. **pypinyin 依赖**：自动派生 code 需要 `pypinyin` 库。如果不想加依赖，可用 MD5/序号 fallback，但 code 不够语义化
2. **code 修改的连锁影响**：管理员修改已有分类的 code 后，CSV 中引用旧 code 的行会报错。建议：code 一旦创建不鼓励修改，UI 提示"修改 code 可能影响 CSV 导入"
3. **词表膨胀**：标注同学可以自由新增标签，需要定期治理。建议每月 review 一次词表，合并近似项
4. **KNOWN_SCENES 动态化后的兼容性**：词表中 intervention_scene 的 code 和 KNOWN_SCENES 可能不完全一致。动态化后以词表为准，代码中的硬编码列表应删除
