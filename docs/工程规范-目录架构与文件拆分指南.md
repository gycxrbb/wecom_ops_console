# 工程规范：目录架构与文件拆分指南

> 本文总结项目中出现的大文件问题根因，给出前后端新增功能时的标准代码组织方式，并以一个完整功能为范本串讲前后端协作全流程。

---

## 一、为什么会出现 1000+ 行的文件

### 1.1 「再加一个函数就行」陷阱

以 `ai_coach.py` 为例，最初只是一个 `ask_ai_coach()` 函数。随着功能迭代，陆续追加了 streaming、thinking、regenerate、RAG、attachment、feedback——每次只加"一个函数"，但文件从 200 行无声膨胀到 1116 行。

**根本原因**：没有在功能边界处主动拆分，而是习惯性地往已有文件追加。

### 1.2 路由堆叠不拆包

`router.py` 把 29 个端点写在一个文件里，靠 `if _ai_coach_enabled:` 块包裹 15 个 AI 相关端点。每新增一个 AI 功能（上传、反馈、重新生成），就在这个 if 块里继续堆叠。

**根本原因**：路由层没有按资源分组，"客户列表"和"AI 反馈管理"这样完全不同的资源混在一个文件。

### 1.3 Composable 的单闭包膨胀

`useAiCoach.ts` 把会话状态、SSE 流、配置加载、历史记录、上传、反馈全部放在一个 `setup()` 闭包里。所有函数共享 20+ 个 `ref`，导致无法拆出任何一部分——动一发牵全身。

**根本原因**：初期没有区分"核心状态"和"外围能力"，把不相关的逻辑耦合在同一个响应式闭包中。

### 1.4 Vue SFC 模板不拆组件

`AiCoachPanel.vue` 的模板里塞入了侧边栏三个 Tab（历史、上下文、配置）、附件上传条、模型选择器、引用栏——这些视觉区域完全不同但没有拆成子组件。

**根本原因**：图省事直接写在同一个 template 里，没有按视觉区域/职责边界建立子组件。

### 1.5 共性规律

| 现象 | 根因 |
|------|------|
| 文件超过 300 行还没拆 | 边写边加，没有提前规划模块边界 |
| 函数之间共享大量 ref 拆不开 | 状态设计时没区分核心 vs 外围 |
| 路由文件堆叠多资源 | 没有按资源建立子路由 |
| Vue 模板又长又杂 | 没有按视觉区域拆子组件 |
| CSS 按页面堆而非按组件堆 | 样式没有跟随组件走 |

**核心教训**：文件超标的根因不是某一次写多了，而是**没有在功能增长时主动建立边界**。正确的做法是写第一个版本时就按模块边界组织，后续增量自然归位。

## 二、后端目录架构

### 2.1 功能模块化结构（标准模式）

所有新功能必须以**功能模块**为单位组织，结构如下：

```
app/
├── crm_profile/                  # ← 功能模块（已有的标准范例）
│   ├── __init__.py
│   ├── models.py                 # ORM 模型
│   ├── router.py                 # 路由聚合器（仅 include_router，不写端点）
│   ├── routers/                  # 按资源拆分的路由文件
│   │   ├── __init__.py
│   │   ├── customers.py          # 客户列表/搜索
│   │   ├── profile.py            # 档案/安全快照
│   │   ├── ai_coach.py           # AI 对话/流式
│   │   └── ...
│   ├── schemas/                  # Pydantic 请求/响应结构
│   │   ├── api.py
│   │   ├── context.py
│   │   └── enums.py
│   ├── services/                 # 业务逻辑
│   │   ├── audit.py
│   │   ├── cache.py
│   │   ├── ai/                   # 复杂服务可再建子包
│   │   │   ├── __init__.py       # 公开 API
│   │   │   ├── _helpers.py       # 内部辅助
│   │   │   └── _prepare.py
│   │   └── modules/              # 模块化数据加载器
│   └── prompts/                  # 提示词管理
│       ├── registry.py
│       ├── base/
│       └── scenes/
```

#### 场景 C：在已有路由文件里新增端点

**前提**：该端点属于同一资源且路由文件未超过 300 行。

```
# routers/customers.py — 文件不到 300 行 → 直接追加
@router.get("/export")            # 新增端点
def export_customers(...):
    ...
```

**如果路由文件已接近 300 行**：必须拆出新文件，不能继续追加。

### 2.3 分层职责

| 层 | 文件位置 | 职责 | 禁止 |
|----|---------|------|------|
| 路由层 | `routers/*.py` | 参数校验、调用 service、返回响应 | 写业务逻辑、写超过 5 行的 DB 查询 |
| 服务层 | `services/*.py` | 业务编排、外部调用、状态管理 | 直接处理 HTTP 请求/响应 |
| 数据层 | `models.py` | ORM 定义、复杂查询封装为 classmethod | 调用外部服务 |

### 2.4 路由聚合器模板

```python
# crm_profile/router.py — 聚合器，不写端点
from fastapi import APIRouter
from ..config import settings
from ..route_helper import UnifiedResponseRoute
from .routers import customers, profile, ai_config

router = APIRouter(
    prefix="/api/v1/crm-customers",
    tags=["crm-customers"],
    route_class=UnifiedResponseRoute,
)

router.include_router(customers.router)
router.include_router(profile.router)
router.include_router(ai_config.router)

# 条件注册
if settings.some_feature_enabled:
    from .routers import some_feature
    router.include_router(some_feature.router)
```

### 2.5 服务子包拆分模板

当一个 service 文件超过 300 行：

```python
# services/ai/__init__.py  — 公开 API（只做 re-export）
from ._core import ask_ai_coach, stream_ai_coach_answer
from ._helpers import some_helper

# services/ai/_core.py      — 核心逻辑
# services/ai/_helpers.py   — 辅助函数
# services/ai/_types.py     — 数据类/常量
```

**规则**：`_` 前缀的文件是内部实现，外部只通过 `__init__.py` 导入。

### 3.3 全局 vs 局部：放哪里的唯一判断标准

新增一个组件或 composable 时，面对两个放置位置，决策标准只有一句话：

```
被 ≥2 个不同模块引用 → 放 src/components/ 或 src/composables/     (全局)
永远只被 1 个模块引用 → 放 views/<模块>/components/ 或 composables/ (局部)
```

**为什么这样分？**

- 放在局部：读代码时能一眼看出「这个文件只服务于这个页面」，不污染全局命名空间，后续如果不再被引用可以直接随页面删除。
- 提到全局：说明这个抽象被多处复用，改它会影响多个页面，需要更谨慎的测试和版本兼容。

**判断是否"被第二个模块引用"的时机**：当你要第二次 import 同一个文件时，就是提到全局的时机。不要提前——如果一开始就放全局但只被一个地方用，反而增加了查找成本。

**常见误判**：

| 情况 | 正确做法 | 理由 |
|------|---------|------|
| "以后可能会复用" | 先放局部 | 没发生的需求不算数；真复用时再提，IDE 重构一键搞定 |
| "两个页面都用了同一个 composable，但逻辑不完全一样" | 抽公共部分到全局，差异部分各自局部保留 | 全局只放真正共通的 |
| "是一个通用工具函数" | 放全局 `utils/` | 工具函数天生就是跨模块的 |
| "是一个业务表单组件" | 放局部 | 业务表单通常绑定特定的 schema 和流程 |

**同理适用于后端**：

```
被 ≥2 个模块导入的 service  → 放 app/services/ 或 app/clients/
只被 crm_profile 使用的 service → 放 app/crm_profile/services/
```

**现状检查**：如果你发现 `src/composables/useMobile.ts` 被多个页面使用 → 应该留在全局。如果发现 `src/composables/useAiCoach.ts` 只被 `CrmProfile` 使用 → 应该在 `views/CrmProfile/composables/` 下（事实上它就在那里，符合规则）。

### 3.4 文件拆分决策树

#### TypeScript 文件（.ts）

```
文件是否超过 400 行？
├── 否 → 保持现状
└── 是 → 能否提取纯函数/类型？
    ├── 能 → 提取到独立文件（如 types.ts、utils.ts）
    └── 否 → 能否拆出独立的 composable？
        ├── 能 → 提取到 useXxx.ts，主 composable 组合调用
        └── 否 → composable 内部按功能段用注释分隔，但必须计划重构
```

#### Vue SFC 文件（.vue）

```
<template> 是否超过 200 行？
├── 否 → 检查 <script> 是否超过 300 行
│   ├── 否 → 保持现状
│   └── 是 → 提取 <script> 中的逻辑到 composable
└── 是 → 模板中有视觉/逻辑独立的区域吗？
    ├── 有 → 拆成子组件（接收 props，emit 事件）
    └── 没有 → 重新审视页面设计，可能需要重新划分组件
```

### 3.4 Composable 设计原则

**原则：核心状态与外围能力分离**

```typescript
// ❌ 反模式：所有东西塞进一个闭包
export function useAiCoach() {
  const sessionId = ref('')
  const chatHistory = ref([])
  // ... 20 个 ref
  const sendChat = () => { /* 引用上面所有 ref */ }
  const uploadFile = () => { /* 只需要 customerId */ }
  const submitFeedback = () => { /* 只需要 chatHistory */ }
  return { /* 37 个导出 */ }
}

// ✅ 正确模式：按能力拆分，主 composable 组合
// sseStream.ts — 无状态纯函数
export function streamSse(...) { ... }

// useAiFileUpload.ts — 独立能力
export function useAiFileUpload(deps: { loading, uploadFn, getCustomerId }) {
  const pendingAttachments = ref([])
  // ...
  return { pendingAttachments, addAndUploadFile, ... }
}

// useAiCoach.ts — 核心 + 组合
export function useAiCoach() {
  // 核心状态
  const chatHistory = ref([])
  const loading = ref(false)
  // ...

  // 组合外围能力
  const { pendingAttachments, ... } = useAiFileUpload({
    loading, uploadAttachmentDirect, getCustomerId: () => props.customerId
  })

  return { chatHistory, loading, pendingAttachments, ... }
}
```

## 四、行数红线与预警机制

| 文件类型 | 红线 | 触达红线时 |
|---------|------|-----------|
| Python 路由文件 | 300 行 | 必须按资源拆出新文件 |
| Python 服务文件 | 300 行 | 必须按职责拆分子包 |
| TypeScript composable | 400 行 | 提取类型/纯函数/独立能力 |
| Vue SFC `<template>` | 200 行 | 拆子组件 |
| Vue SFC `<script>` | 400 行 | 提取 composable |
| CSS 文件 | 500 行 | 拆子组件样式并跟随组件走 |

**预警习惯**：当文件行数达到红线的 70% 时（Python 210 行、TS 280 行），就应该主动规划拆分方案。

## 五、新增功能 Checklist

每次新增功能前过一遍：

### 后端

- [ ] 确定归属模块（已有模块 or 新建模块）
- [ ] 路由文件按资源命名，不超过 300 行
- [ ] Service 层封装业务逻辑，路由层只做参数转发
- [ ] Pydantic schema 管理请求/响应结构
- [ ] 如有新 ORM 模型，检查是否需要 schema migration
- [ ] 路由聚合器（`router.py`）只做 `include_router`

### 前端

- [ ] 确定归属页面（已有页面 or 新建页面目录）
- [ ] 页面入口 `index.vue` 只做组装，不写复杂逻辑
- [ ] 复杂逻辑提取到 `composables/` 下的 composable
- [ ] 类型定义集中到 `xxxTypes.ts`
- [ ] 纯工具函数（无 Vue 响应式）提取到独立 `.ts` 文件
- [ ] 视觉独立区域拆为子组件
- [ ] CSS 跟随组件，不堆在全局样式文件

## 七、完整实战：新增一个功能的前后端全流程

> 以「客户健康打卡」功能为例，从需求到上线，串讲每一步动了什么文件、为什么这么放、前后端怎么衔接。

### 7.0 需求概要

教练可以给客户创建「健康打卡任务」（如：每日测血糖、每周称体重）。客户完成打卡后教练可查看打卡记录和统计。

**涉及的数据**：
- 打卡任务（task）：客户ID、任务类型、频率、创建时间
- 打卡记录（log）：任务ID、打卡时间、打卡值、备注

**需要的页面**：
- 教练在客户档案里看到「打卡」Tab，管理任务、查看记录

### 7.1 第一步：想清楚再动手（5 分钟设计）

动手写代码之前，先回答这些问题：

```
① 这是一个全新模块，还是挂在已有模块下？
   → 客户打卡属于客户维度的功能，挂在 crm_profile 模块下。

② 需要新数据表吗？还是读已有数据？
   → 需要两张新表：health_task 和 health_task_log。
   → 需要新增 ORM 模型。

③ 后端需要哪些接口？
   → CRUD：任务列表、创建任务、删除任务
   → 打卡记录列表、提交打卡、打卡统计

④ 前端需要什么？
   → 在 CrmProfile 页面加一个「打卡」Tab
   → 任务列表 + 创建/删除操作
   → 打卡记录时间线

⑤ 有没有跨模块依赖？
   → 打卡数据不需要 AI 能力，不需要 RAG，不涉及消息发送。
   → 纯 CRUD 功能，边界清晰。
```

### 7.2 第二步：后端 — 数据层

#### 7.2.1 新增 ORM 模型

**文件**：`app/crm_profile/models.py`（追加到已有文件）

```python
class HealthTask(Base):
    __tablename__ = "health_tasks"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    crm_customer_id: Mapped[int] = mapped_column(Integer, nullable=False, index=True)
    task_type: Mapped[str] = mapped_column(String(32), nullable=False)   # blood_sugar / weight / ...
    frequency: Mapped[str] = mapped_column(String(16), nullable=False)   # daily / weekly
    target_value: Mapped[Optional[str]] = mapped_column(String(32), nullable=True)
    is_active: Mapped[bool] = mapped_column(Boolean, default=True)
    created_by: Mapped[int] = mapped_column(Integer, nullable=False)
    created_at: Mapped[Optional[datetime]] = mapped_column(DateTime, server_default=func.now())


class HealthTaskLog(Base):
    __tablename__ = "health_task_logs"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    task_id: Mapped[int] = mapped_column(Integer, nullable=False, index=True)
    crm_customer_id: Mapped[int] = mapped_column(Integer, nullable=False, index=True)
    value: Mapped[Optional[str]] = mapped_column(String(64), nullable=True)
    note: Mapped[Optional[str]] = mapped_column(String(500), nullable=True)
    logged_at: Mapped[Optional[datetime]] = mapped_column(DateTime, server_default=func.now())
```

**为什么放这里**：`models.py` 是该模块唯一的 ORM 定义文件。新表追加到末尾即可。

#### 7.2.2 建表迁移

**文件**：`app/schema_migrations.py`（追加迁移条目）

```python
# 在 _ensure_crm_ai_phase2_columns 同级的迁移函数中追加：
"health_tasks": {
    "id": "INTEGER PRIMARY KEY AUTOINCREMENT",
    "crm_customer_id": "INTEGER NOT NULL",
    "task_type": "VARCHAR(32) NOT NULL",
    "frequency": "VARCHAR(16) NOT NULL",
    # ...
```

### 7.8 第八步：前端 — Composable

**新建文件**：`frontend/src/views/CrmProfile/composables/useHealthCheckin.ts`

```typescript
import { ref } from 'vue'
import request from '#/utils/request'
import type { HealthTask, HealthTaskLog, TaskStats } from './healthCheckinTypes'

export function useHealthCheckin() {
  const tasks = ref<HealthTask[]>([])
  const logs = ref<HealthTaskLog[]>([])
  const stats = ref<TaskStats[]>([])
  const loading = ref(false)

  const loadTasks = async (customerId: number) => {
    loading.value = true
    try {
      const res: any = await request.get(`/v1/crm-customers/${customerId}/checkin/tasks`)
      tasks.value = res.items || []
    } finally {
      loading.value = false
    }
  }

  const createTask = async (customerId: number, data: {
    task_type: string; frequency: string; target_value?: string
  }) => {
    const res: any = await request.post(
      `/v1/crm-customers/${customerId}/checkin/tasks`, data
    )
    tasks.value.unshift(res)
    return res
  }

  const deleteTask = async (customerId: number, taskId: number) => {
    await request.delete(`/v1/crm-customers/${customerId}/checkin/tasks/${taskId}`)
    tasks.value = tasks.value.filter(t => t.id !== taskId)
  }

  const loadLogs = async (customerId: number, taskId?: number) => {
    const res: any = await request.get(`/v1/crm-customers/${customerId}/checkin/logs`, {
      params: taskId ? { task_id: taskId } : undefined,
    })
    logs.value = res.items || []
  }

  const loadStats = async (customerId: number) => {
    const res: any = await request.get(`/v1/crm-customers/${customerId}/checkin/stats`)
    stats.value = res || []
  }

  return { tasks, logs, stats, loading, loadTasks, createTask, deleteTask, loadLogs, loadStats }
}
```

**关键模式**：
- `request.get/post/delete` 封装了 axios + token + 错误处理
- URL 路径 `/v1/crm-customers/${customerId}/checkin/...` 与后端路由一一对应
- composable 管理 ref 状态，组件只管调用和渲染

### 7.13 高内聚低耦合：从例子看原则

**内聚**：一个文件只做一件事

| 文件 | 只负责 | 不碰 |
|------|--------|------|
| `models.py` | 表结构定义 | 业务逻辑 |
| `schemas/health_checkin.py` | 请求/响应数据结构 | 数据库操作 |
| `services/health_checkin.py` | 业务逻辑（查/增/删/统计） | HTTP 请求响应 |
| `routers/health_checkin.py` | 接收参数、调 service、返回 | 业务规则 |
| `healthCheckinTypes.ts` | TypeScript 类型 | API 调用 |
| `useHealthCheckin.ts` | 前端状态 + API 调用 | DOM 渲染 |
| `HealthCheckinPanel.vue` | UI 渲染 + 用户交互 | 网络请求细节 |

**耦合**：层与层之间通过最小接口通信

```
┌─────────────┐    props / events    ┌──────────────────┐
│  index.vue   │ ◄──────────────────► │ HealthCheckinPanel │
│  (页面入口)   │    :customer-id      │   (子组件)         │
└─────────────┘                       └────────┬─────────┘
                                                │ 调用
                                                ▼
                                       ┌──────────────────┐
                                       │ useHealthCheckin  │
                                       │   (composable)    │
                                       └────────┬─────────┘
                                                │ request.get/post
                                                ▼
                                       ┌──────────────────┐
                                       │   后端 API 层      │
                                       │ routers/xxx.py    │
                                       └────────┬─────────┘
                                                │ 调用
                                                ▼
                                       ┌──────────────────┐
                                       │  后端 Service 层   │
                                       │ services/xxx.py   │
                                       └────────┬─────────┘
                                                │ 操作
                                                ▼
                                       ┌──────────────────┐
                                       │  数据库 (SQLite)   │
                                       └──────────────────┘
```

每一层只依赖下一层的公开接口，不跨层访问：
- 前端组件不直接调 axios，通过 composable
- 后端路由不直接写 SQL，通过 service
- Service 不处理 HTTP 请求，只接收参数返回结果

### 7.15 常见错误与避坑

| 错误 | 原因 | 解决 |
|------|------|------|
| 前端请求 404 | URL 路径写错或后端没注册路由 | 检查 `router.py` 是否 include_router，检查 URL 是否带 `/api` 前缀 |
| 后端 422 Validation Error | 前端字段名和 schema 不一致 | 对比 `CreateTaskRequest` 和前端 `request.post` 的 body |
| 前端拿到数据但不显示 | 后端返回 `{ data: { items: [] } }` 但前端直接取 `res.items` | 打开 console 看 res 实际结构（本项目用 `UnifiedResponseRoute` 自动包了一层 `data`） |
| 后端启动报 `ImportError` | 相对导入层级错误 | 子路由文件在 `routers/` 下，导入上层用 `..`，导入 app 层用 `...` |
| 数据库没有新表 | 忘记执行 schema migration | 确认 `schema_migrations.py` 中有新表的定义 |
