### 旧服务器后续会受影响吗？

**完全不影响。本次代码改动 100% 向后兼容：**

- `app/clients/crm_db.py` 加了 fallback：若 `CRM_TEST_DB_*` / `CRM_PROD_DB_*` 全部为空，**自动回退到旧的 `CRM_ADMIN_DB_*`**，启动日志会显示 `legacy_fallback=True`。
- 旧机 `.env` **完全不需要改**，git pull 拿到新代码后 `docker compose build && up -d`，照常运行。
- 老配置走的就是 `mfgcrmdb`（测试 CRM），和今天的行为一模一样。
- 想要把旧机的 .env 升级到新格式（明确写 `CRM_TEST_DB_*`、`APP_ENV=development`），随时可以做，不做也没问题。

## 1. CRM 双环境配置原理

### 1.1 选择规则（优先级从高到低）

```
1. CRM_ENV=production              → 走 CRM_PROD_DB_*
2. CRM_ENV=test                    → 走 CRM_TEST_DB_*
3. CRM_ENV 为空 + APP_ENV=production → 走 CRM_PROD_DB_*
4. CRM_ENV 为空 + APP_ENV=其他      → 走 CRM_TEST_DB_*
5. 选中的那组 host 为空              → 回退到旧 CRM_ADMIN_DB_*（向后兼容）
```

启动日志会打印一行：

```
CRM DB initialized: env=production, host=rm-2zew061b0tlpd75m2.mysql.rds.aliyuncs.com, database=habitilitydb, legacy_fallback=False
```

如果看到 `legacy_fallback=True`，说明走的是老 .env 字段，需要补 `CRM_TEST_DB_*` / `CRM_PROD_DB_*`。

### 1.2 .env 模板（推荐三机统一格式）

```env
# 三机通用：在 .env 同时写两套，APP_ENV 控制走哪套
APP_ENV=production           # 本地填 development；旧服务器填 development；新服务器填 production
CRM_ENV=                     # 留空跟随 APP_ENV；显式 test/production 可临时覆盖

# 测试 CRM 库（旧 mfgcrmdb）
CRM_TEST_DB_HOST=rm-2zedqey0120581pj9do.mysql.rds.aliyuncs.com
CRM_TEST_DB_PORT=3306
CRM_TEST_DB_USER=mfgcrmuser
CRM_TEST_DB_PASSWORD=Mfg@135!%
CRM_TEST_DB_NAME=mfgcrmdb

# 生产 CRM 库（新 habitilitydb）
CRM_PROD_DB_HOST=rm-2zew061b0tlpd75m2.mysql.rds.aliyuncs.com
CRM_PROD_DB_PORT=3306
CRM_PROD_DB_USER=habituser
CRM_PROD_DB_PASSWORD=hbt@135!%
CRM_PROD_DB_NAME=habitilitydb

# 旧字段保留兼容，新增的环境字段填了就不需要
# CRM_ADMIN_DB_HOST=...
CRM_ADMIN_TABLE_NAME=admins
CRM_ADMIN_AUTH_ENABLED=true
```

**安全说明**：把生产密码同时写在开发机 .env 里有泄露风险。如果不放心，开发机的 `CRM_PROD_DB_*` 可以留空，本地永远跑测试库；新服务器的 `CRM_TEST_DB_*` 也可以留空，生产永远跑生产库。

### 2.5 内网放行（不配 Nginx / SSL）

新服务器只服务同内网的 CRM，**不需要配域名、不需要 SSL、不需要 Nginx**。`docker-compose.prod.yml` 用的是 `network_mode: host`，所以容器内的 8000 端口直接监听在宿主机的 `0.0.0.0:8000`，CRM 调 `http://<新服务器内网IP>:8000` 即可。

#### A. 阿里云 / 云厂商安全组

在阿里云控制台 → ECS → 安全组 → 入方向：

| 协议 | 端口 | 授权对象 | 用途 |
|---|---|---|---|
| TCP | 22 | 你的运维 IP 或 0.0.0.0/0 | SSH |
| TCP | **8000** | **同 VPC 网段**（如 `172.16.0.0/12`）或 **CRM 服务器的具体内网 IP** | CRM 调本服务 |

**不要给 8000 端口开 0.0.0.0/0**，否则等于暴露在公网。

#### B. 系统防火墙

```bash
# 如果用了 firewalld
firewall-cmd --permanent --add-port=8000/tcp
firewall-cmd --reload

# 如果用了 ufw
ufw allow from <CRM内网IP> to any port 8000

# 阿里云 ECS 默认很多时候 iptables/firewalld 都没开，先确认状态：
systemctl is-active firewalld 2>/dev/null
ufw status 2>/dev/null
```

#### C. 验证 CRM → 本服务连通

在 CRM 服务器上：

```bash
# 内网连通
curl http://<新服务器内网IP>:8000/api/v1/health
# 期望：{"status":"ok"}
```

#### D. CORS

`.env.production.example` 已经默认把 `127.0.0.1`、`localhost`、`39.106.212.142` 放进 `CORS_ALLOWED_ORIGINS`。如果 CRM 用的是 server-to-server（不走浏览器），CORS 可以**留空**（更安全）；如果 CRM 前端浏览器直调，把 CRM 的 origin 加到这里。

#### E. 旧机处理

旧机**不立即下线**，停掉容器即可：

```bash
ssh root@【旧服务器IP】
cd /www/wwwroot/wecom-ops-console
docker compose -f docker-compose.prod.yml down
```

保留 ≥ 1 周作为热备。如果想让旧机继续作为"测试预发布环境"长期运行，把它的 `.env` 升级到 [`.env.staging.example`](../../.env.staging.example) 的格式，然后 `docker compose up -d` 拉起。它走的 CRM 仍然是 mfgcrmdb 测试库，不影响生产。

### 3.1 三机角色定位

| 机器 | APP_ENV | CRM 库 | 用途 |

| 本地（Windows）| development | mfgcrmdb（测试） | 开发 / 调试 |

### 3.3 数据库 schema 变更

`wecom_ops` 业务库的结构变更走应用启动时的 `Base.metadata.create_all(checkfirst=True)`，**新增表/列会自动建**，删除/重命名列必须手写 SQL。

如果需要在生产手动跑 ALTER：

```bash
docker exec -it wecom-ops-console bash
mysql -h127.0.0.1 -uwecom_ops -p wecom_ops
# ALTER TABLE ...
```

CRM 库**只读**，不存在 schema 漂移问题。

### 3.4 数据备份（推荐每日 cron）

新服务器加一条定时备份：

```bash
crontab -e
# 每天凌晨 3 点 dump wecom_ops（只 dump SQL，跳过 Qdrant/uploads 避免占空间）
0 3 * * * cd /home/xr/wecom-ops-console && SKIP_QDRANT=1 SKIP_UPLOADS=1 bash scripts/migrate/export_on_old_server.sh > /var/log/wecom_ops_backup.log 2>&1
# 保留 14 天，删 14 天前的备份目录
30 3 * * * find /tmp -maxdepth 1 -name "wecom-ops-migrate-*" -mtime +14 -exec rm -rf {} \;
```

> `export_on_old_server.sh` 脚本名虽然带"old_server"，但它和服务器无关，**任意机器都可以跑**——只要项目结构正确就行。

### 3.5 临时把生产切回测试库（应急排查用）

如果生产怀疑被 CRM 数据污染，又不想动 .env，可以临时：

```bash
# 在新服务器
ssh root@39.106.212.142
vim /home/xr/wecom-ops-console/.env       # 加一行 CRM_ENV=test
cd /home/xr/wecom-ops-console
docker compose -f docker-compose.prod.yml restart wecom-ops-console
docker logs --tail 30 wecom-ops-console | grep "CRM DB initialized"  # 确认 env=test
# 排查完删掉 CRM_ENV 行 + restart 恢复
```

### 3.6 本地开发临时连生产 CRM

只读权限下临时验证生产 CRM 数据：

```env
# 本地 .env 临时改
CRM_ENV=production
# 配上生产 CRM_PROD_DB_*
```

**用完务必改回**，避免本地误改写到生产。`habituser` 应当只授 `SELECT` 权限，从权限层兜底。

## 4. 回滚

| 场景 | 操作 |
|---|---|
| 新机启动失败 | `docker logs --tail 200 wecom-ops-console` 看日志；多数是 .env 字段拼错 |
| 新机数据有问题 | 把旧机容器重新 `up -d`，DNS 切回旧机 IP |
| CRM 库连错 | 改 .env 的 `CRM_ENV` 或 `APP_ENV`，`docker compose restart` |
| 完全推倒重来 | `import_on_new_server.sh` 加 `FORCE_DROP_DB=1` 重跑 |

## 5. 已知踩坑预防

| 坑 | 现象 | 预防 |
|---|---|---|
| .env 漏改 APP_ENV | 新机仍连到测试 CRM | 启动日志看 `CRM DB initialized: env=...` 那行 |
| APP_SECRET_KEY 与旧机一致 | webhook 解密可能能用，但密钥泄露面扩大 | 重新生成，新老隔离 |
| MySQL 字符集不一致 | 中文乱码 | 新机 `CREATE DATABASE` 必须 `utf8mb4 / utf8mb4_unicode_ci` |
| Qdrant 拷贝时旧机仍在写 | 索引可能不一致 | 导出前可短暂 stop 应用；或导入后跑一次重建脚本 |
| Docker 端口被旧 Nginx 占用 | `bind: address already in use` | `ss -lntp` 看是不是 8000/8080/443 被占；按 [`deploy-guide.md §6.9`](../../deploy-guide.md) 收口 |
| 七牛 webhook 走签名密钥 | 沿用即可，不需要换 AK/SK | — |
