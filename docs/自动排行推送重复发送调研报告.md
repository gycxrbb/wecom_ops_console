# 自动排行推送重复发送调研报告

- **日期**: 2026-05-12
- **影响事件**: 5/12 00:00 自动排行推送
- **涉及模块**: `app/services/auto_ranking_push.py`, `app/services/scheduler_service.py`
- **严重程度**: 高（产生重复消息，污染企微群，误报成功数）

---

## 一、用户报告的现象

1. 企微群中出现大量重复消息，顺序混乱
2. 发送记录中无任何报错
3. 发送记录中成功条数 > 理论成功条数
4. 企微群连续出现两个"发送成功，发送了 21/21"通知
5. 手动推送正常，仅自动推送有问题

---

## 二、前置问题：为什么自动推送会失败？

用户明确指出：**手动推送排行从未失败，自动推送却会失败触发重试**。这是整个问题的起点。

### 对比两条发送路径

| 维度 | 手动推送排行 (`api.py` → `perform_job_send`) | 自动推送排行 (`auto_ranking_push.py`) |
|------|----------------------------------------------|--------------------------------------|
| 入口 | `_send_to_single_group` | `_send_batch` |
| per-message 重试 | **有**（`max_retries=3`，`send_retry_delay_seconds=1.0`） | **无**（任何异常直接 `failed += 1`） |
| 限流计数 | 独立计数器 `group_send_count`，达 20 条主动等 61s | 完全依赖 WeComService 内存限流桶 |
| 限流错误处理 | 检测 `45009`/`freq out of limit` 等关键字，等 30-35s 重试 | 无，异常直接失败 |
| 企微 API 临时错误 | 自动重试 3 次 | 不重试，计为失败 |

### 根因：缺少 per-message 重试

企微 webhook API 在生产中会出现**临时性错误**：
- 网络超时（30s 超时限制）
- 企微服务端偶发 500
- API 频率限制（45009 errcode）
- 消息体过大（markdown > 4096 字节）

手动推送在 `_send_to_single_group` 中有完整的 per-message 重试（最多 3 次），遇到限流错误还会长等待后重试。所以这些临时性错误被透明地消化了，用户感知不到。

自动推送的 `_send_batch` **没有任何 per-message 重试**——只要 `WeComService.send()` 抛出任何异常，这条消息就计入 `failed`，然后可能触发整队重试。

**一旦有 1 条消息因临时网络抖动失败 → `failed > 0` → 触发整队重试 → 删库 → 重发全部 21 条 → 企微群里出现重复消息。**

这就是"手动从不失败，自动总是出问题"的根本原因。

## 三、Bug 分析

### Bug #1：整队重试删库记录但无法撤回企微消息（致命）

**文件**: `auto_ranking_push.py` 第 180-198 行

```python
# 删除上一轮的 Message 记录
old_msgs = db.query(Message).filter(
    Message.source_type == 'auto_ranking',
    Message.source_id == cfg.id,
).all()
for om in old_msgs:
    db.query(MessageLog).filter(MessageLog.message_id == om.id).delete()
    db.delete(om)
db.commit()

# 整队重发
sent, failed, last_error = await _send_batch(pending_items)
```

**问题**: 重试时先删除数据库中所有 Message/MessageLog 记录，然后重新发送全部消息。但企微消息一旦发送就无法撤回，导致：
- 企微群中每条消息出现 N+1 次（首轮 + N 轮重试）
- 删除 DB 记录后，发送记录中看不到首轮的任何痕迹
- 看起来"没有报错"，实际是首轮的错误记录被删了

**这就是重复消息和"无报错"的直接原因。**

### Bug #2：`sent` 计数器被重试覆盖

**文件**: `auto_ranking_push.py` 第 198 行

每轮重试都会 `sent, failed, last_error = await _send_batch(pending_items)`，覆盖上一轮的计数。最终 DB 里只记录最后一轮的数量，但企微已经收到了所有轮次的消息。

**这就是"成功条数 > 理论成功条数"的原因**：DB 记录的是最后一轮 21 条，但企微实际收到了 21 × (重试轮数+1) 条。

### Bug #3：推送完成通知也被重复发送

**文件**: `auto_ranking_push.py` 第 201-213 行

每轮重试结束后，如果 `sent > 0`，都会发送一次"推送完成"通知。如果首轮成功 19 条失败 2 条触发重试，重试又成功 21 条，则企微群收到两个"发送完成"通知。

**这就是"两个发送成功 21/21"的原因。**

### Bug #4：冷却期时区不一致

**文件**: `auto_ranking_push.py` 第 69-73 行

```python
# last_run_at 是用 _dt.now(tz) 写入的上海时间，无 tzinfo
last_run_utc = cfg.last_run_at.replace(tzinfo=None) if cfg.last_run_at.tzinfo else cfg.last_run_at
elapsed = (datetime.utcnow() - last_run_utc).total_seconds()
```

**问题**: `scheduler_service.py` 第 169 行用 `_dt.now(tz)` 写入 `last_run_at`，这是带上海时区的时间。但 `replace(tzinfo=None)` 只去掉时区标记不转换时差，然后与 `datetime.utcnow()` 比较，差 8 小时。

**影响**: 冷却期检查完全失效。如果 `last_run_at` 存的是 `2026-05-12 00:01:00+08:00`，`replace(tzinfo=None)` 变成 `2026-05-12 00:01:00`（本地时间），与 `utcnow()`（比上海慢 8 小时）比较，`elapsed` 算出来是 8 小时多，远超 300s 冷却期，永远触发不了保护。

### Bug #5：重试通知和完成通知的异常被静默吞掉

**文件**: `auto_ranking_push.py` 第 194 行和第 212 行

```python
except Exception:
    pass
```

两处 `except Exception: pass` 使得重试通知和完成通知的失败完全无日志，无法排查通知是否发送成功。

### Bug #6：`last_run_at` 异常时不写入

**文件**: `scheduler_service.py` 第 172-180 行

```python
except Exception as exc:
    logger.exception('Auto ranking %s failed', cfg_name)
    try:
        cfg = db2.query(AutoRankingConfig).get(cfg_id)
        if cfg:
            cfg.last_error = str(exc)[:200]
            db2.commit()
    except Exception:
        pass
```

异常分支只写 `last_error`，不写 `last_run_at`。结合 Bug #4 冷却期失效，如果执行过程中异常，下次触发时没有冷却保护，可能立即重复执行。

### Bug #7：cron 提前 1 分钟触发的跨午夜计算有边界问题

**文件**: `scheduler_service.py` 第 190-191 行

```python
prep_minute = (minute - 1) % 60
prep_hour = (hour - 1) % 24 if minute == 0 else hour
```

当推送时间为 00:00 时，prep 时间为 23:59（前一天），正确。但当推送时间为 01:00 时，prep 为 00:59，也正确。这个逻辑本身没问题，但与 `auto_ranking_push.py` 第 62-63 行的跨午夜判断配合时：

```python
target_time = now.replace(hour=push_hour, minute=push_minute, ...)
if target_time <= now - timedelta(minutes=1):
    target_time += timedelta(days=1)
```

如果 cron 在 23:59 触发，`now` 是 23:59，`target_time` 设为 00:00（当天），此时 `00:00 <= 23:59 - 1min = 23:58` 为 True，`target_time` 推到明天 00:00。这样 `today` 变成明天，`wait_seconds` 约 24 小时，导致推送延迟到下一天。

**但实际上**由于 cron 触发时间精确，23:59:xx 触发时 `target_time = 00:00:00`，`00:00:00 <= 23:59:xx - 60s` 为 True，所以 `target_time` 被推到明天。这个 bug 在 00:00 推送场景下是隐性的（因为 wait_seconds 约 24h >> 120s 阈值，不会等待），但 `today` 判断用了明天的日期，可能影响跳过日期/周末判断。

### Bug #8：`pending_items` 在重试轮之间不重新生成

**文件**: `auto_ranking_push.py` 第 142 行

`pending_items` 在第 142 行一次性生成，重试时直接复用，不会重新从排行数据生成。这意味着即使数据在几分钟内变化，重试仍然发送旧数据。

### Bug #9：Message 删除范围过宽

**文件**: `auto_ranking_push.py` 第 180-186 行

```python
old_msgs = db.query(Message).filter(
    Message.source_type == 'auto_ranking',
    Message.source_id == cfg.id,
).all()
```

删除条件只有 `source_type` 和 `source_id`，没有时间范围。如果同一天内有其他正常发送记录，也会被误删。

### Bug #10：重试通知使用与正式推送相同的 webhook

**文件**: `auto_ranking_push.py` 第 190-193 行

```python
await WeComService.send(webhook, 'text', {
    'content': f'⚠️ 自动排行推送存在失败条目，正在第 {retry_round} 次整队重试...'
}, group_key=f'auto-ranking-retry-{cfg.id}')
```

重试通知也发到目标群，用户会看到"正在重试"的系统消息混在排行消息中，体验差。

## 五、修复方案

### 修复 #0（根本）：给自动推送添加 per-message 重试

这是最关键的修复——消除触发整队重试的根因。

```python
async def _send_single_with_retry(webhook, msg_type, content, group_key, max_retries=3, delay=2.0):
    """单条消息带重试发送。"""
    last_exc = None
    for attempt in range(1, max_retries + 1):
        try:
            return await WeComService.send(webhook, msg_type, content, group_key)
        except Exception as exc:
            last_exc = exc
            error_str = str(exc).lower()
            if '45009' in error_str or 'freq' in error_str or '限流' in error_str:
                await asyncio.sleep(30)
            elif attempt < max_retries:
                await asyncio.sleep(delay)
    raise last_exc
```

### 修复 #1：改为只重试失败条目，不删除已有记录

```python
# 不要删除首轮成功的记录，只重试失败的
# 原方案（错误）：
#   删除全部 → 重发全部
# 新方案（正确）：
#   记录首轮失败条目 → 只重发失败条目
```

### 修复 #2：累计 sent/failed 计数

```python
total_sent = sent
total_failed = failed

# 重试后累加，不覆盖
retry_sent, retry_failed, retry_err = await _send_batch(failed_items)
total_sent += retry_sent
total_failed = retry_failed  # 重试后仍失败的
```

### 修复 #3：推送完成通知只在最终发送一次

把完成通知移到所有重试轮结束后，只发一次。

### 修复 #4：统一时区处理

```python
# scheduler_service.py 中写入时去掉时区，直接存 naive UTC
cfg.last_run_at = datetime.utcnow()

# auto_ranking_push.py 中冷却期比较
elapsed = (datetime.utcnow() - cfg.last_run_at).total_seconds()
```

或者两处都使用带时区的 datetime，统一用 `datetime.now(tz)` 比较。

### 修复 #5：消除 `except Exception: pass`

至少记录 warning 级别日志：

```python
except Exception as e:
    logger.warning('Auto ranking retry notification failed: %s', e)
```

### 修复 #6：异常时也写入 last_run_at

```python
except Exception as exc:
    cfg.last_run_at = datetime.utcnow()
    cfg.last_error = str(exc)[:200]
    db2.commit()
```

### 修复 #7：target_time 跨午夜判断

改用明确日期计算，避免 `replace` + 条件加一天的隐式逻辑：

```python
# 如果 cron 在前一天 23:59 触发，目标就是今天
push_date = now.date()
if push_hour * 60 + push_minute <= now.hour * 60 + now.minute - 1:
    push_date += timedelta(days=1)
target_time = datetime(push_date, push_hour, push_minute, tzinfo=_tz)
today = push_date
```

### 修复 #9：Message 删除加时间范围（如果仍需整队重试）

```python
from_midnight = datetime.combine(today, datetime.min.time())
old_msgs = db.query(Message).filter(
    Message.source_type == 'auto_ranking',
    Message.source_id == cfg.id,
    Message.created_at >= from_midnight,
).all()
```

## 七、结论

5/12 自动排行推送的问题链条是：

1. **根因（Bug #0）**：自动推送缺少 per-message 重试，网络抖动/企微 API 临时错误导致个别消息失败
2. **放大器（Bug #1）**：整队重试时删除全部 DB 记录并重发全部消息，企微已发消息无法撤回 → 重复
3. **数据失真（Bug #2）**：sent 计数被覆盖 → 成功数虚高
4. **体验恶化（Bug #3）**：完成通知重复发送

手动推送正常是因为手动推送路径有 per-message 重试（最多 3 次），临时性错误被透明消化。

最关键的修复是 **#0：给自动推送加上 per-message 重试**。加上这个之后，绝大多数临时性错误不会触发整队重试，后续的 Bug #1/#2/#3 自然不会被触发。
