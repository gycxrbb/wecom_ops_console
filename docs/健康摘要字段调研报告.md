# 健康摘要模块字段调研报告

> 调研日期：2026-05-13  
> 调研范围：`app/crm_profile/services/modules/health_summary.py` 及其子模块、CRM schema 文档、AI 上下文构建、前端健康摘要渲染  
> 目的：核对健康摘要当前使用字段、数据来源、精度处理和展示链路，定位“数值对不上”的根因，并补齐可落地的开发闭环。

---

## 1. 审查结论

本次按代码真值核对后，原文对后端健康摘要主体逻辑的判断总体成立：当前 `health_summary_7d` 实际只读取 `customer_health` 和 `customer_glucose`，再由前端 `CrmProfile/index.vue` 直接展示 payload。

需要修正和补充的关键点如下：

| 项目 | 当前代码真值 | 文档/落地影响 |
|---|---|---|
| 模块数量 | `profile_loader.py` 当前编排 13 个 profile 模块，不只是“7 个模块之一” | 原文表述偏旧，应改为“CRM 用户档案模块之一” |
| 模块 key | `MODULE_KEY = "health_summary_7d"` 为兼容保留，真实窗口在 `payload.window_days` 和缓存 `health_window_days` 中表达 | 7/14/30 切换成立，但命名有历史包袱 |
| AI/前端标签 | `context_builder.py`、`AiCoachPanel.vue` 仍把 `health_summary_7d` 标成“近7天健康摘要” | 当用户切到 14/30 天后，标签口径可能误导，需要动态化或改为“健康摘要” |
| P0 预加载 | `profile_loader._P0_KEYS` 使用 `"health_summary"`，但正式 key 是 `"health_summary_7d"` | P0 预加载可能漏掉健康摘要，影响 AI 上下文“先可用再补全”的闭环 |
| 数据源 | 后端只读 `customer_health` 与 `customer_glucose`，`body_comp` 由独立模块读取 | 原文判断成立；BMI/心率不应在健康摘要里假装已覆盖 |
| 精度 | 体重 `round(weight/100, 1)`，血糖/血压/热量 `_avg()` 均保留 1 位，前端饮水/餐食热量再 `Math.round` | “数值对不上”的主因仍是后端和前端展示口径未统一 |
| schema 精度 | CRM schema 文档中 `decimal(4,`、`decimal(5,` 被截断 | 不能直接断言 DB 精度，只能列为待 `SHOW CREATE TABLE` 确认项 |

## 2. 模块概览

健康摘要的正式模块 key 仍是 `health_summary_7d`，这是为了兼容历史接口和缓存 key；真实统计窗口由 `window_days` 控制，目前只接受 7/14/30 天，其他值会回退到 7 天。

**数据来源**：

| 数据源表 | 定位 | 粒度 | 当前状态 |
|---|---|---|---|
| `customer_health` | 每日健康汇总宽表（业务口径） | 1 行 = 1 客户 x 1 天 | 已接入 |
| `customer_glucose` | 连续/离散血糖日数据 | 1 行 = 1 客户 x 1 天的血糖点位集合 | 已接入 |

**当前未接入健康摘要但 CRM 中存在的相关表**：

| 表名 | 定位 | 当前状态 |
|---|---|---|
| `customer_heartrate` | 日心率点位表 | 未接入 |
| `customer_sleep` | 每日睡眠明细汇总 | 未接入，仅使用 `customer_health.sleep_min` |
| `customer_sport` | 每日运动汇总 | 未接入，仅使用 `customer_health.step_count` |
| `customer_stress` | 压力点位表 | 未接入 |
| `body_comp` | 体成分日快照（体脂秤） | 由独立模块 `body_comp_latest_30d` 处理 |

## 3. 当前运行链路

### 3.1 后端加载链路

```text
profile/profile cache API
  -> profile_context_cache.normalize_window_days(7/14/30)
  -> profile_loader.load_profile(..., health_window_days=w)
  -> health_summary.load(conn, customer_id, window_days=w)
  -> fetch_health_rows(customer_health)
  -> fetch_glucose_rows(customer_glucose)
  -> build_base_summary / build_glucose_highlights / build_meal_highlights
  -> ModulePayload(key="health_summary_7d", payload.window_days=w)
```

补充口径：

- 如果窗口内 `customer_health` 和 `customer_glucose` 都没有数据，会返回 `empty` 或带 `stale_hint` 的 `partial`。
- 如果窗口内没有 `customer_health`，但有 `customer_glucose`，会返回 `partial`，基础摘要为空，但血糖曲线聚合仍可展示。
- `source_tables` 固定为 `["customer_health", "customer_glucose"]`，目前没有把专项睡眠、心率、运动或体成分揉进健康摘要。

### 3.2 前端展示链路

前端主卡片位于 `frontend/src/views/CrmProfile/index.vue`：

- 通过 `getCard('health_summary_7d')` 找到 payload。
- 用单选按钮切换 `currentWindowDays = 7/14/30`。
- 切换后请求 `/v1/crm-customers/{id}/profile?window={days}`。
- 体重、血压、血糖、睡眠、运动、饮食直接展示后端字符串。
- 饮水 `water_avg_ml` 和餐食 `m.kcal` 在前端再 `Math.round`。

### 3.3 AI 上下文链路

`context_builder.py` 将 `health_summary_7d` 标记为“近7天健康摘要”，并给字段加中文标签。这个标签目前没有跟 `payload.window_days` 联动；当 AI 请求传入 14/30 天窗口时，payload 是正确的，但模块展示名仍可能保持“近7天”。

## 4. 硬件设备与数据流向

```text
华米手环
  -> huami_heartrates / huami_sleeps / huami_sports 原始表
  -> customer_heartrate / customer_sleep / customer_sport / customer_health

百洋血压计
  -> device_bp_data 原始测量
  -> customer_health.blood_sbp / customer_health.blood_dbp

乐福体脂秤
  -> device_fat_data 原始测量
  -> body_comp
  -> customer_health.weight
```

当前健康摘要读取的是业务汇总层和血糖专项表，不直接读取设备原始表。设备原始数据更适合作为数据质量排查或未来专项扩展来源，不应直接替代 `customer_health` 的 official 业务口径。

## 6. 精度与数值问题分析

### 6.1 体重字段

`customer_health.weight` 是 `int`，代码常量说明存储放大 100 倍：

```python
WEIGHT_DIVISOR = 100
weights = [round(w / WEIGHT_DIVISOR, 1) for w in raw_weights]
```

潜在偏差：

- DB 存 `6523`，代码输出 `65.2`。
- 如果 App 或体脂模块展示 `65.23 kg`，健康摘要会少掉百分位。
- 趋势判断也用 1 位小数，会让小幅变化被抹平。

建议：先确认业务展示口径。如果需要和设备/App 对齐，健康摘要 structured payload 保留 2 位小数，展示层再按业务决定显示 1 位或 2 位。

### 6.2 血糖字段 `fbs` / `pbs`

当前 `_avg()` 统一保留 1 位小数：

```python
def _avg(vals: list) -> float | None:
    clean = [float(v) for v in vals if v is not None]
    return round(sum(clean) / len(clean), 1) if clean else None
```

schema 文档中 `decimal(4,` 被截断，无法从现有文档确认是 `decimal(4,1)` 还是 `decimal(4,2)`。因此这里不能直接断言代码错，只能确认存在“展示精度可能低于存储精度”的风险。

需要执行：

```sql
SHOW CREATE TABLE customer_health;
SELECT fbs, pbs FROM customer_health WHERE fbs IS NOT NULL OR pbs IS NOT NULL LIMIT 20;
```

### 6.3 血压字段 `blood_sbp` / `blood_dbp`

血压原始字段是整数，但健康摘要展示均值，当前可能输出 `121.3 mmHg` 这类小数。这个值不是原始测量值，而是窗口均值。

建议：

- payload 内拆成结构化字段：`blood_pressure_avg_sbp`, `blood_pressure_avg_dbp`。
- 展示字符串明确为“窗口均值”。
- 前端展示可取整，AI 上下文可保留均值但标明统计口径。

### 6.4 睡眠字段 `sleep_min`

当前把分钟换算成小时并保留 1 位：

```python
avg_h = round(sum(sleep_mins) / len(sleep_mins) / 60, 1)
sleep_summary = f"均值 {avg_h}h"
```

这会把 `440 min` 展示为 `7.3h`，和“7小时20分”的自然语言显示略有差异。偏差不大，但如果面向教练决策，建议同时保留 `sleep_avg_min`，展示层格式化为“7小时20分”。

### 6.5 血糖曲线 `customer_glucose.points`

当前按日解析 `points`：

- 日均值 `round(avg, 1)`。
- 日波动 `round(max-min, 1)`。
- 总体峰值 `round(max(all_maxs), 1)`。
- 高血糖天数阈值 `>= 10.0 mmol/L`。

这条链路与 CGM 常见一位小数口径基本一致。主要缺口不是精度，而是低血糖统计已经计算但没有进入 payload。

### 6.6 饮水和餐食热量

- `water_avg_ml` 后端 `round(avg, 0)`，前端再 `Math.round`。
- 餐食 highlight 中的 `m.kcal` 后端保留 float，前端再 `Math.round`。
- `diet` 字符串使用 `_avg(kcals)`，可能显示 `1523.7 kcal`。

建议：饮水和 kcal 属于整数型业务指标，后端 payload 统一输出 int，前端不要二次修正。

## 8. 与实际情况不一致或容易误解的点

1. “健康摘要是 7 个模块之一”应改为“CRM 用户档案模块之一”。当前 `profile_loader.py` 编排的模块已经超过 7 个。
2. `health_summary_7d` 不是只能代表 7 天。它是历史 key，真实窗口以 `payload.window_days` 为准。
3. AI/前端部分标签仍写“近7天健康摘要”，在 14/30 天窗口下会形成口径错位。
4. `profile_loader._P0_KEYS` 中 `"health_summary"` 与正式 key `"health_summary_7d"` 不一致，P0 预加载可能漏掉健康摘要。
5. schema 文档的 decimal 精度被截断，现阶段只能说“待确认”，不能把 `decimal(4,2)` 当作事实。
6. 后端已经计算 `glucose_low` / `glucose_low_days`，但没有写入最终 payload，这属于“半完成字段”。
7. `hba1c`、`stress`、`medication`、营养素字段被查询但未展示，属于“可扩展候选”，不是已落地能力。

## 10. 开发需要做什么：落地、闭环与可扩展方案

### 10.1 先修口径闭环

| 任务 | 改动点 | 验收标准 |
|---|---|---|
| 修复 P0 key | 把 `profile_loader._P0_KEYS` 中的 `"health_summary"` 改为 `"health_summary_7d"` | P0 预加载结果包含健康摘要，不再只靠 full preload 补齐 |
| 动态化健康摘要标签 | `context_builder.py`、`AiCoachPanel.vue` 不再固定写“近7天健康摘要”，改为“健康摘要”或结合 `window_days` 显示 | 7/14/30 天窗口下 AI 引用模块名称不误导 |
| 明确窗口来源 | 继续保留 `payload.window_days` 和 cache key `profile:{id}:hw{days}` | 前端卡片、详情弹窗、AI 请求使用同一个窗口值 |
| 透出低血糖统计 | `health_summary.load()` 把 `_glucose.py` 已计算的 `glucose_low`、`glucose_low_days` 写入 payload | 前端详情和 AI 上下文可看到低血糖风险 |

### 10.2 再修数值精度闭环

建议新增一份健康摘要字段格式注册表，例如：

```python
FIELD_FORMATS = {
    "weight": {"unit": "kg", "precision": 2, "source": "customer_health.weight/100"},
    "blood_pressure_avg_sbp": {"unit": "mmHg", "precision": 0, "stat": "avg"},
    "blood_pressure_avg_dbp": {"unit": "mmHg", "precision": 0, "stat": "avg"},
    "fbs_avg": {"unit": "mmol/L", "precision": 1},
    "pbs_avg": {"unit": "mmol/L", "precision": 1},
    "water_avg_ml": {"unit": "ml", "precision": 0},
    "kcal_avg": {"unit": "kcal", "precision": 0},
    "sleep_avg_min": {"unit": "min", "precision": 0},
}
```

落地原则：

- 后端输出 official structured payload，不把字符串当作唯一真值。
- 字符串摘要继续保留，保证旧前端和 AI prompt 兼容。
- 前端只负责展示格式化，不再偷偷改变数值口径。
- 对 `weight/fbs/pbs/body_comp.weight` 先执行 `SHOW CREATE TABLE` 和样本查询，再决定最终 precision。

### 10.3 补齐字段扩展闭环

建议按优先级拆成三层，不一次性把所有表揉进健康摘要：

| 优先级 | 能力 | 数据源 | 建议输出 |
|---|---|---|---|
| P0 | 已有字段口径稳定 | `customer_health`, `customer_glucose` | 结构化均值、最新值、记录天数、缺失天数、异常天数 |
| P1 | 睡眠和运动增强 | `customer_sleep`, `customer_sport` | 睡眠总时长、深睡/REM、睡眠评分、静息心率、运动消耗 |
| P1 | 心率增强 | `customer_heartrate`, `customer_sleep.rhr`, `body_comp.heart_rate` | 日均心率、静息心率、异常心率天数，并标注来源 |
| P2 | 压力和用药 | `customer_stress`, `customer_health.stress`, `customer_health.medication` | 压力趋势、用药记录摘要；仅做提醒，不生成医疗调整建议 |
| P2 | 营养素增强 | `customer_health.cho/fat/protein/fiber/kcal_out` | 摄入结构、消耗、净热量趋势 |

扩展时要保留 source metadata：

```json
{
  "heart_rate": {
    "avg": 72,
    "source": "customer_heartrate.points",
    "days": 5,
    "quality": "partial"
  }
}
```

这样以后同一个指标来自多个表时，可以解释“为什么这个值和设备 App 不一样”。

### 10.4 建立测试和验收闭环

后端 focused validation：

- 给 `_summarize.py` 增加体重、血压、血糖、睡眠、kcal 的单元测试。
- 给 `_glucose.py` 增加高血糖、低血糖、空 points、异常 JSON 的测试。
- 给 `_meals.py` 增加三餐完整、加餐可选、数组/对象两种 JSON 格式的测试。
- 给 `profile_loader.load_profile_p0()` 增加断言：P0 cards 必须包含 `health_summary_7d`。

数据验收：

- 用 3 个真实客户抽样核对 7/14/30 天窗口。
- 每个客户对比 `customer_health` 原始值、payload、前端展示值。
- 对 stale 场景至少验证两种：完全无数据、有历史但窗口内无数据。

前端验收：

- 切换 7/14/30 天后，健康摘要卡片、详情弹窗、AI 面板使用同一个窗口。
- 饮水、餐食、血压等字段不再出现后端和前端双重 rounding。
- 覆盖不足提示能正确显示 `expected_days` 和实际记录天数。

上线门禁：

- 后端启动通过。
- 前端启动通过。
- 健康摘要 API 示例响应留档。
- 文档同步更新字段注册表、source table、precision 和缺失字段说明。

### 10.5 面向未来的扩展设计

建议把健康摘要从“字段堆叠”演进成“指标注册 + 数据源加载 + 展示格式”的结构：

```text
metric registry
  -> source loader
  -> normalizer
  -> aggregator
  -> structured payload
  -> display formatter / AI context formatter
```

每个指标至少定义：

- `metric_key`：稳定字段名。
- `source_table`：official 数据源。
- `source_priority`：多数据源时的优先级。
- `unit`：单位。
- `precision`：存储精度和展示精度。
- `stat_type`：latest / avg / max / min / trend / count。
- `quality_fields`：记录天数、缺失天数、最后记录日期。
- `ai_safety_note`：AI 使用边界，例如用药只能摘要，不做调整建议。

这样未来新增 BMI、心率、睡眠结构、压力、营养素时，只新增指标定义和 loader，不需要每次改一大段前端和 prompt。

## 11. 待确认事项

1. `fbs` / `pbs` 的真实 decimal 精度：需执行 `SHOW CREATE TABLE customer_health`。
2. `hba1c` 的真实 decimal 精度：需执行 `SHOW CREATE TABLE customer_health`。
3. `body_comp.weight`、`body_comp.bmi` 等体成分字段真实 decimal 精度：需执行 `SHOW CREATE TABLE body_comp`。
4. `device_fat_data.weight` 的真实单位换算：不能只按 `/100` 推断，必须结合 `weight_unit` 和样本值确认。
5. 业务展示口径：体重到底统一 1 位还是 2 位；血压均值展示整数还是 1 位小数。
6. 健康摘要是否应该直接展示 BMI/心率，还是继续让 `body_comp_latest_30d` 承担体成分和体脂秤心率。

## 13. 结论

1. 健康摘要当前数据源和实现边界基本清楚：official 读取 `customer_health + customer_glucose`，`body_comp` 是独立模块，专项睡眠/心率/运动/压力还没有进入健康摘要。
2. “数值对不上”的最大风险不是 SQL 取错表，而是精度和展示口径没有注册化：体重、血压、血糖、kcal、饮水分别在不同层做了 rounding。
3. 当前闭环缺口主要有四个：P0 key 不一致、动态窗口标签不一致、低血糖统计未透出、结构化 payload 不足。
4. 下一步开发应先修闭环和口径，再扩展字段；否则新增心率/BMI/睡眠结构后，只会放大“来源不清、数值不一致、AI 解释不清”的问题。
