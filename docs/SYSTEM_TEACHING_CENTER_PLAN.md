# 系统教学中心落地 Plan

## 1. 目标

为平台新增一个面向运营同学的“系统教学中心”，实现以下目标：

1. 新同学登录后，可以直接在侧边栏看到“系统教学”入口。
2. 进入后，可以像常见网站帮助中心一样阅读 `Markdown` 图文文档。
3. 管理员 `admin` 可以在系统内便捷维护教学文档。
4. 教学文档中使用的图片走当前七牛存储体系，复用现有素材上传能力。
5. 教学内容最终以 `docs` 目录中的 Markdown 组织，便于版本管理和项目协作。

---

## 2. 当前阶段判断

### 已经能复用的能力

1. **侧边栏导航体系**
   - 已有统一路由和菜单分组，可新增一级入口。

2. **文档中心交互骨架**
   - `SopDocuments.vue` 已有“分类 + 搜索 + 管理员增删改”的文档入口模型。

3. **Markdown 内容真值载体**
   - 仓库已有 `docs/` 目录，适合承载教学文档源码。

4. **图片存储能力**
   - 现有素材/七牛上传链路可复用。

5. **管理员权限体系**
   - 已有 `admin` 角色判断，可控制文档编辑权限。

### 还没有的能力

1. 系统内 Markdown 阅读页
2. `docs` 目录的后端读取 API
3. 管理员在线编辑 Markdown 的页面
4. 教学图片上传后插入 Markdown 的专用流程
5. 教学文档目录树 / 排序 / 发布规则

## 3. 推荐命名

候选名称有 3 个：

1. **系统教学**
2. **帮助中心**
3. **新手上路**

### 推荐结论

推荐使用：**系统教学**

原因：

1. 和“飞书文档”区分更清晰
2. 对内部运营同学更直接
3. 比“帮助中心”更像正式培训入口

---

## 4. 推荐入口位置

### 方案 A：放在“系统设置”

优点：

1. 容易理解成平台内置帮助
2. 和“发送记录 / 审批 / 用户 / 权限”都属于系统层能力

缺点：

1. 对新同学来说不够显眼

### 方案 B：放在“运营配置”

优点：

1. 新运营更容易注意到
2. 和“话术管理”一起形成“运营学习/配置区”

缺点：

1. 容易被误解成运营内容配置页面，而不是平台帮助

### 推荐结论

推荐：**先放在“系统设置”分组，名称为“系统教学”**

并在看板“快速入口”中增加一个二次入口，提升可见性。

## 6. 推荐实现方案

## 6.1 方案对比

### 方案 A：纯文件系统方案

做法：

1. 后端直接读取 `docs/system_teaching/*.md`
2. 前端调用 API 渲染 Markdown
3. 管理员编辑时直接修改服务器本地文件

优点：

1. 实现简单
2. 与“docs 路径”诉求最一致
3. Git 管理清晰

缺点：

1. 在线编辑直接改服务器文件，需要后端具备写磁盘能力
2. 多实例/多机部署时一致性较弱
3. 不适合做复杂版本控制

### 方案 B：数据库主存 + docs 导出镜像

做法：

1. 文档正式内容存在数据库
2. 发布时同步导出到 `docs/system_teaching/*.md`
3. 阅读页读数据库或读导出结果

优点：

1. 在线编辑体验更好
2. 可做草稿/发布态
3. 可做版本历史

缺点：

1. 复杂度明显更高
2. “docs 路径是真值”会变弱

### 推荐结论

推荐先做：**方案 A 的增强版**

即：

1. **正式真值仍是 `docs/system_teaching/*.md`**
2. 管理员通过系统内编辑页修改文件
3. 后端负责读写 Markdown 文件
4. 图片不写入本地，统一走七牛

这是当前最符合你诉求、也最容易落地的方案。

## 8. 后端落地设计

### 8.1 推荐新增 API

建议新增一组独立路由，例如：

```text
/api/v1/system-docs
```

建议接口：

1. `GET /api/v1/system-docs/tree`
   - 返回目录树和元信息

2. `GET /api/v1/system-docs/{slug}`
   - 返回单篇 Markdown 正文与元信息

3. `POST /api/v1/system-docs`
   - 新建文档

4. `PUT /api/v1/system-docs/{slug}`
   - 更新文档正文和元信息

5. `DELETE /api/v1/system-docs/{slug}`
   - 删除文档

6. `POST /api/v1/system-docs/upload-image`
   - 上传教学图片

### 8.2 权限建议

#### 所有登录用户

- 可读

#### 仅 admin

- 可写
- 可删
- 可上传教学图片
- 可调整排序

## 10. 教学图片与七牛存储方案

## 10.1 目标

教学文档里的图片不走本地仓库，统一走七牛。

## 10.2 推荐路径

建议教学图片统一使用：

```text
qiwei/docs/YYYY/MM/DD/文件名
```

或在当前七牛前缀下形成：

```text
<QINIU_PREFIX>/qiwei/docs/YYYY/MM/DD/文件名
```

### 10.3 推荐实现方式

推荐不要单独发明一套“文档图片上传”存储逻辑，而是复用现有素材上传体系：

1. 上传时仍通过现有 `storage_facade` / 七牛 provider
2. 仅在 object key 上使用教学文档专属前缀
3. 如需管理留痕，可继续写入 `Material`
4. 在 `Material` 上补一个来源标识，例如：
   - `source_module = 'system_docs'`
   - 或用专属文件夹“系统教学”

### 10.4 为什么推荐复用 `Material`

因为这样可以天然获得：

1. 上传记录
2. 删除能力
3. 统一的公共 URL
4. 未来迁移能力

## 11. 管理员编辑体验建议

管理员编辑必须尽量“顺手”，否则教学中心会很快失效。

建议编辑页具备：

1. Markdown 编辑器
2. 实时预览
3. 上传图片后自动插入 Markdown
4. 自动保存草稿提示
5. 发布前预览
6. slug 自动生成
7. 排序字段
8. 分类选择
9. 是否显示在目录树中

### 推荐插图工作流

1. 管理员点击 **上传图片**
2. 选择本地文件
3. 后端上传到七牛 `qiwei/docs/...`
4. 返回公共 URL
5. 自动插入：

```md
![图片说明](https://...)
```

## 13. 推荐开发阶段

## Phase 1：只读教学中心

目标：

让所有登录用户能在系统内看到教学文档。

范围：

1. 新增侧边栏入口
2. 新增文档目录 API
3. 新增 Markdown 阅读页
4. 读取 `docs/system_teaching/*.md`

验收标准：

1. 可浏览目录
2. 可打开文章
3. 可搜索标题
4. 图片可正常显示

## Phase 2：管理员在线维护

目标：

让 `admin` 不改代码也能维护教学内容。

范围：

1. 新建/编辑/删除 Markdown 文档
2. 上传图片到七牛
3. 自动插入 Markdown 图片链接
4. 保存后落盘到 `docs/system_teaching`

验收标准：

1. 管理员能新建一篇文档
2. 能上传一张图片并插入正文
3. 普通用户刷新后可见

## Phase 3：增强能力

目标：

让教学中心更像正式帮助中心。

范围：

1. 文档搜索
2. 目录树折叠
3. 最近更新
4. 页面内目录
5. 草稿/发布态
6. 版本历史

## 14. 当前 blocker

当前不是技术做不到，而是有两个实现选择需要明确：

1. **文档真值是否坚持放在 `docs/` 文件系统**
2. **管理员在线编辑是否允许直接改服务器文件**

基于你的诉求，我的推荐口径是：

1. **正式真值：`docs/system_teaching/*.md`**
2. **管理员可在线编辑：是**
3. **图片真值：七牛 `qiwei/docs/...`**

---

## 15. 推荐下一步

最值得继续推进的顺序是：

1. 先把教学文档内容骨架补齐
2. 再做只读版“系统教学”页面
3. 最后再做管理员在线编辑和七牛图片上传

原因：

1. 内容骨架先行，避免页面做出来却没有可用文档
2. 只读版先上线，新同学能先用起来
3. 在线编辑属于体验增强，不必卡住第一阶段上线
