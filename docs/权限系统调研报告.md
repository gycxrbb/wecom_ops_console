# 权限系统调研报告

## 1. 角色体系

系统采用 **两层角色** 设计，角色存储在 `users.role` 字段：

| 角色 | 说明 |
|------|------|
| `admin` | 超级管理员，绕过所有权限检查，可管理用户、审批、话术、RAG 等全部功能 |
| `coach` | 教练/运营，通过细粒度权限键（`permissions_json`）控制可访问的功能模块 |

新用户默认角色为 `coach`。

## 2. 用户模型关键字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `role` | String(32) | `admin` 或 `coach` |
| `permissions_json` | Text, nullable | JSON 格式权限键值对，如 `{"send":true,"schedule":true}` |
| `status` | Integer | 1=启用, 0=禁用 |
| `auth_source` | String(32) | `local`=本地账户, `crm`=CRM 同步账户 |
| `crm_admin_id` | Integer, nullable | 关联 CRM 管理员 ID，用于客户数据可见性判断 |

## 3. 权限键一览

| 权限键 | 中文 | 后端强制 | 前端路由守卫 | 所属分组 |
|--------|------|----------|-------------|---------|
| `send` | 发送中心 | Yes | Yes | 核心业务 |
| `schedule` | 定时任务 | Yes | Yes | 核心业务 |
| `group` | 群管理 | Yes | Yes | 数据管理 |
| `template` | 模板中心 | Yes | Yes | 数据管理 |
| `plan` | 运营计划 | Yes | Yes | 数据管理 |
| `asset` | 素材库 | Yes | Yes | 数据管理 |
| `sop` | 飞书文档 | Yes | Yes | 数据管理 |
| `crm_profile` | 客户档案 | **No** | Yes | 客户运营 |
| `log` | 发送记录 | Yes | Yes | 系统设置 |
| `approval` | 审批中心 | Yes | Yes | 系统设置 |

Admin 角色自动拥有全部权限，无需逐项配置。

## 4. 权限执行层次

### 4.1 前端路由守卫

`router/index.ts` 的 `beforeEach` 守卫执行三层检查：

1. **认证检查**：`requiresAuth=true` 的路由，未登录则跳转 `/login`
2. **角色检查**：`meta.role='admin'` 的路由（话术管理、用户管理、权限管理），非 admin 跳转 `/dashboard`
3. **权限检查**：`meta.permission` 的路由，调用 `hasPermission(user, key)` 检查，无权限跳转 `/dashboard`

`hasPermission` 逻辑：admin 直接返回 true，coach 读取 `user.permissions[key]`。

### 4.2 前端侧边栏

`SidebarContent.vue` 通过 `moduleVisible(key)` 控制菜单项可见性，逻辑与 `hasPermission` 一致。

### 4.3 后端 API 层

三种执行机制：

| 机制 | 函数 | 用途 |
|------|------|------|
| 角色门控 | `require_role(user, *roles)` | 限制仅 admin 可访问 |
| 权限门控 | `require_permission(user, key)` | Admin 绕过；Coach 检查 `permissions_json[key]` |
| 数据级隔离 | `assert_can_view(user, customer_id)` | Admin 看全部客户；Coach 只能看分配给自己的客户 |

### 4.4 CRM 客户数据隔离

`app/crm_profile/services/permission.py` 实现：

- `assert_can_view(user, customer_id)` — 通过 `customer_staff` 和 `group_coachs` 表判断教练与客户的关联关系
- `resolve_visible_customers(user)` — 返回教练可见的全部客户 ID 列表

## 5. 各端点权限要求

| 端点 | 检查方式 | 要求 |
|------|---------|------|
| 群管理 CRUD | `require_permission(user, 'group')` | `group` 权限 |
| 模板 CRUD | `require_permission(user, 'template')` | `template` 权限 |
| 素材 CRUD | `require_permission(user, 'asset')` | `asset` 权限 |
| 发送消息 | `require_permission(user, 'send')` | `send` 权限 |
| 定时任务 CRUD | `require_permission(user, 'schedule')` | `schedule` 权限 |
| 审批操作 | `require_role('admin')` + `require_permission('approval')` | admin + `approval` |
| 发送记录 | `require_permission(user, 'log')` | `log` 权限 |
| 飞书文档 | `require_permission(user, 'sop')` | `sop` 权限 |
| 运营计划 | `require_permission(user, 'plan')` + 所有权 | `plan` 权限 |
| CRM 客户档案 | `get_current_user` + `assert_can_view` | 仅数据级隔离 |
| CRM 积分 | 仅 `get_current_user` | **无权限检查** |
| CRM 群绑定 | 仅 `get_current_user` | **无权限检查** |
| 用户管理 | `require_role('admin')` | 仅 admin |
| 权限管理 | `require_role('admin')` | 仅 admin |
| 话术管理 | `require_role('admin')` | 仅 admin |
| RAG 管理 | `require_role('admin')` | 仅 admin |

## 6. 种子数据

系统初始化时创建两个默认账户（`app/services/seed.py`）：

| 账户 | 角色 | 权限 |
|------|------|------|
| Admin（配置的用户名） | `admin` | 全部隐式拥有 |
| Coach（配置的用户名） | `coach` | 默认开启全部 9 个后端权限键 |

## 7. 已知问题

### 问题 1：`crm_profile` 权限后端未强制

前端定义了 `crm_profile` 权限键，用于路由守卫和侧边栏可见性控制。但后端 `ALL_PERMISSIONS` 列表不包含此键，CRM 路由也从未调用 `require_permission(user, 'crm_profile')`。

**影响**：前端隐藏了入口，但任何已登录用户可直接调用 CRM API 端点（仅受数据级隔离限制）。

### 问题 2：`crm_profile` 权限无法通过管理界面持久化

`PUT /api/v1/permissions/members/{user_id}` 端点在保存时会过滤键：`if k in ALL_PERMISSIONS`。由于 `crm_profile` 不在 `ALL_PERMISSIONS` 中，保存时会被丢弃。

**影响**：在权限管理界面勾选 `crm_profile` 保存后不会生效。

### 问题 3：CRM 积分和群绑定端点缺少权限检查

`api_crm_points.py` 和 `api_crm_groups.py` 的所有端点仅调用 `get_current_user`，未做任何权限或角色验证。

**影响**：任何已登录用户（包括零权限 coach）均可访问 CRM 积分和群绑定数据。

### 问题 4：运营计划权限检查逻辑重复

`api_operation_plans.py` 中的 `ensure_plan_access` 手动读取 `permissions_json` 检查 `plan` 键，与 `require_permission` 逻辑重复，可能导致不一致。

### 问题 5：权限变更无审计日志

权限更新接口 `PUT /api/v1/permissions/members/{user_id}` 未写入 `AuditLog`，权限变更无法追溯。
