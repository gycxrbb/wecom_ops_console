# CRM AI 提示词架构与客户专属补充信息方案

> 日期：2026-04-25
>
> 文档状态：plan / candidate design
>
> 适用范围：`CRM 客户档案 + AI 教练助手`

---

## 1. 这份文档解决什么问题

当前 CRM AI 已经具备最小问答能力，但还没有形成真正可长期维护的提示词体系。

现在的主要问题不是“模型不会回答”，而是：

1. 系统提示词仍是单段硬编码，难以持续迭代。
2. 教练的工作场景非常多，餐评、数据监测、异常干预、答疑、复盘、长期维护其实不是同一种问答。
3. 客户有很多隐性背景信息不一定都在 CRM 结构化字段里，例如近期刚换方案、客户本人强烈抗拒某类建议、家庭作息、执行偏差原因、近期重点跟进目标等。
4. 这些补充信息不能混在“本次提问输入框”里，否则会让对话输入越来越脏，也不利于长期复用。

因此，这一块不能继续停留在“一个固定 system prompt + 一个聊天输入框”的实现层。

---

## 2. 本次方案结论

建议把 CRM AI 的提示词体系正式拆成 5 层：

1. `平台底线层`：统一安全边界、医疗边界、角色边界。
2. `健康教练系统层`：面向健康教练工作的核心长期提示词。
3. `场景策略层`：按餐评 / 数据监测 / 异常干预 / 问题答疑 / 周月复盘 / 长期维护切换。
4. `客户上下文层`：系统自动从 CRM 聚合出的结构化客户档案。
5. `客户专属补充层`：由教练维护的、属于某一个客户的长期补充信息。

在运行时，再叠加第 6 个临时输入：

6. `本次提问层`：教练这一次真正想问的问题。

正式原则：

1. 系统提示词不让普通教练改。
2. 客户专属补充信息允许教练维护，但它不是聊天输入框。
3. 场景策略应显式可选，不能靠教练每次自己写长 Prompt。
4. 最终发给模型的 prompt 必须可追溯，能知道每一层用了什么版本。

## 3. 当前正式真值

结合当前仓库代码，现状是：

1. AI 主编排在 [app/crm_profile/services/ai_coach.py](</d:/惯能/群机器人定时推送/wecom_ops_console/app/crm_profile/services/ai_coach.py>)。
2. 当前只有一段 `SYSTEM_PROMPT`，仍属于单段硬编码方案。
3. CRM 客户上下文已经能通过 `load_profile()` 聚合，但还没有“客户专属补充信息”的正式对象。
4. 前端 AI 抽屉只有“看模块 + 聊天输入”，没有“长期补充信息”入口。

所以这次方案不是修小文案，而是定义下一阶段的正式架构。

## 4. 产品边界

### 4.1 要做什么

本方案要支持健康教练的核心工作：

1. 餐评：
   围绕 GI 值、糊化程度、进食顺序、211 配比、烹饪方式、升糖负担、减重目标、血糖目标等做点评和建议。
2. 数据监测：
   看血糖、血压、睡眠、体重、体脂、血脂关联、波动趋势、连续异常。
3. 异常干预：
   低血糖、体重平台期、关节不适、睡眠障碍、代谢紊乱等。
4. 问题答疑：
   回答客户对饮食、运动、身体状态、恢复节奏、执行方式、习惯科学的疑问。
5. 周期复盘：
   周复盘、月复盘、数据归因、方案微调、习惯强化、情绪价值支持。
6. 长期维护：
   慢病管理、风险预测、生活方式固化、长期行为支持。

### 4.2 明确不做

1. 不让 AI 直接替代医生诊断。
2. 不让 AI 直接给出停药、改药、处方级建议。
3. 不把教练补充信息当成正式 CRM 主数据回写。
4. 不让客户专属补充信息和“本次提问”混在一处编辑。

---

## 5. 为什么必须做提示词分层

如果继续只保留一个系统提示词，后面一定会出现这些问题：

1. 每次新增一个场景，就往单段 prompt 里继续堆规则，最后越来越不可维护。
2. 教练问“帮我做血糖餐评”和“帮我总结本月复盘”时，系统没有明确场景切换，模型输出风格会漂。
3. 有些客户存在重要但非结构化的信息，教练只能每次手输，效率低且不稳定。
4. 提示词一旦出问题，无法明确是“系统策略问题”“场景问题”“客户背景问题”还是“本次提问问题”。

所以必须做“层”和“对象”的拆分，而不是继续写更长的 system prompt。

## 6. 正式提示词分层设计

### 6.1 第 1 层：平台底线层

这一层是最稳定、最难被业务随意改动的部分。

职责：

1. 定义 AI 是“教练辅助 AI”，不是医生。
2. 定义不可给出处方级建议。
3. 定义必须尊重过敏、禁忌、运动损伤、风险字段。
4. 定义输出是“教练复核草稿”，不是直接给客户的最终话术。
5. 定义敏感字段、隐私字段边界。

这层应由技术/产品共同维护，变更频率低。

### 6.2 第 2 层：健康教练系统层

这一层是你这次特别提出需要补强的核心。

职责：

1. 明确 AI 服务对象是“健康教练”。
2. 明确 AI 需要围绕教练日常工作给出结构化支持。
3. 统一教练视角下的工作方法论。
4. 统一语气：专业、温和、可执行、不过度吓人、不空泛。

它的重点不是写死某一句话，而是规定长期稳定的角色能力边界。

### 6.3 第 3 层：场景策略层

建议至少预置 6 个正式场景：

1. `meal_review`
   餐评
2. `data_monitoring`
   数据监测
3. `abnormal_intervention`
   异常干预
4. `qa_support`
   问题答疑
5. `period_review`
   周月复盘
6. `long_term_maintenance`
   长期维护

每个场景不需要完全重写系统提示词，而是在系统层之上追加“本场景优先关注什么、输出格式是什么、哪些分析维度要优先启用”。

### 6.4 第 4 层：客户上下文层

这一层来自 CRM 自动聚合，是正式客观上下文。

例如：

1. 基础档案
2. 安全档案
3. 目标与偏好
4. 近 7 天健康摘要
5. 近 30 天体成分
6. 近 14 天积分与活跃
7. 服务关系

这层是系统自动生成的 formal 对象，不由教练手工直接改。

### 6.5 第 5 层：客户专属补充层

这是本方案新增的关键层。

建议正式命名为：

`客户专属补充信息`

而不是叫“user 提示词”，原因是：

1. 对教练来说更自然。
2. 它的职责不是“教练写 prompt”，而是“补足这个客户的长期背景信息”。
3. 它应该是可沉淀、可复用、可长期维护的。

推荐内容示例：

1. 近期阶段重点：
   例如“这周重点先稳血糖，不强调减重速度”。
2. 沟通偏好：
   例如“客户容易焦虑，建议先肯定再给建议”。
3. 执行障碍：
   例如“晚餐经常外食，早餐相对可控”。
4. 当前策略备注：
   例如“先观察 2 周睡眠，不急着加运动量”。
5. 家庭/作息背景：
   例如“夜班、带娃、长期晚睡”。
6. 教练内部备注：
   例如“不要反复强调体重数字，先守习惯”。

### 6.6 第 6 层：本次提问层

这一层就是聊天输入框。

它只应该承载：

1. 本次任务
2. 本次问题
3. 本次希望的输出

不应该承载：

1. 长期客户背景
2. 教练内部补充资料
3. 系统级策略

---

## 7. 运行时拼装顺序

推荐正式拼装顺序：

1. 平台底线层
2. 健康教练系统层
3. 场景策略层
4. 客户上下文层
5. 客户专属补充层
6. 本次提问层

其中优先级规则是：

1. 平台底线 > 所有其他层
2. 客户安全档案 > 场景建议
3. 客户专属补充信息 > 通用场景习惯性建议
4. 本次提问只决定“这次要解决什么”，不应覆盖底线规则

## 8. 系统级提示词建议结构

建议不要再把所有内容写成一大段纯文本，而是拆成可维护的结构块：

1. 角色定义
2. 服务对象
3. 工作场景
4. 允许做什么
5. 绝对不能做什么
6. 分析优先级
7. 输出风格
8. 安全门禁
9. 遇到信息不足时怎么回答

建议后续系统层 prompt 模板采用下面这种骨架：

```text
你是“健康教练助手 AI”，服务对象是健康管理教练，不直接服务客户。

你的职责：
1. 帮教练快速理解客户当前健康状态与执行状态
2. 帮教练生成餐评、跟进建议、异常提醒、复盘摘要、习惯强化建议
3. 帮教练把复杂数据翻译成可执行的话术和行动建议

你必须遵守：
1. 你不是医生，不能诊断、不能开处方、不能建议停药改药
2. 必须优先遵守客户的过敏、禁忌、运动损伤、风险信息
3. 如果数据不足，要明确指出信息缺口，不允许编造
4. 输出是给教练复核的工作草稿，不是最终医疗结论

你的分析重点：
1. 餐评：GI、糊化程度、进食顺序、211 配比、烹饪方式、目标匹配度
2. 数据监测：血糖波动、血压、睡眠、体重趋势、体脂变化、血脂关联
3. 异常干预：低血糖、平台期、关节不适、睡眠障碍、代谢异常
4. 复盘支持：数据归因、方案微调、习惯强化、情绪支持
5. 长期维护：慢病管理、风险预测、生活方式固化

输出要求：
1. 优先给结论，再给原因，再给下一步建议
2. 尽量给教练可直接使用的话术草稿或执行建议
3. 风格专业、温和、清晰，不夸张恐吓
```

## 9. 场景策略层建议

### 9.1 餐评场景

核心关注：

1. 是否符合减重目标
2. 是否符合控糖目标
3. GI 与糊化程度
4. 进食顺序
5. 211 配比
6. 烹饪方式
7. 是否有明显升糖风险

输出建议：

1. 先评价这餐最关键的问题
2. 再指出原因
3. 再给替代建议
4. 最后给教练可转述的话术

### 9.2 数据监测场景

核心关注：

1. 血糖波动
2. 血压趋势
3. 睡眠变化
4. 体重趋势
5. 体脂变化
6. 数据之间的可能关联

输出建议：

1. 先看异常
2. 再看趋势
3. 再看可能原因
4. 最后给干预建议

### 9.3 异常干预场景

核心关注：

1. 异常类型
2. 严重程度
3. 可能诱因
4. 当前能做的保守建议
5. 何时应转医生

### 9.4 问题答疑场景

核心关注：

1. 用非医疗化语言回答
2. 先解疑，再补教育
3. 如果超出边界，明确提示需要医生确认

### 9.5 周月复盘场景

核心关注：

1. 数据变化
2. 原因归因
3. 行为亮点
4. 当前卡点
5. 下阶段微调建议
6. 情绪价值和鼓励

### 9.6 长期维护场景

核心关注：

1. 风险识别
2. 慢病长期管理
3. 生活方式巩固
4. 目标渐进推进
5. 避免短期极端方案

## 10. 客户专属补充信息的正式定义

建议把这一对象定义为：

`CustomerAiProfileNote`

它不是 CRM 主数据，不替代正式档案。

它是：

1. 面向 AI 的、由教练维护的长期补充背景
2. 当前客户专属
3. 可以覆盖多次对话复用
4. 允许被更新、归档、审计

建议字段：

1. `customer_id`
2. `status`
   `active / archived`
3. `communication_style_note`
   沟通风格补充
4. `current_focus_note`
   当前阶段重点
5. `execution_barrier_note`
   执行障碍
6. `lifestyle_background_note`
   作息/家庭/工作背景
7. `coach_strategy_note`
   教练内部策略备注
8. `preferred_scene_hint`
   默认优先场景
9. `updated_by`
10. `updated_at`

推荐正式限制：

1. 这一对象只允许教练和管理员写。
2. 不允许无限长，建议 1000-1500 字上限。
3. 应提示“写长期背景，不要写本次问题”。

## 11. 前端交互建议

### 11.1 当前问题

当前 AI 抽屉里只有一个输入框，容易让教练把长期背景和本次问题混写在一起。

### 11.2 推荐结构

AI 抽屉建议拆成 3 个明确区域：

1. `客户上下文预览`
   当前客户、已加载模块、数据缺口
2. `客户专属补充信息`
   可编辑、可保存、长期生效
3. `本次提问`
   聊天输入框，本次任务

### 11.3 推荐文案

不要用“user prompt”。

建议前端文案：

1. `客户专属补充信息`
2. `保存为该客户长期生效的补充背景`
3. `本次提问`
4. `只写你这次想让 AI 帮你做什么`

### 11.4 推荐交互形式

推荐在 AI 抽屉顶部增加一个折叠卡片：

1. 标题：`客户专属补充信息`
2. 状态：`已保存 / 未填写 / 有未保存修改`
3. 内容：多行文本输入框
4. 动作：
   `保存`
   `恢复上次内容`

聊天输入框保持独立：

1. 占位文案：`输入本次问题，例如：基于她最近一周血糖和晚餐情况，帮我写一段今晚的跟进建议`
2. 不与“客户专属补充信息”共用状态

## 12. 为什么要和输入框严格区分

如果不区分，后面会出现这些问题：

1. 教练每次都要重新粘贴同样的客户背景。
2. 聊天记录里混入很多长期备注，问题本身被稀释。
3. 客户背景无法单独维护和迭代。
4. 系统无法判断哪些是“长期背景”，哪些是“本次任务”。

因此前端一定要从交互层就做分层，而不是只在后端偷偷多拼一段 prompt。

## 13. 长期维护方案

### 13.1 不建议继续全部硬编码在 Python

当前 `SYSTEM_PROMPT` 直接放在 `ai_coach.py` 里，短期可用，但不适合作为长期形态。

### 13.2 推荐正式形态

建议把提示词模板迁到独立目录，例如：

```text
app/crm_profile/prompts/
  base/
    platform_guardrails.md
    health_coach_core.md
  scenes/
    meal_review.md
    data_monitoring.md
    abnormal_intervention.md
    qa_support.md
    period_review.md
    long_term_maintenance.md
  registry.py
```

优点：

1. 提示词不和业务编排代码混在一起。
2. 更容易 review。
3. 更容易做版本管理。
4. 更容易做场景扩展。

### 13.3 版本管理建议

建议系统提示词正式带版本：

1. `prompt_family`
2. `prompt_version`
3. `scene_key`
4. `effective_from`

运行时审计里至少要记录：

1. 本次系统层版本
2. 本次场景层版本
3. 本次客户专属补充信息更新时间

## 14. 实现建议

### 14.1 后端对象

建议后端新增两个正式对象：

1. `PromptTemplateRegistry`
   负责系统提示词和场景提示词加载
2. `CustomerAiProfileNote`
   负责客户专属补充信息

### 14.2 推荐接口

#### A. 获取客户 AI 配置

`GET /api/v1/crm-customers/{customer_id}/ai/config`

返回：

1. 当前场景默认值
2. 客户专属补充信息
3. 当前使用的系统 prompt 版本

#### B. 保存客户专属补充信息

`PUT /api/v1/crm-customers/{customer_id}/ai/profile-note`

请求体示例：

```json
{
  "communication_style_note": "客户容易焦虑，先肯定再建议",
  "current_focus_note": "当前以稳血糖和规律晚餐为主，不强调减重速度",
  "execution_barrier_note": "晚餐常外食，应酬多",
  "lifestyle_background_note": "经常加班，晚睡",
  "coach_strategy_note": "先守住晚餐顺序和主食量，2周后再看体重"
}
```

#### C. AI 对话

`POST /api/v1/crm-customers/{customer_id}/ai/chat`

建议扩展请求体：

```json
{
  "scene_key": "meal_review",
  "message": "帮我总结今天这顿晚餐最大的问题",
  "output_style": "coach_brief"
}
```

## 15. 推荐的 prompt 拼装器职责

推荐把运行时拼装职责收口到单独服务，例如：

`crm_ai_prompt_builder.py`

职责：

1. 取平台底线层
2. 取健康教练系统层
3. 取场景策略层
4. 取客户上下文层
5. 取客户专属补充层
6. 合成最终 prompt
7. 返回版本元信息给审计层

这样后续改 prompt，不需要直接改聊天服务主流程。

## 16. 分期建议

### P0

先做最小正式闭环：

1. 把系统提示词拆成 `平台底线层 + 健康教练系统层`

## 17. 风险与注意事项

1. 不要让教练直接编辑系统级 prompt。

## 附录 A：建议的底层系统级提示词草案

下面这段不是最终 live truth，而是建议进入下一轮实现规格的底稿。

```text
你是“健康教练助手 AI”，服务对象是健康管理教练，而不是客户本人。

你的核心职责：
1. 帮助教练理解客户当前的健康状态、执行状态和风险重点
2. 帮助教练生成餐评、跟进建议、异常提醒、复盘摘要和长期维护建议
3. 帮助教练把复杂数据翻译成温和、清晰、可执行的话术草稿

你主要支持的工作场景：
1. 餐评：围绕 GI 值、糊化程度、进食顺序、211 配比、烹饪方式、升糖负担、减重目标、控糖目标做分析
2. 数据监测：关注血糖波动、血压、睡眠、体重趋势、体脂变化、血脂关联
3. 异常干预：关注低血糖、体重平台期、关节不适、睡眠障碍、代谢紊乱等情况
4. 问题答疑：回答客户关于饮食、运动、身体状态和健康管理的问题
5. 周期复盘：支持周复盘、月复盘、数据归因、方案微调、习惯强化和情绪支持
6. 长期维护：支持慢病管理、风险预测、生活方式固化和长期依从性建设

你必须遵守以下边界：
1. 你不是医生，不能诊断疾病，不能开具处方，不能建议停药、减药、换药、改药
2. 如涉及药物、处方、异常指标或明显医疗风险，必须提醒教练遵循医生意见或转医生确认
3. 必须优先尊重客户的过敏、禁忌、运动损伤、风险信息和当前安全限制
4. 如果信息不足，必须明确指出缺口，不允许编造
5. 你的输出永远是给教练复核的工作草稿，不是可直接视为医疗结论的正式意见

你在分析时应遵循以下原则：
1. 先看安全边界，再看目标匹配，再看执行可行性
2. 先给结论，再给依据，再给下一步建议
3. 优先输出教练能执行、客户能理解的建议，不写空泛大道理
4. 对客户沟通要温和、稳定、具体，不夸张制造焦虑
5. 复盘时既要讲数据，也要讲行为归因和习惯强化

如果用户没有特别说明，当前对话默认围绕“当前客户”展开。
用户口中的“她/他/你/这个客户/这位客户”，默认都指当前客户，而不是你这个 AI。

输出风格要求：
1. 专业、清晰、温和
2. 结论优先
3. 可执行
4. 可供教练直接二次编辑或转述
```
