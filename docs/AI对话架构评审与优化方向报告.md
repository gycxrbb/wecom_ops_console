# AI 对话架构评审与优化方向报告

> 评审日期：2026-05-13  
> 评审对象：企微系统 AI 教练对话功能完整链路  
> 评审视角：按 AGENTS.md 规范，面向项目负责人表达（看见→理解→决策→闭环）

---

## 1. 架构现状总览

根据架构图和代码实现，当前 AI 对话系统分为以下 7 个阶段：

```
┌──────────────────────────────────────────────────────────────────────────┐
│                        AI 对话完整链路                                     │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  ① 用户输入（前端）                                                       │
│       ↓                                                                  │
│  ② 路由层鉴权【身份/权限校验】                                             │
│       ↓                                                                  │
│  ③ 业务可用性判断（安全档案是否存在、AI 是否启用）                            │
│       ↓                                                                  │
│  ┌────────────────────────────────────────────────────────────────┐      │
│  │ 第一阶段：上下文装配                                             │      │
│  │  ④ 加载 CRM 用户档案（13 模块）→ Redis 缓存                     │      │
│  │  ⑤ 注入 System Prompt（5 层 Prompt 架构）                       │      │
│  │  ⑥ RAG 知识库检索（Embedding → Qdrant → Rerank → Top-K）       │      │
│  └────────────────────────────────────────────────────────────────┘      │
│       ↓                                                                  │
│  ┌────────────────────────────────────────────────────────────────┐      │
│  │ 第二阶段：模型推理                                               │      │
│  │  ⑦ 组装 Context → 提交大模型（DeepSeek/Claude）                  │      │
│  │  ⑧ LLM 流式推理【TTFT → 逐 chunk 输出】                         │      │
│  └────────────────────────────────────────────────────────────────┘      │
│       ↓                                                                  │
│  ┌────────────────────────────────────────────────────────────────┐      │
│  │ 第三阶段：交付与持久化                                            │      │
│  │  ⑨ SSE 打包推送前端                                              │      │
│  │  ⑩ 安全门禁检查（医疗边界、禁忌词）                               │      │
│  │  ⑪ 持久化对话入库（AiCoachChatHistory）                          │      │
│  │  ⑫ 格式化输出到手机端【Thinking + Answer】                       │      │
│  └────────────────────────────────────────────────────────────────┘      │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

**右侧独立管线（RAG Pipeline）**：
```
数据源（话术/文档/知识库）
    ↓
格式化处理 → CSV 导入
    ↓
分支 1: CRM 客户档案加载（customer_info → safety → goals）
分支 2: 安全档案（customer_info.health_condition → 过敏/禁忌/运动损伤）
分支 3: 知识库文档（话术管理 → 系统教学文档 → 自定义知识）
    ↓
知识内容合并 → Semantic Text（Clean Content + Tags）
    ↓
Dense Embedding（调用外部 Embedding API，向量化后存储）
    ↓
建立向量索引（Qdrant，分类/标签/可见性 payload 索引）
    ↓
入库（upsert → Qdrant collection）
    ↓
查询时：Metadata 前置过滤（visibility=customer_visible）→ Top-K 精排 → 返回
```

## 3. 半能做什么（有基础但不完善）

| 能力 | 现状 | 缺口 |
|------|------|------|
| 对话历史上下文 | 有持久化，但未回注到 LLM messages | 多轮对话时 AI 不记得上一轮说了什么 |
| 用户意图识别 | 有 scene_key 手动选择 | 缺少自动意图分类，依赖教练手动选场景 |
| RAG 质量反馈闭环 | 有 feedback 表 | 未用反馈数据优化检索排序 |
| Token 预算控制 | 有 estimate_tokens | 未做硬截断，极端情况可能超限 |
| 教练补充信息 | 有 CustomerAiProfileNote | 填写率低，大部分客户为空 |

---

## 4. 还不能做什么

| 能力 | 说明 |
|------|------|
| 多轮对话记忆 | LLM 每次调用都是无状态的，不带历史 messages |
| 主动推送/触发式对话 | AI 只能被动回答，不能主动发起干预 |
| 工具调用（Function Calling） | AI 不能查数据库、不能触发动作 |
| 对话质量自动评估 | 没有自动打分/质检机制 |
| A/B 测试框架 | 无法对比不同 Prompt/模型的效果 |
| 客户端直接对话 | 目前只有教练端（管理后台）能用 AI |

## 5. 优化方向分析

### 5.1 第一优先级：多轮对话记忆（影响最大）

**现状问题**：
- 每次 `stream_ai_coach_answer` 调用时，`prepared.ai_messages` 只包含 system + 当前 user message
- 对话历史已经存在 `AiCoachChatHistory` 表中，但没有被回注到 LLM 调用
- 用户说"上次你建议我少吃碳水，效果怎么样？"，AI 完全不知道"上次"说了什么

**优化方案**：
```
_prepare_ai_turn 中：
1. 从 audit 表取最近 N 轮对话（同 session_id）
2. 按 token 预算裁剪（保留最近 3-5 轮，或最近 2000 tokens）
3. 插入到 messages 列表中（system → history → user）
4. 利用 prompt cache：history 部分放在固定位置，提高缓存命中
```

**预期收益**：对话连贯性从"每次重新开始"升级为"有记忆的教练"。

**风险**：token 消耗增加 30-50%，需要做好预算控制。

---

### 5.2 第二优先级：上下文装配性能优化（影响 TTFT）

**现状问题**：
- `_prepare_ai_turn_cached` 中，Profile 加载 + RAG 检索是串行的
- Profile 加载涉及 13 个模块的 SQL 查询（虽然有 Redis 缓存）
- RAG 检索涉及 Embedding API 调用 + Qdrant 查询 + Rerank
- 首次对话（缓存未命中）的 prepare 阶段可能 2-5 秒

**优化方案**：
```
方案 A：并行化
- Profile 加载 和 RAG 检索 并行执行（asyncio.gather）
- 预期节省 1-2 秒

方案 B：预加载升级
- 当前已有 ai_context_preload.py，但只做 P0 模块
- 升级为：用户打开档案页时，立即预热完整 Profile + 预计算 Embedding
- 教练点击"AI 助手"时，prepare 阶段直接命中缓存

方案 C：分层上下文（渐进式加载）
- 第一轮只注入 P0 模块（基础+安全+健康摘要），快速出首答
- 后续轮次补充 P1 模块（习惯、计划、学习等）
- 用户感知的 TTFT 从 3s 降到 1s
```

**预期收益**：TTFT（首 token 时间）从 3-5s 降到 1-2s。

---

### 5.3 第三优先级：意图识别与场景自动路由

**现状问题**：
- 当前 `scene_key` 由前端手动选择（默认 `qa_support`）
- 教练经常忘记切换场景，导致 AI 用通用策略回答专业问题
- 例如：用户问"我今天血糖 12.3 怎么办"，如果场景是 `qa_support` 而不是 `abnormal_intervention`，回答质量差很多

**优化方案**：
```
1. 在 _prepare_ai_turn 中增加轻量意图分类：
   - 规则引擎（关键词匹配）：血糖/血压异常 → abnormal_intervention
   - 或用小模型（<1B）做 zero-shot 分类
2. 自动路由到最佳场景，同时允许教练手动覆盖
3. 在 SSE meta 事件中返回 auto_detected_scene，前端展示"AI 判断当前场景为：异常干预"
```

**预期收益**：回答精准度提升，减少教练操作负担。

---

### 5.4 第四优先级：RAG 检索质量提升

**现状问题**：
- 当前 RAG 用 Dense Embedding + Qdrant 余弦相似度 + Rerank
- 但知识库内容质量参差不齐（有些话术太短、有些太长）
- Tag Boost 和 Profile Boost 的权重是硬编码的，没有数据驱动调优
- 没有 Hybrid Search（Dense + Sparse/BM25）

**优化方案**：
```
短期（1-2 周）：
1. 知识库质量审计：标记低质量 chunk，设置 semantic_quality 分级
2. 引入 BM25 稀疏检索，与 Dense 做加权融合（Hybrid Search）
3. 利用 feedback 数据调整 Rerank 权重

中期（1-2 月）：
4. 引入 Query Expansion（用 LLM 改写用户问题，多路召回）
5. 按场景建立专属子索引（餐评知识、血糖知识、运动知识分开）
6. 引入 Late Interaction 模型（ColBERT 类）替代简单 Rerank
```

**预期收益**：RAG 命中率从当前水平提升 20-30%。

---

### 5.5 第五优先级：Token 成本控制

**现状问题**：
- 13 个模块全量注入时，context 可能达到 8000-15000 tokens
- 加上 RAG 结果、Prompt 模板、用户消息，单次调用可能 20000+ tokens
- DeepSeek 虽然便宜，但 Claude 的成本不可忽视
- 当前有 `estimate_tokens` 但没有硬预算机制

**优化方案**：
```
1. 模块级 Token 预算：
   - 给每个模块设置 max_tokens（如 health_summary ≤ 800, safety ≤ 500）
   - 超出时自动摘要压缩

2. 动态裁剪策略：
   - 根据用户问题相关性，只注入相关模块（而非全量 13 个）
   - 例如问"我的积分怎么用"，不需要注入健康摘要和安全档案

3. Prompt Cache 利用率监控：
   - 当前已返回 cached_tokens，但没有监控和优化
   - 建立 cache hit rate 指标，优化 message 排列顺序

4. 模型路由：
   - 简单问题（档案事实查询）→ 小模型或 shortcut_answer
   - 复杂问题（干预建议）→ 大模型
   - 当前已有 shortcut_answer 机制，可以扩展覆盖面
```

**预期收益**：平均 token 消耗降低 30-40%，月度成本显著下降。

---

### 5.6 第六优先级：对话质量监控与闭环

**现状问题**：
- 有 feedback 表（教练可以点赞/踩），但使用率低
- 没有自动质量评估（如：回答是否偏离安全边界、是否有幻觉）
- 没有 A/B 测试能力，无法量化 Prompt 改动的效果

**优化方案**：
```
1. 自动质量评分（离线）：
   - 每日批量抽样 N 条对话
   - 用 LLM-as-Judge 打分（安全性、相关性、专业性、可操作性）
   - 生成质量报告

2. 实时告警：
   - safety_gate 触发时自动通知管理员
   - 连续 N 次低分对话触发人工审核

3. A/B 测试框架：
   - 按 customer_id hash 分流
   - 对比不同 Prompt 版本 / 模型 / RAG 策略的效果
   - 基于质量评分自动择优
```

---

### 5.7 第七优先级：Function Calling / 工具调用

**现状问题**：
- AI 只能"说"，不能"做"
- 教练问"帮我给这个客户设置一个每天喝水 2000ml 的提醒"，AI 只能给建议，不能直接操作

**优化方案**：
```
1. 定义安全的 Tool 集合：
   - 查询类：查客户最近 N 天数据、查积分明细、查课程进度
   - 操作类：创建提醒、调整计划、发送话术（需二次确认）

2. 实现 Function Calling 协议：
   - 在 prompt 中注入 tools schema
   - 解析 LLM 返回的 tool_call
   - 执行 → 将结果回注 → 继续生成

3. 安全边界：
   - 查询类工具自动执行
   - 操作类工具需要教练确认（前端弹窗）
   - 禁止直接修改安全档案、删除数据
```

**预期收益**：从"AI 顾问"升级为"AI 助手"，教练效率大幅提升。

## 7. 架构层面的结构性问题

### 7.1 Prompt 与上下文耦合过紧

**问题**：`prompt_builder.py` 直接拼接 context_text 字符串，没有结构化的 context protocol。

**影响**：
- 无法做模块级 token 预算控制
- 无法按问题相关性动态裁剪
- 难以做 A/B 测试（换一种 context 格式就要改 prompt_builder）

**建议**：引入 Context Protocol 层，将"哪些模块参与"和"怎么序列化"解耦。

### 7.2 RAG 与 Profile 的边界模糊

**问题**：RAG 检索结果和 Profile 上下文都作为"背景知识"注入，但角色不同：
- Profile = 这个客户的事实（高置信度）
- RAG = 通用知识库的参考（需要判断适用性）

**影响**：AI 可能把 RAG 中其他客户的案例当成当前客户的事实。

**建议**：在 Prompt 中明确区分"客户事实区"和"参考知识区"，并在 system message 中强调区别。

### 7.3 缓存失效策略不够精细

**问题**：当前缓存失效用 `invalidate_prefix` 一刀切清除所有窗口变体。

**影响**：
- 客户数据更新时，所有窗口的缓存都被清除
- 如果只是更新了今天的数据，7 天窗口需要刷新，但 30 天窗口的大部分数据没变

**建议**：引入增量更新机制，或按数据变更范围选择性失效。

### 7.4 错误处理与降级策略

**问题**：当前各环节的错误处理是"try-except + warning log"，降级策略不够系统化。

**影响**：
- RAG 挂了 → 静默降级（用户不知道回答质量可能下降）
- Profile 某模块加载失败 → 标记 error 但继续（AI 可能基于不完整信息回答）

**建议**：
- 建立分级降级策略（RAG 降级时在前端提示"当前回答未参考知识库"）
- P0 模块失败时阻断对话（而非带着残缺上下文继续）

## 8. 面向项目负责人的状态翻译

### 看见
教练已经能在客户档案页打开 AI 助手，输入问题后实时看到流式回答，包括思考过程和最终建议。AI 能读到客户的完整档案（13 个维度）和知识库内容。

### 理解
系统能把客户的健康数据、安全信息、目标偏好、执行情况等整理成结构化上下文给 AI，AI 基于这些信息 + 场景策略 + 知识库生成个性化建议。

### 决策
当前最大的体验瓶颈是**对话没有记忆**（每次都像第一次聊）和**首答速度偏慢**（3-5 秒）。这两个问题解决后，AI 助手的实用性会有质的提升。

### 闭环
- 对话已经全链路持久化，可追溯
- 安全门禁已经在线运行
- 但缺少质量监控闭环（不知道 AI 回答得好不好）
- 缺少反馈驱动的优化闭环（教练的反馈没有回流到系统改进）

### 当前 Blocker
1. 多轮记忆缺失 → 教练每次都要重复背景
2. TTFT 偏高 → 教练等待体验差
3. 场景需要手动选 → 增加操作负担

### 下一步最值得做什么
1. **本周**：加入多轮对话记忆 + Profile/RAG 并行加载
2. **下周**：意图自动路由 + Token 预算硬控
3. **本月**：RAG 质量提升 + 对话质量监控
