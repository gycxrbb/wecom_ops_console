# 系统教学页面 UI/UX 评审与改进方案（v3，含编辑体验：对标 Typora）

> 日期：2026-04-22
>
> 评审依据：你提供的“阅读页 + 编辑页”截图与当前页面结构观感（未逐条走通所有线上流程）。
>
> 目标：以用户任务为中心，降低阅读与编辑的摩擦；在不引入新的视觉体系、不硬编码新颜色的前提下，尽量复用系统既有样式与 token，让“系统教学”从观感与交互上都回到“企业后台的一部分”。

---

## 0. TL;DR（最核心结论）

- **阅读端问题本质是“注意力被 UI 切碎”**：卡片过多、装饰过重、导航重复，导致阅读不沉浸、定位不稳定。
- **编辑端问题本质是“写作与填表强耦合”**：元数据字段首屏密集 + 写作区被挤压，导致开始写内容之前就要做大量决策，编辑体验自然会“复杂”。
- **对标 Typora 的关键不是把编辑器做大，而是把界面噪音做小**：默认只做写作、其余配置可折叠；写作区要“所见即所得/实时渲染”、有大纲、支持快捷键、且保存可靠。

---

## 1. 用户与关键任务（以用户为中心的验收标准）

### 1.1 用户角色

- **阅读者（新同学/一线同学）**
  - 任务：找文档 → 快速理解 → 按步骤完成。
  - 体验底线：不要让我“找入口/找重点”。

- **维护者（管理员/运营/教练）**
  - 任务：新建/更新内容 → 插图/代码/步骤 → 组织目录 → 保存/发布。
  - 体验底线：写作不要被打断；保存发布要可靠。

### 1.2 可量化验收口径（建议做成埋点或可观察指标）

- 阅读者：3 次点击内到达目标文档；进入内容区 5 秒内明白“这篇讲什么”。
- 维护者：进入编辑 10 秒内开始写正文；编辑过程中不需要频繁回到顶部处理字段。
- 保存可靠性：编辑 10 分钟不丢字；断网/刷新有明确提示与恢复路径。

## 2. 问题优先级总表（建议用它来排期）

|---|---|---:|---|---|---|
| R1 | 阅读页卡片碎片化、装饰过重 | 高 | 阅读者/维护者 | 信息层级与容器过多，注意力被切碎 | “减法”布局：减少容器、弱化装饰、拉开留白 |
| R2 | 左目录 + 右本页目录同时常驻 | 中 | 阅读者 | 导航概念重复，占用内容宽度 | 本页大纲可折叠/可弹出，命名区分 |
| R3 | 搜索结果反馈不清晰（范围/状态/无结果） | 中 | 阅读者 | 预期不明确 | 明确范围、结果列表化、加载/空态明确 |
| E1 | 编辑页元数据首屏密集，写作启动成本高 | 高 | 维护者 | “填表 + 写作”强耦合 | 元数据收进抽屉/折叠面板，正文为主画布 |

## 3. 阅读页体验评审（UI + 交互）

### 3.1 视觉一致性：像“外嵌页面”而不是系统原生

- 目前阅读页的装饰性背景、强调性卡片与系统其他模块（侧边栏、列表页）的企业级简约风格不一致。
- 视觉重心落在“块状 UI”上而不是“内容”上，与“教学中心”的属性相冲突。

建议（不引入新风格，只对齐既有体系）：
- 头部区域减少背景装饰，主背景保持中性；内容区成为唯一视觉中心。
- 按钮/输入框/圆角/阴影严格复用系统既有样式与 token（不要另起一套颜色与阴影）。

### 3.2 信息架构：三栏合理，但边界应由留白而不是“框”来表达

- 三栏（知识库目录 / 正文 / 本页大纲）本身是对的；问题是每一栏都被“卡片化容器”包裹，导致框套框。

建议：
- 外层用布局与留白分栏，内层容器尽量“扁平”；减少阴影；边框只在必要处出现。

### 3.3 交互：导航概念要去重并清晰命名

- 左侧是“文档集合导航”，右侧是“本页锚点大纲”。两者同时常驻会抢占内容宽度，也容易概念混淆。

建议：
- 右侧“本页大纲”默认折叠，提供一个入口按钮打开；或仅在标题较多（例如 >= 4 个小节）时自动出现。
- 命名明确：左侧“教学目录/知识库”，右侧“本页大纲”。

### 3.4 搜索：让结果可预期

建议最小补齐：
- 在搜索框附近明确“搜索范围”（例如：标题/摘要/分类/slug；是否支持全文取决于实现）。
- 状态要可见：加载中 / 无结果 / 结果数量。
- 结果呈现用系统统一的列表样式，并对命中词高亮。

## 4. 编辑页体验评审（为什么会“复杂”）

一句话总结：**当前编辑页是“CMS 表单页 + Markdown 编辑器”的叠加，而不是“写作工具”**。

### 4.1 写作与元数据强耦合 → 启动成本高

- 元数据字段（标题、slug、分类、排序、摘要、封面、难度、推荐阅读等）首屏密集。
- 这迫使维护者在写正文之前先做大量“填写/选择”，并且这些字段很多并不是每次都要改。

改进原则：
- 默认只让用户做一件事：**写正文**。
- 元数据：**可选、可折叠、可晚点填**。

### 4.2 沉浸感不足 → 写作被打断

典型表现：
- 写作区被表单区挤压；长文必须滚动回顶部。
- 缺少专注模式（全屏/隐藏侧栏）。

改进原则：
- 提供“专注模式”：隐藏非必要 UI，只保留顶部工具栏 + 编辑画布。

### 4.3 缺少“作者效率工具”

Typora 让人觉得顺滑，核心在于“写作肌肉记忆”能被满足：

- 大纲：从标题自动生成，点击跳转。
- 快捷键：加粗/斜体/标题/引用/代码块/链接/列表。
- 图片：粘贴/拖拽即插入（上传中有进度/占位）。

### 4.4 保存/发布不透明 → 不信任感

编辑器体验的底线：用户永远不该担心“刚才写的会不会丢”。

建议最小闭环：
- 明确保存状态：已保存 / 保存中 / 保存失败（可重试）。
- 自动保存：基于变更的节流保存（例如 2–5 秒一次）或固定间隔保存。
- 离开提醒：有未保存更改时阻止误离开。

## 5. 目标体验（对标 Typora：交互规格级描述）

> 这里的“对标 Typora”是体验对标，不是像素级复刻。

### 5.1 页面结构（建议的最小可行形态）

- 顶部工具栏（固定）：返回、标题（可编辑或展示）、保存状态、主操作（保存草稿/发布）、更多设置入口。
- 中央主画布：编辑/渲染一体的内容区域（默认占据首屏 70%+）。
- 侧栏（可折叠）：
  - 左侧：知识库目录（切换文档）
  - 右侧：本页大纲（写长文时打开）
- “更多设置”：右侧抽屉或弹层，承载所有元数据字段。

### 5.2 编辑行为（写作即预览 / 所见即所得）

- 支持“实时渲染”的 WYSIWYM（What You See Is What You Mean）：
  - 输入 Markdown 标题语法后立即呈现标题样式；
  - 列表、引用、代码块有明显排版；
  - 保留源码可编辑性（不强行变成富文本黑盒）。

### 5.3 大纲（必须）

- 自动从 H1–H3 生成大纲。
- 点击大纲可跳转；滚动时高亮当前章节。
- 大纲面板默认折叠，且在专注模式中可一键呼出。

### 5.4 快捷键（建议最小集合）

- `Ctrl/Cmd + B`：加粗
- `Ctrl/Cmd + I`：斜体
- `Ctrl/Cmd + K`：插入链接
- `Ctrl/Cmd + S`：手动保存（即使有自动保存，也要支持肌肉记忆）
- `Ctrl/Cmd + /`：显示快捷键帮助

### 5.5 图片插入（必须低摩擦）

- 支持粘贴图片、拖拽图片到编辑区域。
- 上传中显示占位与进度；失败可重试；成功后自动替换为实际链接并立刻渲染。

### 5.6 保存与发布（必须“可见、可靠”）

- 自动保存：变更后节流保存；保存状态实时可见。
- 发布：明确区分“草稿/已发布”，并可回滚到上一个版本（如果暂时做不到版本，至少要有草稿状态）。

## 6. 分阶段改进（从快到稳，减少返工）

### P0（低风险，快速见效：让编辑不再“填表为主”）

- 编辑页：元数据字段全部收进“更多设置”抽屉，默认收起。
- 编辑页：增加专注/全屏模式；顶部工具栏固定；保存状态可见。
- 阅读页：减少装饰性块与卡片边框/阴影，拉开留白，统一到系统既有样式。

### P1（写作效率补齐：让它像工具）

- 大纲：生成/跳转/滚动高亮。
- 图片：粘贴/拖拽上传插入（含进度与失败重试）。
- 快捷键：最小集合 + 帮助面板。
- 轻量预览：实时渲染或双栏（可选）。

### P2（体验趋近 Typora：更顺滑的结构化编辑）

- 更强的编辑内核：更自然的实时渲染、更友好的表格/代码块/任务列表编辑。
- 版本与发布流程完善：至少支持历史版本/差异对比（如果需求存在）。

## 7. 给开发的实施清单（可直接拆任务）

### 阅读页与搜索流

- 统一布局层级：减少“卡片容器”数量，优先用留白与分隔线组织区域。
- 本页大纲：做成可折叠面板或弹出面板；并修正命名与层级。
- 全文搜索：增加范围说明；后端补齐全字段过滤/倒排索引；结果列表化、高亮命中词组；加载/空态显示。

### 编辑页与图片上传机制

- 元数据抽屉：把 slug/分类/排序/摘要/封面/难度/推荐阅读全部放入抽屉（默认收起）。
- 顶部工具栏：固定；包含保存状态、保存草稿/发布按钮、更多设置、专注模式。
- 草稿/发布状态流转：
  - 后端：数据模型支持 `is_published` / `status=draft` 等属性区分。
  - 前端：文档支持独立于线上读取版本的“草稿保存”；读取时发布版和草稿内容分别落库。
- 专注模式：隐藏侧栏与元数据，仅保留内容画布与关键操作。
- 自动保存：节流保存草稿 + 状态展示 + 离开页面未保存提示。
- 图片上传隔离：
  - 前端：粘贴/拖拽上传组件（含占位与失败重试）。
  - 后端/对象云：定义专属上传逻辑或参数，要求独立接口返回七牛云 URL，且前缀路径固定为 `qiwei/docs/images/`，隔离于正常企业微信消息使用的素材库模块。
- 快捷键：至少 `Ctrl/Cmd+S`、`B`、`I`、`K`。

---

## 8. 非目标（避免越做越散）

- 不新增一套全新视觉主题（颜色/阴影/圆角都应复用系统既有 token）。
- 不在 P0 阶段追求“完全富文本化”的复杂编辑器。
- 不在第一阶段引入大量高级能力（协作、评论、审阅流、版本 diff），除非有明确需求。

## 9. 核心需求确认与落地决策（已确认）

经过确认，系统教学的核心逻辑敲定如下：

1. **状态流转**：**明确区分“草稿”与“已发布”两种状态**。编辑过程中可以随时保存为草稿而不影响线上阅读者，确认无误后再点击“发布”。
2. **文档检索**：**要求支持全文检索**。搜索不能仅停留在标题/摘要层面，需要确保用户能搜到文档正文里的关键步骤和说明。
3. **图片存储隔离**：**文档专用存储路径，与核心业务“企微素材库”物理隔离**。插入文档的图片直接并自动上传至七牛云 `qiwei/docs/images` 目录下。教学文档的图片管理不干扰、不混入向群里下发的日常运营素材。

---

## 10. 最重要的一句话

系统教学是内容密集型模块，体验的核心是：让用户更少注意到 UI、更快获取信息或产出内容。做到这一点的路径不是加装饰，而是：一致的系统样式 + 更少的视觉噪音 + 对标写作工具的交互（大纲/专注/快捷键/可靠保存）。
