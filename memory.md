## 经验 #1: 旧数据库真值未迁移完成时，优先做接口桥接而不是直接改列名

- **类别**: 架构
- **场景**: 前端和路由层已经按新字段工作，但本地数据库和 ORM 仍保留旧列名，且项目没有正式迁移体系。
- **经验内容**: 第一批收口应优先在模型属性、路由序列化和调度逻辑层做兼容桥接，让当前接口稳定，再考虑正式迁移。直接把 ORM 改成新列名而不做数据库迁移，容易在启动或查询时立即失败。
- **验证状态**: 已验证

## 经验 #4: 三栏工作台布局要 sticky + overscroll-behavior: contain

- **类别**: 前端 / UX
- **场景**: 左右固定 + 中间滚动的三栏工作台，左侧导航和右侧编辑器需要 `position: sticky` + `overflow-y: auto` + `overscroll-behavior: contain`，否则用户滚动到列表底部时会触发页面级滚动。
- **经验内容**: 三栏工作台的 sticky 面板必须设 `max-height: calc(100vh - header高度)`，并且用 `overscroll-behavior: contain` 防止滚动链。列宽推荐用固定值 (`260px 280px minmax(0, 1fr)`) 而不是 `fr` 比例，避免长名称撑开布局。
- **验证状态**: 已验证

## 经验 #5: 大 Vue 文件拆分策略——先建子组件再集成模板

- **类别**: 前端 / 重构
- **场景**: 单个 Vue 文件超过 2000 行，需要拆分为子组件 + composable。
- **经验内容**: 拆分步骤：1) 先创建子组件（props/emits 接口要和现有数据对齐）；2) 再创建增强 composable（接收外部状态作为 deps）；3) 用 Python 脚本做模板替换（Edit 工具对大段内容不可靠）。关键约束：子组件只做展示和事件上报，状态管理留在父级。
- **验证状态**: 已验证 轻量 schema 迁移要按真实数据库方言写，不能默认按 SQLite 心智

- **类别**: 架构
- **场景**: 本地代码看起来支持 SQLite，但实际运行环境通过 `.env` 连接的是 MySQL。
- **经验内容**: 轻量迁移脚本必须先按 live engine 方言设计，尤其是 `TEXT` 默认值、`ALTER TABLE` 支持度和幂等行为。不能因为仓库里有 `data/app.db` 就默认当前环境一定是 SQLite。
- **验证状态**: 已验证

## 经验 #3: 演示种子里的占位配置不能被 UI 当成正式已配置

- **类别**: 架构
- **场景**: 系统需要提供默认样例数据，但又依赖外部 webhook、Token 或第三方地址才能真正执行。
- **经验内容**: 占位凭证只能作为 support 示例，不能在接口层被序列化成 official 的“已配置”状态；否则用户会看到按钮可用、配置看似完成，但运行时必失败。更稳妥的做法是 seed 留空，并在执行前做显式校验和清晰报错。
- **验证状态**: 已验证

## 经验 #4: 模板引擎只应处理“看起来像模板”的字符串

- **类别**: 调试
- **场景**: 业务内容里允许自由文本、URL、说明文案，同时又支持 Jinja 变量替换。
- **经验内容**: 不要对所有字符串无差别执行模板编译；应先判断是否包含模板标记，再决定是否渲染。对于语法不合法的模板片段，优先降级为原文而不是直接抛 500，这样预览链路更稳，错误范围也更可控。
- **验证状态**: 已验证

## 经验 #5: 企业微信图文消息依赖公网 URL，内部相对地址不能直接复用

- **类别**: 调试
- **场景**: 图文消息需要填写文章链接和封面图地址，系统内部又有素材库和下载接口。
- **经验内容**: 素材库下载地址、相对路径、登录后可见地址，都不能直接当企业微信图文的 `url` / `picurl` 使用。图文链路要么接入公网可访问资源，要么在前后端显式校验并提前报错，避免让第三方平台在发送阶段才返回难懂错误码。
- **验证状态**: 已验证

## 经验 #6: 发送链路要优先复用“用户可操作对象”，不要暴露底层传输字段

- **类别**: 架构
- **场景**: 图片、文件等消息类型最终需要 `image_path`、`file_path`、`media_id` 这类底层字段，但系统已经有素材库。
- **经验内容**: UI 应优先让用户选择素材库对象，再由后端把 `asset_id` 转成真实发送所需字段。直接把服务器路径或第三方媒体 ID 暴露给运营用户，虽然 support 了技术链路，但不等于形成了 official 的可用产品能力。
- **验证状态**: 已验证

## 经验 #7: 发送防护要覆盖所有出口，不能只修主入口

- **类别**: 架构
- **场景**: 一个系统同时存在立即发送、测试发送、异步队列、日志重试、定时任务等多个发送出口。
- **经验内容**: 如果只在主入口做 webhook、内容结构或 URL 校验，其他旁路迟早会出现“同一条消息在 A 入口能拦住，在 B 入口却报另一种错”的漂移。发送前防护应尽量在共享层收口，至少保证各出口复用同一套约束。
- **验证状态**: 已验证

## 经验 #8: 发送日志应存“可诊断版本的 payload”，不要原样落超大传输体

- **类别**: 架构
- **场景**: 图片、文件等消息类型的真实请求体可能包含 base64、大字段或上传响应。
- **经验内容**: 日志的目标是可诊断，不是完整复制传输层原文。对超大字段应做压缩存储或摘要化处理，例如保留 `md5`、长度、关键结构，而不是把完整 base64 写进数据库；否则功能会先死在日志落库，而不是死在真正的业务发送。
- **验证状态**: 已验证

## 经验 #9: 受保护资源地址不能直接复用为浏览器图片 src 或新窗口下载链接

- **类别**: 调试
- **场景**: SPA 使用 Bearer Token 鉴权，同时又需要在前端展示图片缩略图或触发浏览器下载。
- **经验内容**: `axios` 请求和浏览器原生资源加载是两套通道。后端若只认 `Authorization` 头，就不能把受保护下载地址直接塞给 `<img>`、`<el-image>` 或 `window.open()`；更稳的做法是显式拆分 `preview_url` / `download_url`，并为浏览器资源加载提供可控的 query token 或 blob 下载方案。
- **验证状态**: 已验证

## 经验 #10: 当前端统一 request 带固定 baseURL 时，资源型接口要区分“相对 API 路径”和“已完整站内路径”

- **类别**: 调试
- **场景**: 列表接口通过统一 axios 实例请求正常，但某些资源对象本身已经携带 `/api/v1/...` 形式的下载路径。
- **经验内容**: 不要把“资源对象里携带的完整站内路径”再次传给带 `baseURL` 的 request 实例，否则很容易出现 `/api/api/...` 这种双前缀 404。更稳妥的方式是对这类下载/预览 URL 单独走原始 axios 或先做 URL 归一化。
- **验证状态**: 已验证

## 经验 #11: 宽表负责“每日业务摘要”，专项表负责“高频曲线”，AI 默认上下文不要直接吃原始 JSON

- **类别**: 架构
- **场景**: CRM 健康档案同时存在 `customer_health` 这类按天业务宽表，以及 `customer_glucose` 这类专项点位表，需要给档案页和 AI 教练助手共用。
- **经验内容**: 更稳的做法不是在一个模块里只读单表，也不是把宽表 JSON 和专项点位原样丢给 AI。应先明确角色：`customer_health` 负责每日体重、饮水、营养与三餐记录，`customer_glucose` 负责全天血糖波动与峰值；服务层再按 `customer_id + 日期` 生成窗口摘要、异常高亮和代表性样本。像 `diet_assessment`、图片 URL、全量 `points` 这种高噪音字段，应默认降级为 support 或按需展开，而不是直接晋升为 AI 的 official 默认上下文。
- **验证状态**: 已验证

## 经验 #11: “是否显示变量编辑区”应该绑定消息能力，而不是绑定入口状态

- **类别**: 架构
- **场景**: 同一套消息编辑器既可能从模板进入，也可能由运营直接选择消息类型后开始编辑。
- **经验内容**: 自定义变量面板的显示条件应该取决于“当前消息类型是否支持变量”，而不是“当前有没有选模板”。入口状态只是 support 信息，不是 official 能力边界；一旦绑定入口状态，就会出现 Markdown 直编缺面板这类交互漂移。
- **验证状态**: 已验证

## 经验 #12: 模板中心要把“系统母版”和“我的可发送成品”分层展示

- **类别**: 产品架构
- **场景**: 运营后台同时存在系统内置模板和用户基于其保存出的自定义模板。
- **经验内容**: 模板页如果只做平铺表格，用户会把“起点模板”和“最终可发送内容”混为一谈。更稳的结构是先按消息类型组织，再拆成“系统模板库”和“我的可发送模板”，并补上搜索、最近更新、创建后自动定位高亮，降低运营回找成本。
- **验证状态**: 已验证

## 经验 #13: 素材库文件夹一旦支持层级，前后端都要同时切到“树形思维”

- **类别**: 架构
- **场景**: 素材库从单层文件夹升级到支持子文件夹。
- **经验内容**: 只给数据库补 `parent_id` 不够，接口序列化、同级重名校验、删除约束、侧边栏树形展示、面包屑导航、移动弹窗缩进都要一起调整；否则数据上支持了层级，界面和交互仍然会停留在平铺心智，用户会越来越混乱。
- **验证状态**: 已验证

## 经验 #14: 文件夹拖拽调整层级时，后端必须显式拦住“移动到自己或子孙节点下”的非法结构

- **类别**: 架构
- **场景**: 素材库支持拖拽把文件夹挂到其他文件夹下面。
- **经验内容**: 前端可以做 drop 区域限制，但真正的结构约束必须在后端再校验一遍，包括“不能移动到自己下面”“不能移动到自己的后代下面”“目标目录下不能出现同名文件夹”。这类规则如果只放在前端，迟早会被旁路请求打穿。
- **验证状态**: 已验证

## 经验 #15: 写原始迁移 SQL 时要优先按“最严格数据库方言”约束字段默认值

- **类别**: 架构
- **场景**: 同一套应用同时兼容 SQLite 和 MySQL，并通过 `CREATE TABLE IF NOT EXISTS ...` 这类原始 SQL 做轻量迁移。
- **经验内容**: `TEXT` / `JSON` 这类大字段在 SQLite 上即使声明 `DEFAULT ''` 或 `DEFAULT '{}'` 往往也能通过，但 MySQL 会直接拒绝。只要迁移 SQL 不走 ORM 方言适配，就应该优先按更严格的数据库规则设计字段定义，把默认值收回到应用层显式写入，避免“本地能起、线上起不来”的迁移漂移。
- **验证状态**: 已验证

## 经验 #16: 固定流程型运营编排里，真正提效的不是“多建模板”，而是“复制一天”和“给节点套模板”

- **类别**: 产品架构
- **场景**: 运营内容按“主题 -> 天 -> 固定流程节点”组织，每天结构固定但文案和素材有增量变化。
- **经验内容**: 这类场景下，如果只提供主题、天数、节点三个层级，运营仍然要重复做大量低价值输入。更高效的设计是同时提供“从其他天复制整天编排”和“给当前节点快速套用模板”两条捷径：前者适合把整周/整阶段的骨架快速铺开，后者适合把单节点内容从模板库拖到运营工作台。
- **验证状态**: 已验证
## 经验 #20: 运营编排页优先用页面草稿层接管即时保存

- **类别**: 前端 / 交互 / 架构
- **场景**: 复杂运营编排页已经通过 composable 直接调用保存接口，但输入项很多，继续在底层全面重构会放大回归风险。
- **经验内容**: 优先在页面层增加 `draft + dirty + save/reset + discard confirm`，把 `currentNode/currentDay` 这类服务端真值和页面编辑态分开。这样可以先解决“输入即保存、切换易丢编辑上下文”的使用痛点，又不必立刻改写底层数据流。
- **验证状态**: 已验证

## 经验 #21: 模板弹窗的未保存保护要覆盖入口函数和离开钩子两层

- **类别**: 前端 / 交互
- **场景**: 模板编辑弹窗是本地 reactive 表单，存在“新增模板 / 编辑 / 基于此创建 / 关闭弹窗 / 路由离开 / 刷新页面”多种离开路径。
- **经验内容**: 只拦截弹窗关闭还不够，必须同时包住打开入口函数，并补 `beforeunload` 与路由离开确认；否则本地草稿仍会被新的 `resetForm/fillForm` 覆盖掉。
- **验证状态**: 已验证

## 经验 #22: 页面草稿保存失败时不能吞错后直接清空本地状态

- **类别**: 前端 / 交互 / 稳定性
- **场景**: 页面层有 `save draft -> reset draft` 这类串联逻辑，但底层保存函数内部自己 catch 了异常。
- **经验内容**: 底层保存函数至少要返回 `saved/null` 这类明确结果，页面层只有在成功拿到结果后才能 `reset baseline/reset draft`。否则 UI 会给出“已经保存”的错觉，实际数据没落库，草稿却被清掉。
- **验证状态**: 已验证

## 经验 #23: 运营模板库优先优化“找到正确模板”的决策成本

- **类别**: 产品 / 前端 / 交互
- **场景**: 模板数量开始增长后，运营同学的瓶颈往往不是不会编辑，而是进来后不知道先看哪个、哪个是自己的、哪个刚改过、哪个还能继续复用。
- **经验内容**: 模板页要优先补“默认落点、最近更新、来源标识、更新时间、分类筛选”这些决策信息，而不是先堆更多动作按钮。这样能显著减少翻找、误开和重复确认。
- **验证状态**: 已验证

## 经验 #24: 模板预览应复用发送中心的展示心智，而不是另起一套样式

- **类别**: 前端 / 交互 / 复用
- **场景**: 模板库需要只读预览，运营希望看到“发出去大概长什么样”，但又不值得新做一套独立渲染器。
- **经验内容**: 优先复用发送中心的预览组件，把模板内容组装成 `rendered_content` 的本地 mock 数据传进去。这样模板预览和发送预览保持同一视觉语言，维护成本也更低。
- **验证状态**: 已验证

## 经验 #25: 半结构化 SOP Excel 导入要锚定“业务标识行”，不要依赖固定行号

- **类别**: 导入 / 架构 / 调试
- **场景**: 运营团队维护的是宽表型 `sheet1` SOP，表里同时有周标题、时间行、补充说明行和“第X天”数据行，且未来很可能继续插入备注或空行。
- **经验内容**: 解析这类 Excel 时，最稳的做法不是写死“第几行开始是数据”，而是用业务锚点识别结构，例如用表头里的 `每周` + 节点列识别 header，用 B 列 `第X天` 识别正式数据行，用下一次 `第X天` 之前的内容行视作当前天的补充块。这样对运营文档的小改动更有韧性，也更容易在 preview 阶段给出连续性与重复天数告警。
- **验证状态**: 已验证

## 经验 #26: 当运营已经产出结构化 JSON 时，优先做“双格式导入”，不要强迫回退到 Excel

- **类别**: 产品 / 导入 / 架构
- **场景**: 第一阶段 SOP 先来自 Excel，但运营或项目负责人已经进一步整理出结构化 JSON，并且字段比系统当前模型更丰富。
- **经验内容**: 最稳的做法不是要求业务再回头整理成另一份模板，而是在导入层同时支持 `.xlsx` 与 `.json`。其中 `Plan / PlanDay / PlanNode` 作为 formal 对象直接落库；超出当前正式模型的字段（如固定周事件、执行规则、来源说明）先折叠进 `description` / `focus` / `variables_json`，明确它们当前是 support 信息而不是新的 formal 表结构。这样既不丢业务语义，也避免为了一次导入过早重建数据模型。
- **验证状态**: 已验证

## 经验 #27: 外部用户库接入现有后台时，优先做“外部认证 + 本地镜像”，不要直接推翻本地 user 主键体系

- **类别**: 架构 / 认证
- **场景**: 现有后台所有 owner_id、JWT `sub`、权限判断和审计链路都已经绑定本地 `users.id`，但业务希望登录账号来自另一个独立 MySQL 库。
- **经验内容**: 最稳的接入方式是保留关键本地管理员账号作为正式兜底入口，其余账号先走外部认证；认证通过后再在本地 `users` 表做镜像创建或同步。这样可以复用现有 token、权限、外键和审计体系，而不必在一次迭代里重做所有用户关系。
- **验证状态**: 已验证

## 经验 #28: 遇到遗留 CRM 用户表时，密码校验策略要允许“有限多候选算法”过渡，而不是一开始就假定单一哈希规则

- **类别**: 认证 / 调试
- **场景**: 外部 `admins` 表只提供 `password` 与 `salt`，没有正式密码算法文档，但从字段长度可判断是旧式摘要而不是现代 bcrypt。
- **经验内容**: 在没有源码真值前，可以先把校验实现成少量候选策略白名单，例如 `sha1(password + salt)`、`sha1(md5(password) + salt)` 等，严格限制在已知可能范围内。这样比硬猜单一算法更稳，也比无边界试探更安全；后续一旦拿到 CRM 真源码，再收口成正式算法即可。
- **验证状态**: 已验证

## 经验 #29: 个人中心先补正式 profile 资源，再做 UI 重构，能显著降低“漂亮但空心”的返工风险

## 经验 #36: 图文可以直接吃公网 URL，但 image/file 发送切对象存储时必须改成“取字节再发”

- **场景**: 素材从本地磁盘切到七牛后，预览和图文封面都能直接用公网 URL，但企业微信 webhook 的 image/file 发送仍要求图片字节或 media 上传。

## 经验 #103: 功能开关页不要只返回“是否可用”，要同时返回“为什么当前不可用”

- **场景**: 某个功能既受系统开关控制，也受当前对象上下文完整度限制，例如 CRM 客户档案里的 AI 对话既依赖全局 AI 配置，也依赖客户安全档案是否加载成功。

- **场景**: CRM、运营、知识库等系统把结构化模块数据拼成 prompt，同时 UI 里也要展示“AI 当前参考了哪些信息”。
- **经验内容**: 如果上下文正文和 UI 标签直接使用 `basic_profile / display_name / service_scope` 这类技术键，模型理解和用户理解都会打折。更稳的做法是同时维护“模块中文名 + 字段中文名”映射，让 AI 和用户看到的都是“基础档案 / 客户姓名 / 服务关系”这套业务语言。

## 经验 #105: 单客户 AI 对话必须显式规定代词默认指向当前客户

- **场景**: 教练在客户档案页里和 AI 对话，经常会问“你叫什么名字”“她最近怎么样”“这个客户现在什么状态”等自然语言问题。
- **经验内容**: 如果 prompt 里不明确代词规则，模型会把“你”理解成自己，回答成“我是 AI 助手”。更稳的做法是在 system prompt 和 context header 里双重声明：当前会话围绕唯一客户展开，除非用户明确在问 AI，否则“她/他/你/这个客户”默认都指当前客户。

## 经验 #106: 面向复杂业务场景的 AI 提示词要拆成“系统层 + 场景层 + 客户层 + 本次提问层”

- **类别**: AI / 架构 / 产品

- **经验内容**: 不要继续把所有规则塞进一个单段 system prompt。更稳的结构是：平台底线层负责安全边界，系统层负责角色与方法论，场景层负责业务任务切换，客户层负责长期背景补充，本次提问层只承载这次任务。这样才适合长期维护、版本管理和前端明确分层。

## 经验 #107: MySQL 迁移里的索引幂等不要写 `CREATE INDEX IF NOT EXISTS`

- **类别**: 架构 / 迁移 / 数据库兼容
- **场景**: SQLAlchemy 项目同时兼容 SQLite 和 MySQL，并在轻量迁移里手写索引创建 SQL。
- **经验内容**: `CREATE INDEX IF NOT EXISTS` 在 SQLite 可以工作，但 MySQL 不支持。更稳的正式写法是统一走“先查后建”：用 `inspect(conn).get_indexes()` / `get_unique_constraints()` 判断索引是否存在，再执行普通 `CREATE INDEX` 或 `CREATE UNIQUE INDEX`。不要把 `IF NOT EXISTS` 直接写进索引 SQL。

## 经验 #108: 客户档案里的简单资料问答，不要完全交给大模型自由发挥

- **场景**: 教练在客户档案 AI 抽屉里询问“这个客户叫什么名字”“几岁”“当前状态是什么”“谁在跟进”这类明确资料问题。

## 经验 #109: 缺失字段提醒应属于“进入时上下文提示”，不应挂在每条 AI 回复下面

## 经验 #37: 把本地存储保留为正式降级方案时，迁移后也要保留 local_path 并给读取链路补回退

- **类别**: 存储 / 迁移 / 架构
- **场景**: 新素材主存储切到七牛，但项目希望保留本地文件作为生产降级通道。
- **经验内容**: 仅仅保留 `ASSET_STORAGE_FALLBACK_PROVIDER=local` 只能覆盖上传失败回退，不能覆盖云端读取失败。要形成真正可用的降级方案，迁移历史素材时应默认保留原 `storage_path`，并在下载/发送读取链路里支持“对象存储读取失败时按本地路径回退”。这样即使七牛短时不可用，历史素材仍可继续预览、下载和发送。

## 经验 #38: 对象存储接入上线前，必须做“最小真实上传探针”而不是只看配置是否已填

- **类别**: 调试 / 存储 / 上线
- **场景**: `.env` 看起来已经把 AK/SK、bucket、domain 都填好了，但还不确定云端是否真的可用时。
- **经验内容**: 最有价值的验证不是打印配置，而是实际上传一个极小文本文件、拿到公网 URL、再执行删除。这样可以一次性暴露签名错误、bucket region 配错、域名错误等真问题。本次就通过这个探针先抓到了 `BadToken`，再抓到了 `incorrect region`。

## 经验 #39: 对象存储 key 不能只按文件名推导，必须显式引入唯一因子

- **类别**: 存储 / 架构 / 迁移
- **场景**: 素材库允许出现同名图片、同名文件，而对象存储又要承接正式素材地址时。
- **经验内容**: 仅用标准化文件名生成对象 key 会在迁移或重复上传时造成覆盖。更稳的规则是 `目录前缀 + 日期 + slug + 随机/唯一后缀`，前端仍显示中文名，但云端对象路径必须独立。这样既不会互相覆盖，也便于后续按日期或目录排查对象。

## 经验 #40: 接口重构时，最容易漏的是“小工具级导入”，要给高频入口补最小 smoke

- **类别**: 调试 / 工程质量
- **场景**: 路由函数从本地文件上传切到对象存储上传，逻辑能跑通但运行期突然报 `NameError` 时。
- **经验内容**: 这类问题往往不是架构错，而是重构时遗漏了 `Path`、`BytesIO`、`urlparse` 这类小工具导入。对“高频入口”比如素材上传、登录、发送，重构后最好补一轮最小 smoke 检查，避免把这种低级运行时错误留到用户手点时才发现。

## 经验 #41: 时间展示如果跨后端和浏览器，必须先统一“时间真值口径”再谈格式化

- **类别**: 时间 / 前后端契约
- **场景**: 后端数据库用 UTC naive 时间，前端素材库又直接用 `new Date()` 展示时。
- **经验内容**: 只在前端改 `toLocaleString` 往往不够，因为无时区字符串本身就已经有歧义。更稳的做法是后端先把时间转换成明确时区的 ISO 字符串（例如 `+08:00`），前端再做统一格式化。这样时间真值和展示层才不会漂。

## 经验 #42: 当消息构建开始依赖对象存储读取时，payload builder 应升级成类方法

- **场景**: `build_payload` 这类原本只拼 JSON 的工具函数，后来开始在图片、文件分支里调用类级读取能力（例如 `_read_asset_bytes`）以兼容本地存储和对象存储双 provider。

## 经验 #46: 混合认证最稳的顺序不是“先按来源分流”，而是“先本地验 hash，再决定是否回源”

## 经验 #49: Element Plus 按需引入的 CSS 会丢失基础样式，全量 CSS 更稳妥

- **类别**: 前端 / 构建 / 踩坑
- **场景**: 用 `unplugin-vue-components` + `unplugin-auto-import` 做 Element Plus 按需引入，`importStyle: 'css'` 让每个组件独立加载 CSS。

## 经验 #52: sticky 面板的嵌套滚动容器在移动端会造成手势冲突

- **场景**: 运营编排的左导航和编辑器都设置了 `position: sticky` + `max-height: calc(100vh - Xpx)` + `overflow-y: auto`，在移动端形成独立的嵌套滚动容器。

## 经验 #71: 面向多群的 CRM 运营生成，不要做“前端分批 + 后端逐群查成员”的双重串行

- **类别**: 性能 / 后端聚合 / 运营效率
- **场景**: 发送中心一次为十几个到几十个外部群生成积分排行消息。
- **经验内容**: 如果前端按固定小批次串行请求、后端再对每个群单独查成员和周期积分，就会出现典型的 N+1 放大效应。更稳的做法是：1) 前端一次提交全部选中群；2) 后端批量拉取所有群成员；3) 周/月积分用条件聚合一次算完；4) 前端只负责展示“入队/跳过/未绑定”的结果摘要。这样性能和可解释性都会显著更稳。

## 经验 #72: 预览型重接口除了做快，还必须把“每一段花了多久”返回给前端并打到后端日志

- **类别**: 性能 / 可观测性 / 排障
- **场景**: 用户在前端点击“生成预览 / 批量生成”这类重操作后，只看到一个 loading；一旦超过 axios 超时时间，就只能得到“请求失败”，无法判断卡在数据库、外部依赖还是业务拼装。
- **经验内容**: 对这类接口，优化查询只是第一层，第二层必须补观测性：1) 后端按阶段记录 `群名查询 / 成员积分聚合 / 排行与洞察 / 绑定检查 / total`；2) 返回 `diagnostics` 给前端做耗时摘要；3) 对明显慢的对象输出 `slow_groups` 之类的聚焦信息。这样一旦再次超时，运营和开发都能直接看日志定位，而不是重新盲猜。

## 经验 #73: 需要复用同一套文件存储时，优先给上传载荷增加“显式 object_key”，不要再复制一套上传通道

- **类别**: 架构 / 存储 / 可复用性
- **场景**: 系统教学文档需要上传图片，但图片既可能走本地 `/uploads`，也可能走七牛云，而且希望统一落到 `qiwei/docs/...` 这样的业务路径。

## 经验 #75: 当类型检查先死在 tsconfig 层时，先修编译器配置，再谈真实代码错误

## 经验 #79: 弹窗式编辑器只做“保存”还不够，必须补未保存保护，否则高频维护时很容易误损内容

## 经验 #84: 后台里的富交互弹窗如果出现遮挡和显示不全，优先检查 dialog 容器高度与 body 滚动，而不是只调内部子元素

## 经验 #85: Element Plus 弹窗定制样式时，先确认 class 究竟挂在根节点还是挂在外层容器，否则很多“明明写了却不生效”的问题都是选择器写偏了

## 经验 #87: 编辑器里的“上传图片”只要经过异步请求，想插到光标处就必须在请求发出前先冻结选区

- **场景**: Markdown 编辑器里点击上传图片、粘贴图片或拖拽图片后，上传是异步的；如果等上传完成再去读当前 textarea 选区，光标通常已经丢了，结果内容只会被追加到文末。

## 经验 #88: 文件型知识库新增元数据时，必须同时补齐“归一化真值 + 接口 schema + 编辑器表单”，否则字段会变成假配置

- **场景**: 像系统教学这类 Markdown 文件型 CMS，想新增 `author`、`owner`、`reviewer` 之类元数据时，很容易只补前端输入框或只补后端存储的一半链路。
- **经验内容**: 这类字段要一次性补齐三层：`_normalize_entry()` 里的真值归一化、API 请求/响应 schema 里的显式字段、前端编辑器表单与阅读页展示。少任何一层，都会出现“能填但存不住”“存住但接口不回”“接口有值但界面看不见”的假配置问题。

## 经验 #89: 对象存储直传接入遗留素材表时，先补“旧非空列兼容保底”，再谈新字段收口

- **类别**: 架构 / 存储 / 兼容性
- **场景**: 现有素材主表已经被本地存储时代的 `storage_path` 等字段约束住，现在要接入七牛这类客户端直传对象存储。
- **经验内容**: 直传链路最容易遗漏的是“新字段已经够用，但历史正式列还没退场”。如果 `storage_path` 这类旧列仍是非空或被老链路依赖，确认上传阶段必须显式给出兼容保底值；同时在读取层区分“真实本地回退路径”和“仅用于老表约束的占位值”，避免对象存储素材被误当成本地文件处理。这样才能既保住当前数据库真值，又不让新存储链路被旧约束绊倒。
- **验证状态**: 已验证

## 经验 #90: Python 模块里做连接层重构时，要优先全局检查“同名函数是否被后文覆盖”

- **类别**: 架构 / 调试 / 回归预防
- **场景**: 某个服务模块已经加上了连接池、缓存或超时优化，但线上表现却仍像旧逻辑，甚至比预期更差。
- **经验内容**: 这类情况不能只看文件前半段新代码是否存在，还要全文件搜索同名函数。Python 以后定义覆盖前定义，如果 `_get_connection()`、`query_xxx()` 这类关键入口在后面又被定义了一次，前面那套连接池和超时策略就等于完全没生效。这次 CRM 群目录就是典型案例：前面已有 `read_timeout=30` + 连接池，但后面残留的 5 秒版本把它整个盖掉了。
- **验证状态**: 已验证

## 经验 #91: 运营洞察场景优先识别“趋势回暖”，不要把“回归”写成只能命中绝对静默用户

- **类别**: 产品 / 运营策略 / 洞察规则
- **场景**: 积分排行需要给“最近又活起来的人”贴上“强势回归”一类的运营标签，用于生成鼓励文案。
- **经验内容**: 如果把“回归”定义成“过去完全没有活跃，最近才有积分”，命中率会非常低，也不符合运营同学对“这人最近明显回来了”的直觉。更稳的做法是看短期趋势：例如最近 3 天至少活跃 2 天，且相对前一段时间出现明显回暖，包括“之前完全沉寂”“之前只零星活跃，最近活跃天数明显增加”“最近 3 天积分/日均活跃明显高于更早阶段”。这样既能保住语义，又不会把正常稳定活跃用户误判成“回归”。
- **验证状态**: 已验证

## 经验 #92: 多人运营洞察不能直接复用单人模板，占位符兜底值会被放大成正式文案错误

- **类别**: 后端 / 模板渲染 / 运营内容
- **场景**: 像积分排行这种运营消息，会先识别个人洞察，再把“同场景的多人”合并成一句群消息。
- **经验内容**: 单人模板里常有 `{rank}`、`{detail}`、`{activity}` 这类强依赖个体语义的占位符。如果多人合并阶段为了省事直接复用单人模板，再塞一个 `rank=0`、空 `detail` 之类的默认值，就会把“内部兜底”直接暴露成用户可见的错误文案。稳妥做法是把多人场景单独渲染：单人继续走模板，多人则走专门的 grouped 文案，表达“TOP3/前六/持续活跃”这类群体语义，而不是伪造一个个体名次。
- **验证状态**: 已验证

## 经验 #93: 文件型 CMS 放进 Docker 后，必须同时覆盖“镜像内首包 + 运行期持久化挂载”

- **类别**: 部署 / Docker / 内容系统
- **场景**: 像 `docs/system_teaching` 这种文件型 CMS，正式真值不是数据库，而是仓库里的 JSON + Markdown 文件。
- **经验内容**: 这类目录在 Docker 化时不能只顾 `app/` 和 `data/`。如果 `Dockerfile` 不复制内容目录，容器内首次启动就会缺真值；如果 `docker-compose` 不挂载宿主机目录，容器运行时在后台编辑产生的内容又会在重建后丢失。稳妥做法是两层都补齐：镜像里先 `COPY docs/system_teaching/` 作为首包真值，运行期再用 volume 把宿主机 `./docs/system_teaching` 挂到容器 `/app/docs/system_teaching`，这样既能首启可用，也能保证后续编辑持久化。
- **验证状态**: 已验证

## 经验 #94: 运营文档中心一旦进入真实项目推进场景，底层必须从“分类目录”升级为“资源 + 关系 + 工作台”

- **类别**: 产品 / 信息架构 / 知识管理
- **场景**: 像“飞书文档”这类外部链接库，初期文档数量少时用单一分类还能勉强维护；一旦进入项目前中后期并行推进、模板和正式文档并存、多人协作频繁切换的阶段，文档会迅速失控。
- **经验内容**: 这类能力不能只按“功能分类”或“项目目录”来建。更稳的做法是拆成三层：1) `resource` 记录文档本体和标准链接；2) `workspace` 记录项目/专项/客户等运营推进上下文；3) `binding` 记录文档在该上下文中的角色、阶段和产物类型。再配合 `official / support / candidate / template / archive` 这类关系角色，才能真正告诉运营同学“当前该用哪篇、哪篇只是参考、哪篇可以复用”。
- **验证状态**: 已验证

## 经验 #95: 外部文档入口中心最容易做偏的一步，就是把“入口管理”误做成“内容系统”

- **类别**: 产品 / 架构 / 边界设计
- **场景**: 系统里已有“飞书文档”或其他外部文档入口，团队容易继续往内嵌预览、正文托管、系统内编辑的方向发散。
- **经验内容**: 如果当前目标只是帮助运营同学更快找到、判断、打开正确文档，那系统最该管理的是“上下文和关系”，不是正文内容本身。更稳的边界是：系统只维护链接、角色、阶段、负责人、工作台、状态和使用记录；真正的查看与编辑仍跳转外部平台。这样既能保住交互轻量，也能避免过早卷入正文同步、权限代理、内容一致性这些重问题。
- **验证状态**: 已验证

## 经验 #96: 外链文档中心做实现层设计时，最好把“标准链接”和“实际打开链接”拆开

- **类别**: 架构 / 数据建模 / 外部平台集成
- **场景**: 系统需要管理飞书文档等外部链接，既要做去重和关系绑定，又要保证用户点击后能稳定跳转。
- **经验内容**: 这类场景不要只存一个 `url`。更稳的做法是区分 `canonical_url` 和 `open_url`：前者用于去重、迁移和稳定识别资源，后者用于实际跳转。这样后续即便出现短链、分享链接、不同终端跳转链接，或者平台链接格式变化，也不会把“资源识别逻辑”和“打开行为逻辑”绑死在同一个字段上。
- **验证状态**: 已验证

## 经验 #97: 历史链接迁移进新模型时，宁可先放进 inbox，也不要假装它已经是正式结构化真值

- **类别**: 数据迁移 / 真值管理 / 内容治理
- **场景**: 旧系统里只有标题、分类、链接三元组，新系统开始引入工作台、阶段、角色和治理状态。
- **经验内容**: 迁移这类历史数据时，最容易犯的错是“为了看起来完整”，强行把旧分类映射成新阶段或新正式角色。更稳的做法是：资源先迁进去，旧分类只作为 `legacy_category` 维度保留，关系默认进入系统 `inbox` 工作台，角色默认 `support`，状态默认待校验。这样虽然初期仍需整理，但不会污染新的正式真值，也更方便后续人工归位。
- **验证状态**: 已验证

## 经验 #98: 外链解析能力在第一期一定要“可失败但不中断”，不要把新增链路绑死在自动识别上

- **类别**: 产品 / 后端 / 交互降级
- **场景**: 用户通过“快速登记文档”粘贴飞书链接，希望系统自动识别标题、类型和 token，但不同链接格式、短链或权限限制很容易让解析链路不稳定。
- **经验内容**: 第一阶段最稳的做法不是追求 100% 自动识别，而是把 `resolve-link` 设计成“尽力解析 + 允许手动补录”的辅助能力。也就是说，解析成功时帮用户少填字段；解析失败时不能阻断新增，只要保住 URL、标题、工作台、阶段、角色这些最小必填信息，整条登记链路就还能顺畅完成。这样既降低实现风险，也更符合后台工具的可靠性要求。
- **验证状态**: 已验证

## 经验 #99: 关系型数据库里带 NULL 的联合唯一索引不能当成外链资源的兜底防重方案

- **类别**: 数据建模 / 数据一致性 / 外部链接治理
- **场景**: 外链文档资源尝试用 `source_platform + source_doc_token + doc_type` 做唯一约束，但短期内 `source_doc_token` 可能提取失败或为空。
- **经验内容**: 这类场景不能误以为联合唯一索引就一定能兜底防重。因为在 SQLite/MySQL 这类关系型数据库里，带 `NULL` 的联合唯一约束会让重复记录漏过去。更稳的做法是双保险：1) 对标准化后的 `canonical_url` 计算哈希并加唯一索引；2) 在写入前显式做 `canonical_url` 查重。只有这样，解析失败或 token 缺失时，系统才不会被同一链接反复录入。
- **验证状态**: 已验证

## 经验 #100: 前台“一次登记完成”的链路，后端应优先提供事务型组合接口，而不是让前端串联底层资源接口

- **类别**: 后端 / API 设计 / 事务一致性
- **场景**: 一个后台弹窗同时完成“新增外链资源 + 挂到工作台”，如果前端需要顺序调用两个底层接口，就容易在第二步失败时留下孤儿数据。
- **经验内容**: 这类场景更稳的做法是保留底层 `create resource`、`create binding` 接口给脚本和治理能力复用，但给主交互单独提供一个事务型组合接口，例如 `quick-add`。在一个事务里完成“资源查重复用或创建”与“绑定创建”，失败即整笔回滚。这样既保住数据一致性，也让前端交互和后端真值保持一一对应。
- **验证状态**: 已验证

## 经验 #101: 面向运营同学的后台首页，标题必须和真实数据范围一致，否则用户会先怀疑自己再怀疑系统

- **类别**: 产品 / 交互 / 信息架构
- **场景**: 首页上同时出现“我负责的工作台”“最近打开”“治理队列”这类模块，但实际数据来源并不严格对应标题含义，例如把公共工作台塞进“我负责的”，把全量更新时间列表写成“最近打开”。
- **经验内容**: 这种错位比单纯文案不好看更伤。因为用户看到的是业务语言，脑子里会默认它说的就是真值。一旦标题和数据范围不一致，用户会马上失去信任感，甚至误以为自己权限错了或系统记错了。更稳的做法是：数据先分清“我的 / 公共入口 / 其他可查看 / 我最近操作过”，再决定标题；如果拿不到对应数据，就宁可改标题，也不要硬贴一个更好听但不准确的名字。
- **验证状态**: 已验证

## 经验 #102: 内部实现术语不要直接端给运营用户，尤其是“治理、候选、support/running/wiki”这类系统词

- **类别**: 产品 / 文案 / 后台体验
- **场景**: 新模块刚完成开发时，研发很容易沿用模型和接口里的内部命名，把按钮和卡片直接做成“快速登记”“治理队列”“候选”“running”“wiki”。
- **经验内容**: 这类词对研发是精确的，对运营却是高理解成本。更稳的做法是把系统词翻译成任务词：例如“快速登记”改成“登记飞书链接”，“治理队列”改成“待处理问题”，“official/support/candidate”改成“当前在用/参考资料/备选方案”。模型层可以继续保留精确命名，但用户界面必须优先回答“点了会发生什么、我现在该做什么”，而不是要求用户先学习系统概念。
- **验证状态**: 已验证

## 经验 #101: 客户级 AI 不能从“全量字段注入”开始，必须先建设版本化上下文服务

- **类别**: 架构 / AI / 隐私 / 产品
- **场景**: CRM 库里有大量客户健康、设备、积分、服务和身份字段，业务希望让 AI 基于客户全景信息生成教练跟进建议。
- **经验内容**: 最稳的落地方式不是把 30 多张表和 500 多个字段直接塞进 Prompt，而是先定义版本化 `CustomerProfileContext`：安全底线字段必传，常规维度做摘要，当前问题相关模块再按预算展开，并且每个模块都带 `source_status / freshness / redactions`。AI 输出必须定位为 coach-facing draft，敏感身份字段默认不传，医疗/过敏/禁忌规则要做前后门禁。这样客户档案本身先成为独立能力，AI 只是建立在正式上下文服务上的增强层。
- **验证状态**: 待验证

## 经验 #102: 高风险 AI 上下文字段绑定优先用代码注册表，而不是后台自由配置

- **类别**: 架构 / AI / 隐私 / 成本控制
- **场景**: 健康管理场景下需要把 CRM 字段绑定到 AI 上下文，但字段里同时混有健康敏感信息、身份 PII、权限字段和大量原始 JSON。
- **经验内容**: 第一版应采用代码级 `Context Binding Registry`，把字段白名单、脱敏策略、loader、缓存、超时和 token 预算一起固化，经过 code review 后再上线。后台可以开放“场景 preset / 模块开关 / 展开模块选择”，但不应开放任意数据库字段直连 AI。这样能同时控制隐私风险、医疗安全风险、成本膨胀和实现漂移。
- **验证状态**: 待验证

## 经验 #103: 手写 SQL 涉及 MySQL 保留字表名必须加反引号

- **类别**: 调试 / 数据库
- **场景**: 项目使用 pymysql 直写 SQL 查询 CRM 外部数据库，表名恰好是 MySQL 保留关键字（如 `groups`、`order`、`key` 等）。
- **经验内容**: MySQL 保留字用作表名或列名时，必须用反引号 `` ` `` 转义，否则会触发 `ProgrammingError (1064)` 语法错误。常见踩坑表名：`groups`、`order`、`key`、`status`、`index`、`match`。建议在手写 SQL 时对所有非 `customers`/`admins` 这类明确非保留字的表名统一加反引号，从根本上避免此类问题。
- **验证状态**: 已验证

## 经验 #104: 列表页初始化用 include_filters 合并请求 + GROUP_CONCAT 内联关联名称

- **类别**: 性能 / 架构
- **场景**: 分页列表页初始化需要同时加载列表数据和筛选下拉选项，且每行需要显示关联实体名称（如教练名、群名）。
- **经验内容**: 三层优化组合：1) 后端列表接口增加 `include_filters` 可选参数，首次加载时一个请求同时返回列表 + 筛选选项，省掉一次 HTTP 往返；2) 用 MySQL `GROUP_CONCAT` 相关子查询直接在主查询中内联关联名称，替代先查主表再批量查关联表的多轮查询模式——在分页场景下（每页 20-100 条）相关子查询开销可控；3) 权限查询中多条独立 SELECT 合并为 UNION，减少 DB 往返。整体效果：HTTP 请求 2→1，CRM DB 查询 6→3（权限 1 + COUNT 1 + 数据 1），同一连接复用。
- **验证状态**: 已验证

## 经验 #105: 历史档案型页面应把“历史切换”收敛到单卡片，不要把整页上下文一起切成历史态

- **类别**: 产品 / 前端 / 信息架构
- **场景**: CRM 客户详情页同时展示基础档案、目标偏好、近期健康摘要、AI 助手等多个模块，但只有安全档案底层来自带历史版本的 `customer_info`。
- **经验内容**: 更稳的做法不是做整页“回到某一天”，而是在真正有历史语义的卡片右上角提供快照切换。这样用户能明确知道自己只是在回看某次安全档案，不会误以为 BMI、近期健康摘要、AI 上下文也一起回到了那天。实现上建议拆成“历史快照列表接口 + 指定快照详情接口”，前端只替换目标卡片，其他卡片继续保留当前态。
- **验证状态**: 已验证

## 经验 #106: 双流式 AI 交互里，页面 loading 只能跟随“正式回复流”，不能绑死在“思考流 + 回复流”一起结束

- **类别**: 前端 / AI 交互 / 流式状态管理
- **场景**: 同一轮 AI 提问同时发起“思考过程流”和“正式回复流”，前端想在一个消息卡里同时展示两者。
- **经验内容**: `Promise.all` 看起来整洁，但在真实环境里很容易把页面卡死。因为思考流更容易被代理缓冲、被中途 abort、或者在回复结束后才慢半拍收尾。如果 loading 绑在两条流全部结束，用户就会看到“回答已经出来了，但输入框还锁着、思考卡还挂着”的错觉。更稳的做法是：loading 只跟随正式回复流；思考流作为 sidecar，回复完成后直接隐藏并主动 abort，不再阻塞主交互收口。
- **验证状态**: 已验证

## 经验 #107: AI 会话“有审计留痕”不等于“用户已经能回看历史对话”

- **类别**: 产品 / 架构 / AI
- **场景**: 系统已经把 AI 会话写进 `crm_ai_sessions / crm_ai_messages`，但业务侧开始追问“历史对话查看功能是不是已经有了”。
- **经验内容**: 审计落库只是底层能力，不能等同于前台功能完成。真正的“历史对话可用”至少需要三层同时成立：1) 后端提供按客户列会话摘要、按会话读完整消息的正式接口；2) 前端有明确的历史入口、会话切换和回到新对话的交互；3) 继续追问时能够复用被选中的 `session_id`，否则只是“看旧消息”，不是“接着聊”。以后只要看到“数据已经存了”的说法，都要继续追问是否已经形成用户可用闭环。
- **验证状态**: 已验证

## 经验 #108: 健康教练类 AI 的二期重点通常不是“更聪明”，而是“更安全、更可追溯、更一致”

- **类别**: 产品 / AI / 架构
- **场景**: CRM AI 助手已经完成最小问答闭环，团队开始评估下一阶段优先做什么。
- **经验内容**: 对健康教练这类高语义、高风险场景，MVP 之后最值得优先推进的通常不是更多场景 Prompt 或更炫的交互，而是 4 条治理线：1) 结构化安全门禁，避免只靠关键词拦截；2) 审计真值补全，把 `prompt_version / prompt_hash / scene_key / output_style` 变成可追溯事实；3) 上下文治理，让模块展开和 token 预算真正可控；4) 历史续聊一致性，避免同一会话前后静默切换策略。先补治理，再加花样，长期回归成本会低很多。

## 经验 #115: 运营语料导入要兼容“不严谨分隔符”，不能把规范成本甩给运营

- **类别**: 数据治理 / AI / 工具
- **场景**: 运营同学整理 RAG 话术、图片、视频语料时，常会混用顿号、中文逗号、英文逗号、分号、斜杠或竖线来表达多标签。

## 经验 #116: AI 首问性能优化要把“缓存构建时机”和“对话读取策略”分开

- **类别**: 架构 / AI / 性能
- **场景**: 用户已经进入 CRM 客户档案页，业务要求 AI 对话首问不能再等待 CRM 业务表聚合查询；当前阶段又不希望先引入 Redis。
- **经验内容**: 更稳的模式是：档案页允许同步构建 profile context，并写入 L1 + 本地 DB L2 快照；AI 对话阶段只读 L1/L2，miss 时只后台 preload 并快速提示“缓存准备中”。这样把重查询前移到用户看档案的阶段，把 AI 首问线程变成 cache-only 读取，既满足“不引 Redis”的阶段约束，也避免把 support cache 误写成 official 用户档案 truth。
- **验证状态**: 已验证

## 经验 #117: 缓存预热必须有状态接口，否则前端门禁只能靠猜

- **类别**: 架构 / 前端 / 可观测性
- **场景**: 后端已经有后台 preload，但前端只是 fire-and-forget，用户可能在缓存尚未 ready 时点 AI 提问。
- **经验内容**: 预热接口负责“启动或复用构建”，状态接口负责“只读 ready/building/missing”。两者分开后，前端可以安全地做轻量门禁和提示，调试也能复盘 source/cache_key/过期时间；状态接口不能顺手查 DB，否则它会变成新的慢查询入口。

## 经验 #118: ORM 模型加字段后必须同步幂等启动迁移

- **类别**: 数据库 / 启动迁移 / AI 审计
- **场景**: 项目使用 `Base.metadata.create_all` 创建新表，同时用 `ensure_*_schema()` 维护旧库补列；新增 ORM 字段后，如果只改模型和写入代码，旧库不会自动获得该列。
- **经验内容**: 对已有表新增字段时，要同时更新启动期 schema guard，并在 focused validation 中检查旧表列集合。否则功能主流程可能已经成功，最后卡在审计/日志这类旁路写入上，表现为“业务能跑但日志持续 traceback”。

## 经验 #119: 素材库接 RAG 不能全量向量化，必须先补语义和审核状态

- **类别**: AI / 内容治理 / 架构
- **场景**: 原素材库最初只是图片、视频、表情包、文件的存储地，后来要接入健康教练 AI 的 RAG 推荐。

## 经验 #120: RAG 命中素材不等于前端可发送，公网地址必须回到素材库真值

- **类别**: AI / 素材库 / 发送链路
- **场景**: 素材库-RAG 计划把图片、视频推荐到 AI 对话前端，并支持教练预览、下载或进入发送中心。

## 经验 #121: 外部存储删除要按幂等语义处理，审计错误字段不能只给 255 字符

- **类别**: 存储 / 数据库 / 审计
- **场景**: 本地素材库保存了云存储 object key，但远端对象可能已被人工清理、迁移脚本清理或云端生命周期策略删除。
- **经验内容**: 删除类接口应允许“目标已经不存在”作为成功收敛状态，否则用户无法清理本地记录。审计表的 `error_message` 应使用 `TEXT` 或额外结构化字段保存云厂商错误，不能用 255 字符字段承载完整 HTTP 异常，否则真实业务异常会被审计落库异常二次覆盖。

## 经验 #122: 给前端的列表详情不等于已经进入 AI 默认上下文

- **类别**: AI / 上下文治理 / 缓存
- **场景**: 客户档案模块返回结构化列表给前端，例如 `issues[]`、`records[]`、`events[]`，页面展示完整，但 AI prompt 只使用压缩后的文本上下文。
- **经验内容**: 进入 AI 默认上下文的关键事实应提供稳定的标量摘要字段，例如 `issue_detail_summary`，不能只藏在 list/dict 内等待展开。修改上下文 schema 后还要处理 L1/L2 旧缓存，否则 AI 会继续命中旧快照，表现为“页面有、AI 不知道”。

## 经验 #123: 强制刷新参数不能进入默认导航恢复路径

- **类别**: 前端 / 缓存 / 性能
- **场景**: 页面详情接口支持 `refresh=true` 这类强刷参数，同时侧边栏、路由恢复、sessionStorage 恢复会频繁调用同一个加载函数。
- **经验内容**: 强刷参数必须只由明确的“刷新”按钮或调试入口传入，不能写进通用 `loadProfile()` 默认请求。否则每次普通导航都会绕过 L1/L2 缓存，日志看起来像缓存失效，用户体验就是模块来回切换不再秒开。

## 经验 #124: 生产环境只能有一套 official Nginx 管公网入口

- **类别**: 运维 / 环境配置 / 部署
- **场景**: Ubuntu 系统 Nginx 和宝塔 Nginx 同时运行，项目从 `http://120.48.156.119:8080` 升级到 `https://gezelling.com`。
- **经验内容**: 如果服务器同时存在 `/usr/sbin/nginx` 和 `/www/server/nginx/sbin/nginx`，`nginx -T` 看到配置、`nginx -t` 通过，并不代表当前公网入口真的由这套 Nginx 接管。必须用 `ps -ef | grep '[n]ginx'` 和 `ss -lntp | grep -E ':80|:443|:8080|:8000'` 看实际进程和端口归属。本次故障中，`80` 被宝塔 Nginx 占用，系统 Nginx 只监听 `8080`，`443` 未监听，导致 HTTPS `Connection refused`、HTTP 命中默认页。收口原则是明确一套 official Nginx 管 `80/443/8080`，当前项目以系统 Nginx `/usr/sbin/nginx` + `/etc/nginx/sites-enabled/wecom-ops.conf` 为 official；若改用宝塔 Nginx，必须整体迁移配置，不能两套同时抢入口端口。
- **验证状态**: 已验证

## 经验 #125: 启动迁移文件新增函数时要复跑 import 级启动验证

- **类别**: 数据库 / 启动迁移 / 多线程收口
- **场景**: 多条线同时修改 `schema_migrations.py`，一条线新增索引函数，另一条线维护旧库补列常量。
- **经验内容**: 启动迁移文件是应用 import 级关键路径，不能只验证局部函数。新增函数、移动常量或合并 diff 后，至少要跑 `python -m py_compile app\schema_migrations.py app\main.py` 和一次 `uvicorn app.main:app` 短启动；否则一个缩进或拼接错误会让整个后端在 import 阶段失败。
- **验证状态**: 已验证

## 经验 #126: AI 流式客户端启用 HTTP/2 后必须保留协议降级兜底

- **类别**: AI / 网络 / 稳定性
- **场景**: OpenAI-compatible 上游通过 `httpx.AsyncClient(http2=True)` 做流式输出，业务链路对首包稳定性敏感。
- **经验内容**: HTTP/2 能改善并发和复用，但公司出口代理、CDN 或上游网关可能在响应头前直接断开，表现为 `RemoteProtocolError: Server disconnected without sending a response`。客户端应在“尚未向前端输出任何 chunk”时关闭当前连接池并用 HTTP/1.1 重试一次；一旦已经输出 chunk，则不能透明重放，必须暴露真实失败。启动预热和 keepalive 也要按当前 provider 选择 base url，否则预热了 support provider，却没有预热 live provider。
- **验证状态**: 已验证

## 经验 #127: Markdown fence 被组件化拆分后，特殊语言样式要落在组件分支里

- **类别**: 前端 / Markdown 渲染 / UI
- **场景**: `MarkdownRenderer.vue` 先 parse token，再把 fence token 拆成 Vue 组件渲染，业务希望某类 fence（如 ```txt 客户话术）有专用展示样式。
- **经验内容**: 一旦 Markdown fence 被组件化拆分，`markdown-it` 的 `renderer.rules.fence` 只会影响留在 HTML batch 里的内容，不能覆盖已经被拆到 `CodeBlock.vue` 的 fence。特殊语言（客户话术、mermaid、math 等）的正式 UI 分支必须放在 token 分发层或对应组件内部，同时在组件内处理高亮、折叠、换行和复制行为，避免“样式代码存在但实际链路不经过”的真值漂移。

## 经验 #128: 预加载缓存必须把 HTTP method 纳入门禁

- **类别**: 前端 / 缓存 / 数据一致性
- **场景**: 登录后预热侧边栏列表接口，页面随后可能对同一 URL 发起创建、编辑、删除等变更请求。
- **经验内容**: 预加载缓存只能服务只读 GET 请求，不能只按 URL 透明命中所有请求。否则 `GET /resource` 的缓存可能被 `POST /resource` 消费，形成“前端提示成功、后端没有落库”的假成功；变更请求发出前还应清掉同 URL 预加载缓存，避免创建后列表首次读取旧快照。
- **验证状态**: 已验证

## 经验 #129: 启动迁移新增表时不要用一套原始 SQL 跨 SQLite 和 MySQL

- **类别**: 数据库 / 启动迁移 / 环境兼容
- **场景**: 新增业务表需要在应用启动时自动建表，同时本地可能使用 SQLite、生产使用 MySQL。
- **经验内容**: SQLite 的 `AUTOINCREMENT` 和 MySQL 的 `AUTO_INCREMENT` 不能共用，MySQL 也不允许给 `TEXT/BLOB/JSON` 类字段随意声明默认值。启动迁移新增表时要么使用 SQLAlchemy metadata/DDL，要么显式按 dialect 分支；不能只在自增关键字上做字符串替换后继续保留 `TEXT DEFAULT`。
- **验证状态**: 已验证
