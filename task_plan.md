# Task Plan

## Goal

在多线程模式下继续收口 WeCom Ops Console 的消息发送全流程，重点核对不同消息类型的模板配置、预览、发送、日志、重试和定时任务复用能力，并保证代码、文档、验证结果一致。

## Current Phase

- Phase 1: 上下文恢复与任务拆分 `complete`
- Phase 2: 发送全流程并行盘点 `complete`
- Phase 3: 高优先级发送入口修复 `complete`
- Phase 4: 启动验证与文档回写 `complete`
- Phase 5: 下一批发送链路深挖与补强 `pending`
- Phase 5: 下一批发送链路深挖与补强 `in_progress`
- Phase 6: 消息发送全流程盘点与优化 `in_progress`

## Parallel Workstreams

1. 后端六类消息发送能力盘点
2. 前端模板与编辑器配置能力盘点
3. 预览/发送/日志/重试/定时复用闭环盘点
4. 发送能力与消息类型支持盘点
5. 前端模板编辑与预览/发送一致性盘点
6. 发送-日志-重试-定时任务闭环盘点

## Decisions

- 先不横向扩功能，优先做工程收口
- 当前代码真值优先级高于历史文档口径
- 多线程阶段先做读与分析，再决定具体改动批次
- 第一批执行以“统一后端真值 + 最小前端契约修正”为范围，不在本轮引入 Redis / Celery / Next.js 新能力
- 第二轮以 `Schedule` 正式字段真实落库为目标，采用“新增正式列 + 启动时幂等回填 + 主链路切新列”的策略
- 当前批次优先修“直接影响用户发消息”的问题，不先做大重构
- 图片/文件消息配置统一围绕素材库 `asset_id` 收口，不继续把服务器路径和 `media_id` 暴露给用户

## Risks

- `Schedule` / `Message` / `MessageLog` 字段漂移会导致多处联动修改
- 调度服务仍引用旧模型名，可能牵涉启动与运行验证
- 当前工作区已有未提交文档改动，需要避免覆盖
- `Schedule` 当前仍是旧表结构上的桥接实现，多群定时任务还没有成为正式数据库真值
- 当前 `schedules` 表虽然已有正式新列，但历史数据为空时无法证明多群回填效果；后续仍需要真实任务数据验收
- Celery 正式异步发送路径仍未做一轮按消息类型的系统验收
- `template_card` 仍缺少更强的前后端结构校验，存在“能配置但不一定稳发”的风险
- 当前仍未做真实外网发送验收，focused validation 证明的是本地构造链路与项目内部真值，不等于企业微信侧长期稳定
- 消息类型编辑器与后端发送能力可能存在“页面可配但后端不可发”的假闭环
- 日志/重试/定时任务可能仍只覆盖了部分消息类型或部分运行路径
